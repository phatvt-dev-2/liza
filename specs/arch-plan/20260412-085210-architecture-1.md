# Architecture: CLI RBAC via --agent-id Role Validation

**Task:** architecture-1
**Spec:** specs/goals/20260412-cli-native-access-control.md (Section 1)
**Scope:** Role validation for state-mutating CLI commands using existing identity sources.

## 1. Problem Summary

The CLI currently resolves agent identity (positional arg, `--agent-id` flag, auto-resolved orchestrator) but does not validate whether the resolved role is authorized for the command being invoked. The MCP server performs this validation via `withRole` middleware, but CLI commands bypass it entirely. This architecture adds equivalent RBAC checks to the CLI layer.

## 2. Component Boundaries

### 2.1 RBAC Validation Module (NEW: `cmd/liza/rbac.go`)

**Responsibility:** Provide reusable authorization functions for CLI commands. Each function takes a pre-loaded `*pipeline.Resolver`, extracts the agent's role, and checks authorization.

**Not responsible for:** Identity resolution (handled by existing `requireAgentID`, `resolveOrchestratorID`), resolver loading (callers load via helpers), ops-layer logic, error formatting beyond the error message.

**Two authorization modes** (matching MCP middleware patterns):

| Mode | Function | When to use |
|---|---|---|
| Operation-based | `validateAllowedOperation(resolver, agentID, opName)` | Command's operation name is in pipeline YAML `allowed-operations` |
| Type-based | `validateRoleType(resolver, agentID, allowedTypes...)` | Command uses role-type check (not in `allowed-operations` list) |

**Resolver loading helpers:**

| Helper | Input | Used by |
|---|---|---|
| `loadResolverForRBAC(projectRoot)` | Project root path | Commands that call `requireProjectRoot()`: claim-task, submit-for-review, handoff, submit-verdict, mark-blocked, write-checkpoint, set-task-output, await-verdict, await-resubmission, wt-merge, supersede-task, cancel-task, assess-blocked, assess-hypothesis-exhausted |
| `loadResolverFromDir(lizaDir)` | `.liza/` directory path | `add-task` and `add-tasks` — derive from `filepath.Dir(statePath)` |

**Design rationale:** Authorization functions take `*pipeline.Resolver` rather than `projectRoot` to decouple validation logic from config loading. This is necessary because `add-task` and `add-tasks` can operate without project root detection (see Section 2.1.1). Mirrors MCP's `operationChecker` and `typeChecker` (`internal/mcp/middleware.go:96-127`), adapted as standalone functions for one-shot CLI commands.

### 2.1.1 `add-task`/`add-tasks` Authorization Boundary

These two commands never call `requireProjectRoot()` in their main execution path:

- **`add-task`** (`cmd_task.go:66-137`): Uses relative `paths.LizaDirName` (`.liza/`) for state/log by default, or explicit `--state`/`--log` paths. Calls `resolveOrchestratorID()`, which only calls `requireProjectRoot()` internally when no `--agent-id` flag is provided.
- **`add-tasks`** (`cmd_task.go:496-524`): Always uses relative `paths.LizaDirName` for state/log. Has `--agent-id` flag but no `--state`/`--log` flags. When `--agent-id` is provided, `resolveOrchestratorID()` returns immediately without `requireProjectRoot()`.

Adding a `requireProjectRoot()` call for RBAC would introduce a new dependency that breaks these commands when run with explicit `--agent-id` from a directory where project root detection fails.

**Resolution:** Both `add-task` and `add-tasks` load the resolver via `loadResolverFromDir(filepath.Dir(statePath))`:
- `add-task` default: `statePath = ".liza/state.yaml"` → `loadResolverFromDir(".liza")` → loads `.liza/pipeline.yaml`
- `add-task` with `--state`: `statePath = <user-path>` → `loadResolverFromDir(filepath.Dir(<user-path>))` → loads `<user-dir>/pipeline.yaml`
- `add-tasks`: `statePath = ".liza/state.yaml"` → `loadResolverFromDir(".liza")` → loads `.liza/pipeline.yaml`

This matches the existing config-loading convention: the pipeline config lives alongside the state file in the `.liza/` directory.

### 2.2 Command Layer (MODIFIED: `cmd/liza/cmd_task.go`, `cmd/liza/cmd_review.go`, `cmd/liza/cmd_worktree.go`)

**Change:** Each state-mutating command's `RunE` function gains a validation call after identity resolution and before the `commands.*Command()` call.

**Unchanged:** Identity resolution functions (`requireAgentID`, `resolveOrchestratorID`), command registration, flag definitions, argument parsing.

### 2.3 Existing Components (UNCHANGED)

| Component | Role in RBAC |
|---|---|
| `internal/identity/` | `ExtractRole(agentID)` extracts role from agent ID format. Called by validation functions. |
| `internal/pipeline/` | `Resolver.AllowedOperations(role)` returns allowed ops. `Resolver.RoleType(role)` returns role type. Called by validation functions. |
| `internal/commands/` | Command handlers. No RBAC logic added here — validation happens in CLI layer before handler call. |
| `internal/ops/` | Ops layer. Unchanged — authorization is a CLI-layer concern. |

## 3. Interface Contracts

### 3.1 Validation Function Signatures

```go
// loadResolverForRBAC loads the pipeline resolver from project root.
// Returns fail-closed error if loading fails.
func loadResolverForRBAC(projectRoot string) (*pipeline.Resolver, error)

// loadResolverFromDir loads the pipeline resolver from an explicit .liza/ directory.
// Used by add-task/add-tasks when --state/--log bypass project root detection.
func loadResolverFromDir(lizaDir string) (*pipeline.Resolver, error)

// validateAllowedOperation checks that the agent's role has the given
// operation in its pipeline YAML allowed-operations list.
// Returns nil if allowed, descriptive error if not.
func validateAllowedOperation(resolver *pipeline.Resolver, agentID, operationName string) error

// validateRoleType checks that the agent's role type (doer/reviewer/orchestrator)
// matches one of the allowed types.
// Returns nil if allowed, descriptive error if not.
func validateRoleType(resolver *pipeline.Resolver, agentID string, allowedTypes ...string) error
```

### 3.2 Error Contracts

Validation errors are returned as `error` from `RunE`, which Cobra prints to stderr and exits non-zero. Error messages must be actionable — stating what was attempted, what role was found, and what was expected.

**Operation-based error:**
```
operation "submit-verdict" not allowed for role "coder" (agent coder-1)
```

**Type-based error:**
```
command requires role type [doer] but agent "orchestrator-1" has type "orchestrator"
```

**Resolver load failure (fail-closed):**
```
cannot authorize operation: failed to load pipeline config: <underlying error>
```

The fail-closed behavior matches MCP middleware: if the pipeline config cannot be loaded, all operations are denied.

### 3.3 Operation Name Mapping

CLI command names map directly to pipeline YAML operation names — both use kebab-case. No conversion is needed (unlike MCP's `liza_submit_for_review` -> `submit-for-review` via `mcpToolToOperation`).

Only commands that have both a CLI identity source AND an `allowed-operations` entry use `validateAllowedOperation`. Commands checked by role type (`validateRoleType`) do not appear here.

| CLI Command | Operation Name (for `validateAllowedOperation`) |
|---|---|
| `submit-for-review` | `submit-for-review` |
| `handoff` | `handoff` |
| `submit-verdict` | `submit-verdict` |
| `mark-blocked` | `mark-blocked` |
| `write-checkpoint` | `write-checkpoint` |
| `set-task-output` | `set-task-output` |
| `await-verdict` | `await-verdict` |
| `await-resubmission` | `await-resubmission` |
| `add-tasks` | `add-tasks` |
| `supersede-task` | `supersede-task` |
| `cancel-task` | `cancel-task` |

Commands like `analyze`, `sprint-checkpoint`, `update-sprint-metrics`, `clear-stale-review-claims`, `delete-agent`, and `set-discovery-disposition` have `allowed-operations` entries in pipeline YAML (for MCP RBAC) but have no CLI identity source. They are excluded from CLI RBAC and listed as "No RBAC" in Section 5.

## 4. Data Flow

### 4.1 Command Execution Flow (Before vs After)

**Before (no RBAC):**
```
CLI args → identity resolution → commands.*Command() → ops layer → state mutation
```

**After (with RBAC):**
```
CLI args → identity resolution → load resolver → RBAC validation → commands.*Command() → ops → state mutation
                                      |                |
                                      |                ├─ Extract role (identity.ExtractRole)
                                      |                └─ Check allowed-operations or role type
                                      |
                                      ├─ loadResolverForRBAC(projectRoot)    [most commands]
                                      └─ loadResolverFromDir(lizaDir)        [add-task/add-tasks with --state]
```

### 4.2 Validation Placement Per Identity Pattern

**Pattern 1: Positional `<agent-id>` (claim-task)**

```go
// cmd_task.go — claimTaskCmd.RunE
RunE: func(cmd *cobra.Command, args []string) error {
    taskID := args[0]
    agentID := args[1]

    projectRoot, err := requireProjectRoot()
    if err != nil { return err }

    // NEW: RBAC validation
    resolver, err := loadResolverForRBAC(projectRoot)
    if err != nil { return err }
    if err := validateRoleType(resolver, agentID, "doer"); err != nil {
        return err
    }

    return commands.ClaimTaskCommand(projectRoot, taskID, agentID)
}
```

**Pattern 2: `--agent-id` flag (submit-for-review, handoff, submit-verdict, etc.)**

```go
// cmd_review.go — submitForReviewCmd.RunE
RunE: func(cmd *cobra.Command, args []string) error {
    taskID := args[0]
    commitSHA := args[1]

    agentID, err := requireAgentID(cmd)
    if err != nil { return err }

    projectRoot, err := requireProjectRoot()
    if err != nil { return err }

    // NEW: RBAC validation
    resolver, err := loadResolverForRBAC(projectRoot)
    if err != nil { return err }
    if err := validateAllowedOperation(resolver, agentID, "submit-for-review"); err != nil {
        return err
    }

    return commands.SubmitForReviewCommand(projectRoot, taskID, commitSHA, agentID)
}
```

**Pattern 3: Auto-resolved orchestrator with `requireProjectRoot()` (supersede-task, cancel-task, assess-blocked, assess-hypothesis-exhausted)**

These commands already call `requireProjectRoot()` in their RunE and use `resolveOrchestratorID()`. RBAC loads from project root:
```go
// cmd_task.go — supersedeTaskCmd.RunE (after orchestratorID + projectRoot resolution)
resolver, err := loadResolverForRBAC(projectRoot)
if err != nil { return err }
if err := validateAllowedOperation(resolver, agentID, "supersede-task"); err != nil {
    return err
}
```

For commands whose operation is NOT in `allowed-operations` (assess-blocked, assess-hypothesis-exhausted):
```go
// cmd_task.go — assessBlockedCmd.RunE (after orchestratorID + projectRoot resolution)
resolver, err := loadResolverForRBAC(projectRoot)
if err != nil { return err }
if err := validateRoleType(resolver, orchestratorID, "orchestrator"); err != nil {
    return err
}
```

**Pattern 3a: `add-task`/`add-tasks` — state-path-derived resolver loading**

These commands do not call `requireProjectRoot()` in their main execution path (see Section 2.1.1). RBAC loads the resolver from the state file's parent directory:

```go
// cmd_task.go — addTaskCmd.RunE (after statePath is resolved, after orchestratorID resolution)
resolver, err := loadResolverFromDir(filepath.Dir(statePath))
if err != nil { return err }
if err := validateRoleType(resolver, orchestratorID, "orchestrator"); err != nil {
    return err
}
```

```go
// cmd_task.go — addTasksCmd.RunE (after orchestratorID resolution)
statePath := filepath.Join(paths.LizaDirName, paths.StateFileName)
resolver, err := loadResolverFromDir(filepath.Dir(statePath))
if err != nil { return err }
if err := validateAllowedOperation(resolver, orchestratorID, "add-tasks"); err != nil {
    return err
}
```

**Pattern 4: wt-merge normalization**

`wt-merge` (`cmd_worktree.go:91-123`) currently has inline RBAC that loads the pipeline config and checks role type. This will be refactored to use `loadResolverForRBAC` + `validateRoleType(resolver, agentID, "reviewer")`, removing the inline resolver loading.

## 6. Cross-Cutting Concerns

### 6.1 Pipeline Config Loading

`validateAllowedOperation` and `validateRoleType` take a pre-loaded `*pipeline.Resolver`. Two loading helpers provide it:

- `loadResolverForRBAC(projectRoot)` — calls `pipeline.LoadFrozen(projectRoot)`, same path as `resolveOrchestratorID` (main.go:79) and `wt-merge` (cmd_worktree.go:109).
- `loadResolverFromDir(lizaDir)` — calls `pipeline.Load(filepath.Join(lizaDir, "pipeline.yaml"))`, used by `add-task`/`add-tasks` when `--state`/`--log` flags bypass project root detection. The `.liza/` directory is derived from `filepath.Dir(statePath)`.

Loading is idempotent (reads and parses `pipeline.yaml`). For commands that already load the resolver (e.g., `resolveOrchestratorID`), the double-load is acceptable: the file is small, parsing is fast, and threading the resolver through the call chain adds complexity for negligible performance gain.

### 6.2 Fail-Closed Behavior

If `pipeline.LoadFrozen` fails, the validation functions return an error that prevents command execution. This matches MCP's behavior (`middleware.go:72-77`): a nil resolver blocks all operations.

### 6.3 Agent ID Format Validation

`identity.ExtractRole` validates agent ID format as a side effect. Invalid formats (e.g., "coder" without a number suffix) produce clear errors. This is the same validation MCP performs via `identity.ValidateFormat` before role extraction.

### 6.4 No Syntax Changes

Per the spec: "No syntax changes. `claim-task` keeps its positional `<agent-id>`. The validation is internal." All commands retain their current argument/flag syntax. RBAC validation is invisible to callers when authorization succeeds.

**Note on auto-resolution test (row 5):** This is the only test that exercises the true auto-resolution path where `resolveOrchestratorID` reads state to find the orchestrator agent. Setup registers a non-orchestrator agent as the only agent in state, forcing auto-resolution to return an invalid role. This verifies RBAC catches the mismatch even when identity is auto-resolved rather than explicitly provided.

### 7.3 Testing Approach

Tests use the existing `executeRootCommand` helper (`mutation_wiring_test.go:228`) which runs commands against a temp project root with full pipeline config. The `testhelpers.SetupPipelineConfig` function provides a real pipeline YAML with role definitions, ensuring RBAC checks use production-equivalent configuration.

For RBAC rejection tests: call the command with a role that doesn't have the required permission, assert the error contains the expected rejection message, and verify no state mutation occurred.
