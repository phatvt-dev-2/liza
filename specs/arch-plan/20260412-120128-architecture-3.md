# Architecture: MCP Server Removal and Bootstrap Integration Point Replacement

**Task:** architecture-3
**Spec:** specs/goals/20260412-cli-native-access-control.md (Section 3)
**Scope:** MCP code deletion, 7 bootstrap integration point replacements, embedded config updates. Excludes prompt/doc updates (Phase 4).

## 1. Problem Summary

The MCP server (`cmd/liza-mcp/`, `internal/mcp/`) is a parallel API surface over the same ops layer as the CLI. With CLI RBAC (architecture-1) and `--json` (architecture-2) in place, MCP provides no unique value. It adds ~9,730 lines of code, a second binary to build/distribute, token overhead in every agent session, and a reliability surface that fails under memory pressure.

Seven bootstrap integration points wire `liza-mcp` into the project lifecycle (init, worktree provisioning, agent launch, settings). Each must be replaced or removed to complete the removal.

## 2. Component Boundaries

### 2.1 MCP Code Deletion (DELETED)

**Entire packages removed:**
- `cmd/liza-mcp/` — binary entrypoint (1 file, 53 lines)
- `internal/mcp/` — server, handlers, middleware, registration, protocol (18 files, ~8,900 lines)
- `internal/mcp/protocol/` — JSON-RPC transport and types (6 files, ~780 lines)
- `internal/mcp/testdata/` — test fixtures

**Not responsible for:** Any code outside these two directory trees. The ops layer (`internal/ops/`), commands layer (`internal/commands/`), identity package (`internal/identity/`), and pipeline package (`internal/pipeline/`) are unchanged.

**Deletion order:** Bottom-up to maintain compilability at each step:
1. Delete `cmd/liza-mcp/main.go` (only imports `internal/mcp` and `internal/embedded`)
2. Delete `internal/mcp/` entirely (no other package imports it)

Since `cmd/liza-mcp/` is the only consumer of `internal/mcp/`, and `internal/mcp/` is the only consumer of `internal/mcp/protocol/`, deleting them in this order ensures no dangling imports.

**Verification:** `go build ./...` compiles after deletion. No import of `internal/mcp` exists anywhere else.

## 3. Data Flow

**Before removal:**
```
Agent prompt → MCP tool call → liza-mcp binary → internal/mcp handler → ops layer → state.yaml
Agent prompt → Bash liza CLI → cmd/liza → internal/commands → ops layer → state.yaml
```

**After removal:**
```
Agent prompt → Bash liza CLI → cmd/liza → internal/commands → ops layer → state.yaml
```

Single path. No parallel surface. No MCP schema token cost.

### 4.3 Import Audit

### 4.5 Backwards Compatibility

**Breaking change:** `liza-mcp` binary ceases to exist. Any external configuration pointing to `liza-mcp` (e.g., user's `~/.codex/config.toml` per `contracts/contract-activation.md`) will fail. This is intentional — the goal spec explicitly removes MCP.

**Migration path:** Users update their Codex/Gemini/Vibe configs to remove `mcp_servers.liza` entries. `contracts/contract-activation.md` must be updated (Phase 4 scope).

### 4.6 MCP Resources

Three resources registered: `liza://state`, `liza://tasks`, `liza://agents`. Per goal spec: "These are removed with no functionality loss" — CLI equivalents exist (`liza get tasks --json`, `liza get agents --json`, `liza status --json`).

## 6. Ordering Constraints

```
Scope 0 (delete MCP code) ← no dependencies, can run first
Scope 1 (embedded + init) ← no dependency on Scope 0 (different files)
Scope 2 (settings + wt)   ← no dependency on Scope 0 or 1 (different files)
Scope 3 (Makefile + supervisor) ← no dependency on Scope 0-2 (different files)
```

All four scopes touch disjoint file sets. They can be implemented in any order or in parallel. However, for logical cleanliness and to avoid potential `go build` failures during intermediate states:
- **Scope 0 should go first** — deleting the MCP package removes the main consumer of `internal/mcp/`. If Scope 3 removes the Makefile target before Scope 0 deletes the code, `make build` would fail on the MCP target.

**Recommended order:** Scope 0 → (Scope 1 | Scope 2 | Scope 3) — Scope 0 first, then the rest in parallel.

Dependencies expressed: Scope 1, 2, 3 each depend on Scope 0.
