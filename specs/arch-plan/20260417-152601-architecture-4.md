## 1. Decision: Amend in place

**Outcome:** edit `specs/architecture/ADR/0036-structured-task-output-and-scope-extensions.md` in place. Do NOT write a superseding ADR-0058.

### 1.1 Rationale

Supersession is the repo's convention for **replacing** a decision (see ADR-0057 superseding ADR-0039 + ADR-0043 after the entire MCP server was removed). The `kind` extension is not a replacement of anything in ADR-0036 — the `output[]` channel mechanism, `liza_set_task_output`, `per-subtask` cardinality, and the entire `scope_extensions` half of ADR-0036 remain correct. Only one bullet in the "Limitations accepted" section is factually wrong and needs revising.

Five observations drive the amend decision:

1. **Narrow scope.** The only text whose semantics change is one bullet in "Limitations accepted" (`output[] entries must match the expected schema — no extensibility beyond the four fields`). Every other claim in ADR-0036 is intact: the structured `output[]` channel, `liza_set_task_output`, per-subtask consumption, `ScopeExtensionEntry`, the reader helper, the reviewer template block. Rewriting the full ADR to change one bullet would duplicate ~90% of the text without improvement.

2. **Precedent: `OutputEntry` has already grown three additive fields.** The live struct in `internal/models/task.go:262-269` is:

   ```go
   type OutputEntry struct {
       Desc      string   `yaml:"desc" json:"desc"`
       DoneWhen  string   `yaml:"done_when" json:"done_when"`
       Scope     string   `yaml:"scope" json:"scope"`
       SpecRef   string   `yaml:"spec_ref" json:"spec_ref"`
       PlanRef   string   `yaml:"plan_ref,omitempty" json:"plan_ref,omitempty"`
       ArchRef   string   `yaml:"arch_ref,omitempty" json:"arch_ref,omitempty"`
       DependsOn []string `yaml:"depends_on,omitempty" json:"depends_on,omitempty"`
   }
   ```

   That is three additive fields beyond the original four (`PlanRef`, `ArchRef`, `DependsOn`), each shipped without amending or superseding ADR-0036:
   - `DependsOn []string` — added under ADR-0048 (Multi-Phase Planning); ADR-0048 line 71 declares `**Extends:** ADR-0036 (Structured Task Output) — DependsOn in OutputEntry.`
   - `ArchRef string` — added under ADR-0056 (Architecture Step); ADR-0056 line 116 declares `**Extends:** ADR-0036 (Structured Task Output) — arch_ref on output[].`
   - `PlanRef string` — added via commit `ef80d629` ("feat(ops,prompts): add plan_ref propagation", 2026-03-17) with **no ADR link declared**. It is a silent extension not covered by any ADR.

   `ArchRef` and `PlanRef` are also present on the persisted `Task` struct (see architecture-1 §2.2); the same extensibility question therefore covers `Task` as well, not only `OutputEntry`.

   Separately — and not counted as a schema extension — ADR-0055 (Integration Sub-Pipeline) reuses the existing `output[]` channel to create fix-tasks without adding any new field; its `**Extends:** ADR-0036 ... output[] drives fix-task creation.` (ADR-0055 line 94) describes channel reuse, not schema growth. This plan keeps the two categories separate.

   None of these schema additions amended the "no extensibility beyond the four fields" bullet. They silently rendered it false. This amendment formalizes what the three field additions already established as the pattern (additive optional fields with `omitempty` defaults), documents the retroactive status of `PlanRef` (previously undocumented), and preserves ADR-0055 as an `output[]` channel-reuse precedent rather than a schema one. Removing the obsolete constraint and codifying the actual policy is correction, not redesign — the canonical signal for amend, not supersede.

3. **Supersession would orphan the intact half.** `scope_extensions` is untouched by the bootstrap goal. A superseding ADR would either duplicate the `ScopeExtensionEntry` specification (introducing drift risk — two sources of truth for the same unchanged feature) or scope the supersession narrowly to `output[]` only, leaving readers to chase two ADRs when they want the full schema policy. Neither is better than amending one bullet.

4. **"Reconstructed from commits" footer is not a freeze marker.** The ADR footer `*Reconstructed from commits 45150db, 8e75dee, df10fe7 (2026-03-05 to 2026-03-06)*` documents backfill provenance, not an immutability contract. ADR-0046 already sets a precedent for mutable footers: `*Reconstructed from commits 8663ae3..76db525 (2026-03-16), extended with 1af2159 (2026-03-21)*`. The Revision History section (§2.4) will extend the footer the same way.

5. **Historical record is preserved without supersession.** The done_when requires "preserving historical record of the original constraint and its revision." An amendment with a dedicated Revision History section keeps both the original bullet (struck through, still legible) AND the revised policy in one narrative. Git history also preserves the verbatim original. Supersession scatters the narrative across two files.

### 1.2 Alternatives considered and rejected

| Alternative | Why rejected |
|-------------|--------------|
| **Supersede with new ADR-0058** | Duplicates the unchanged `scope_extensions` half; orphans or re-scopes that feature unnecessarily. Repo convention (ADR-0057) reserves supersession for full system replacements, not one-bullet revisions. Breaks three existing `Extends: ADR-0036` cross-refs (0048, 0055, 0056) — their extension target would need rewriting to point at 0058, inflating the blast radius far beyond this task. |
| **Add another ADR that only `Extends:` ADR-0036** | Matches the pattern of ADR-0048/0055/0056 but does not remove the factually-wrong "no extensibility" bullet. Goal-spec done_when explicitly requires removing that constraint. An Extends-only ADR would make four ADRs all silently contradict one bullet of ADR-0036, worsening the stale state. |
| **Silent edit — rewrite the bullet with no revision record** | Breaks the done_when's "preserving historical record" requirement. Loss of narrative: future readers cannot tell whether the original constraint was ever intended or whether this was always the policy. |
| **Strike the bullet entirely without adding revised policy** | Leaves the schema-extensibility question unanswered. Future contributors would have no documented policy for adding further optional fields. Weaker than stating the current policy explicitly. |

---

```
## Revision History

### 2026-04-17 — Schema-extensibility policy revised

**Trigger.** The pre-commit bootstrap goal (`specs/goals/20260417-precommit-bootstrap.md`, Q2) required adding a typed marker field `kind` to `OutputEntry` (and propagated to persisted `Task`) as the stable dedup primitive. The schema change itself is specified in the architecture plan for task `architecture-1`: `specs/arch-plan/20260417-141659-architecture-1.md` (§2.1 and §2.2).

**What changed.**

The "Limitations accepted" bullet stating *"Output[] entries must match the expected schema — no extensibility beyond the four fields"* is rescinded. That constraint was always aspirational: the live `OutputEntry` struct in `internal/models/task.go:262-269` already carries three additive optional fields beyond the original four, and the persisted `Task` struct carries two of them in turn.

Actual prior `OutputEntry` schema-field additions (chronological):

| Field added | Introducing ADR | Extends link declared? |
|-------------|-----------------|------------------------|
| `DependsOn []string` | ADR-0048 (Multi-Phase Planning) | Yes — ADR-0048 line 71: `**Extends:** ADR-0036 ... DependsOn in OutputEntry.` |
| `PlanRef string` | *none* — added via commit `ef80d629` ("feat(ops,prompts): add plan_ref propagation", 2026-03-17) | No ADR link declared; retroactively acknowledged here. |
| `ArchRef string` | ADR-0056 (Architecture Step) | Yes — ADR-0056 line 116: `**Extends:** ADR-0036 ... arch_ref on output[].` |

The `kind` field (see architecture-1's arch plan §2.1 for the exact struct tag and placement, and §2.2 for the propagated field on `Task`) is the fourth such additive extension, shipping with the pre-commit bootstrap goal.

Distinct category — *channel reuse, not schema growth*: ADR-0055 (Integration Sub-Pipeline) reuses `output[]` to create fix-tasks and declares `**Extends:** ADR-0036 ... output[] drives fix-task creation.` (ADR-0055 line 94), but it adds no new field to `OutputEntry` or `Task`. It is not counted among the schema-extension precedents above; it is a channel-reuse precedent that this policy does not govern directly.

This revision formalizes the additive-field practice as policy (see Current Policy below), retroactively acknowledges `PlanRef` as a prior extension that never received its own ADR link, and records the `Kind` addition under the same policy.

**Current policy — schema extension of `OutputEntry` and `Task`.**

Additive optional fields on `OutputEntry` and the propagated persisted `Task` are permitted without superseding this ADR, subject to all of the following:

1. **Additive only.** New fields MUST be optional with a zero-value default that is behaviorally inert (`omitempty` on YAML and JSON tags; empty string or empty slice is the default).
2. **Backward-compatible on load.** State files written before the field existed must decode without error into a zero-valued field. No migration pass may be required.
3. **No required-field semantics.** An empty value MUST NOT cause `validateOutputEntry` (or any successor validator) to reject the entry. Enforcement of the new field's semantics is the responsibility of the consuming code paths, not the generic struct validator.
4. **Provenance via `Extends:` link.** The introducing ADR MUST declare `**Extends:** ADR-0036 (Structured Task Output) — <field name> on OutputEntry.` (or equivalent wording) in its Consequences section, so the cross-reference graph stays navigable without in-place edits here.
5. **No breaking removals.** Removing a field that has ever shipped requires a superseding ADR (this ADR remains the authoritative record of the additive policy). This revision does not remove any existing field.

**Historical record.** The original "no extensibility beyond the four fields" wording remains struck through in the Consequences section above, not deleted, so readers of this document can see exactly what constraint was revised.

**Non-changes.** Every other claim in this ADR remains in force: the `output[]` channel, `liza_set_task_output` write path, atomic overwrite semantics, per-subtask cardinality consumption, the `ScopeExtensionEntry` model, the checkpoint serialization path, the reviewer template block, and the `GetLatestScopeExtensions()` helper. `scope_extensions` policy is unchanged in every respect.

**Cross-references.**
- Goal: `specs/goals/20260417-precommit-bootstrap.md#q2-idempotency--plan-time-dedup-execution-time`
- Schema delta (code-side): `specs/arch-plan/20260417-141659-architecture-1.md` §2.1 (`OutputEntry`), §2.2 (`Task`), §2.3 (propagation), §2.4 (replan propagation)
- Prior extensions: ADR-0048, ADR-0055, ADR-0056
```
