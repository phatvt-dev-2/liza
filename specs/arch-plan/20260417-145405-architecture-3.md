### 1.2 Why the install-authorization clause goes in the *prompt*, not the spec only

Goal-spec line 41-44 defines the clause text. The spec is read by humans drafting the design; the architect agent reads its own prompt. Without surfacing the clause in the architect prompt, the architect would have to re-derive the wording from the spec on every emission — assumption-budget waste and a drift surface (paraphrase risk). The spec wording is canonical; the prompt **quotes it verbatim** with a "MUST include verbatim or substantively equivalent" constraint, and the spec stays the single source of truth via cross-reference.

### 1.3 Why the bootstrap `done_when` phrasing belongs in the architect prompt

The bootstrap task's `done_when` is two invariants:

- pre-commit runs cleanly: `<dep-manager-runner> pre-commit run --all-files` exits 0 against the bootstrap commit;
- pre-commit is recorded as a dev dependency in the project's dep-manager manifest.

The first invariant is stack-specific (`uv run pre-commit run …`, `npm run pre-commit …`, `python -m pre_commit run …`). The second is stack-specific (`pyproject.toml`, `package.json`, `Cargo.toml`). The architect picks the stack and therefore phrases the `done_when`. The prompt template gives the **template** ("two invariants the architect must specialize"), not the literal stack-specific text — this preserves G1.1 (stack-agnostic prompts) while ensuring the architect captures both invariants.

### 1.4 Why a separate `bootstrap_precommit_guidance.tmpl` block is **rejected**

Considered: extract bootstrap-precommit-specific guidance into a new template block, included via pipeline.yaml `context-sections`. **Rejected** because:

- Adds a new file + a pipeline.yaml edit for two paragraphs of text (G2.2: contract conciseness — growth must not exceed semantic content added).
- The architect role has exactly one consumer for this guidance today; YAGNI.
- Future generalization (`.editorconfig`, lint config) is `Out of Scope` of the goal spec; designing a shared-block surface now is speculation.

If a second bootstrap-style primitive arrives, the extraction is a one-task refactor with clear precedent. Until then, the guidance lives inside the architect branch of `implementation_phase.tmpl`.

### 1.5 Greenfield reproduction is a goal-spec acceptance criterion, not a code-planning criterion

Goal-spec acceptance criterion #1 is a **hard gate on the goal**: implementation may not land without it. It is not a per-code-planning-task gate; it is a one-shot artifact captured once for the whole goal. This plan emits the work as **two paired code-planning scopes**: scope §2 (output[2]) builds the procedure scaffolding and capture script; scope §3 (output[3], `depends_on: ["2"]`) executes the run, commits artifacts, and populates the observation table with evidence and the confirm-or-divert decision.

The populated observation table replaces the hypothesized list in goal spec §Evidence (lines 13-17). Per goal-spec §Evidence: "If reproduction shows the actual failure mode is different from any of the above, the ratified design is revisited before implementation proceeds." Therefore scope §3's `done_when` requires the confirm-or-divert decision to be committed alongside the populated cells.

**Sequencing within the goal:** scopes §0 and §1 of this plan can be implemented in parallel with scopes §2 and §3, but the goal-level acceptance criterion #1 means **none of the code from any goal-wide architecture task may MERGE into integration before scope §3 (output[3]) reaches MERGED.** That cross-task ordering is the orchestrator's responsibility to enforce (via dependency wiring or an explicit gate — see §5.7). This plan documents the requirement; it does not encode the cross-architect dependency in `depends_on` (which is sibling-only).

## 5. Scopes §2 + §3 — Greenfield Reproduction (Scaffolding + Execution)

### 5.0 Why two scopes

Acceptance criterion #1 has two distinct deliverables: (a) a reproducible procedure and capture infrastructure, (b) the actual run that produces the evidence the criterion demands. Collapsing them into one task would make the task non-atomic (planning + live evidence-gathering in one unit) and would make `done_when` ambiguous about whether the harness or the evidence is the deliverable. Splitting:

- **§2 (output[2])** — scaffolding. Atomic, mostly-mechanical: writes the procedure text, the script, and the placeholder table. Reviewable from the diff alone.
- **§3 (output[3])** — execution. Depends on §2. Runs the procedure (live agent runs against a synthetic greenfield project), captures artifacts, populates the observation table with evidence pointers, and commits the confirm-or-divert decision. Reviewable by inspecting the populated table and the committed artifact directory.

§3 owns acceptance criterion #1 — the "no output owns the hard gate" failure mode is closed.

### 5.1 Intent

**Together, scopes §2 and §3 deliver a procedure that:**

- Sets up a greenfield project (no `.pre-commit-config.yaml`, no `.liza/`, no pre-commit binary on PATH inside the project sandbox) and runs the goal pipeline through architect → code-planner → coder.
- Captures observed failure modes against the four hypothesized modes in goal-spec lines 14-17.
- Records observations in a designated location that supersedes the hypothesized list.
- Provides a confirm-or-divert decision artifact that gates implementation per acceptance criterion #1.

### 5.2 Procedure document location and shape

### 5.3 Procedure steps (verbatim text the downstream task writes into §Setup, §Execution, §Capture)

All commands are written to comply with project BASH CONSTRAINTS (see `internal/prompts/templates/base_prompt.tmpl`): no `$()` command substitution, no `cd` combined with `git`, no `git add -A`, no `!` in double-quoted strings. Multi-step capture is encapsulated in the script (§5.4), invoked once by the agent. The operator (or output[3]'s coder agent) sets `REPRO_ROOT` explicitly in advance — no command substitution.

**§Setup:**

1. Choose a scratch directory path outside any liza-managed worktree and export it. Recommended pattern: a date-suffixed path under `/tmp` or another writable scratch area. Example: `export REPRO_ROOT=/tmp/liza-precommit-repro-20260417-001`. Use any unique path; do NOT use command substitution to derive it. Create both project roots: `mkdir -p "$REPRO_ROOT/proj/specs/vision"` then `mkdir -p "$REPRO_ROOT/proj-parallel/specs/vision"`.
2. Initialize the primary greenfield project (no `cd`; use `git -C` and absolute paths throughout):
   - `git init "$REPRO_ROOT/proj"`
   - Use a file-writing tool (not shell heredoc) to create `"$REPRO_ROOT/proj/README.md"` (single line, e.g. "greenfield repro project") and `"$REPRO_ROOT/proj/specs/vision/greenfield.md"` (vision describing one trivial feature, e.g. "add a `hello` script printing 'hi'").
   - Stage explicitly by name: `git -C "$REPRO_ROOT/proj" add README.md specs/vision/greenfield.md`
   - Commit using a heredoc on stdin (the contract-permitted form for git commits):
     ```
     git -C "$REPRO_ROOT/proj" commit -F - <<'EOF'
     initial: README + greenfield vision
     EOF
     ```
   - Repeat the four bullets above for `"$REPRO_ROOT/proj-parallel"` (substituting `proj-parallel` for `proj` in every path).
3. Initialize liza in each project: `liza init "$REPRO_ROOT/proj"` then `liza init "$REPRO_ROOT/proj-parallel"`. Verify each project's state independently (one `test` invocation per assertion — do NOT chain with `&&` across the boundary):
   - `test -f "$REPRO_ROOT/proj/.liza/state.yaml"` (exit 0 expected)
   - `test ! -f "$REPRO_ROOT/proj/.pre-commit-config.yaml"` (exit 0 expected)
   - Same two `test` commands against `"$REPRO_ROOT/proj-parallel"`.
   - Confirm `command -v pre-commit` outside any project-scoped venv either returns nothing (exit 1) or returns a path the procedure agrees to ignore for this run.
4. Configure verbose logging for both supervisor cycles: `export LIZA_LOG_LEVEL=debug` and `export LIZA_TEE_AGENT_OUTPUT=1` (or the current equivalent — `liza --help` lists the active env-var names; the capture script in §5.4 picks up whichever log/tee paths the supervisor wrote to).

**§Execution:**

5. Start the supervisor for the primary cycle: `liza supervise "$REPRO_ROOT/proj"` (consult `liza --help` for the current subcommand name; substitute the equivalent if `supervise` is renamed). Run it under a separate terminal or as a background job (`liza supervise "$REPRO_ROOT/proj" > "$REPRO_ROOT/proj/supervisor.stdout.log" 2>&1 &`). Allow it to run until either (a) the integration task reaches a terminal state, or (b) any task transitions to BLOCKED with a reason mentioning pre-commit, missing config, or install authorization.
6. Start the parallel cycle the same way: `liza supervise "$REPRO_ROOT/proj-parallel" > "$REPRO_ROOT/proj-parallel/supervisor.stdout.log" 2>&1 &`. The two supervisor processes operate against fully independent project trees (separate `state.yaml`, separate `.worktrees/`) so they cannot cooperate.
7. Wait for both cycles to reach a terminal state. Inspection commands (each is a single contract-compliant invocation — no chaining):
   - `liza --project "$REPRO_ROOT/proj" status --json`
   - `liza --project "$REPRO_ROOT/proj-parallel" status --json`
   - Repeat at intervals (operator's discretion) until both report a terminal sprint state or all tasks BLOCKED.

**§Capture:**

8. Run the capture script once per cycle, passing `REPRO_ROOT` and the project subdirectory name as separate arguments: `bash scripts/repro/precommit-bootstrap-greenfield.sh "$REPRO_ROOT" proj` then `bash scripts/repro/precommit-bootstrap-greenfield.sh "$REPRO_ROOT" proj-parallel`. Each invocation is a single shell command — internal multi-step capture happens inside the script, so BASH CONSTRAINTS apply only to the invocation form (which is plain).
9. The script writes per-cycle artifacts under `"$REPRO_ROOT/observations/<cycle>/"` (where `<cycle>` is `proj` or `proj-parallel`), containing at minimum:
   - `state-snapshots/state-postinit.yaml`, `state-snapshots/state-postarchitect.yaml`, `state-snapshots/state-final.yaml` (the script copies `.liza/state.yaml` at three checkpoints — operator triggers the snapshots by re-running the script with a snapshot subcommand, OR the supervisor's own audit hooks if available).
   - `agent-outputs/` — full copy of `<cycle>/.liza/agent-outputs/`.
   - `supervisor.stdout.log` — copy of the supervisor's stdout/stderr capture from step 5/6.
   - `worktree-git-logs/<worktree-id>.log` — output of `git -C "$REPRO_ROOT/<cycle>/.worktrees/<worktree-id>" log --all --oneline --graph` for each worktree.
   - `prompts/` — copies of any rendered-prompt files the supervisor wrote (e.g. under `.liza/agent-prompts/` if `LIZA_TEE_AGENT_OUTPUT=1`).
   - `precommit-config-presence.txt` — output of `git -C "$REPRO_ROOT/<cycle>" ls-tree HEAD -- .pre-commit-config.yaml` (empty stdout = absent at HEAD; non-empty = present).

### 5.4 Capture script

**Location:** `scripts/repro/precommit-bootstrap-greenfield.sh` (new). POSIX bash script. **Interface:** `bash scripts/repro/precommit-bootstrap-greenfield.sh <REPRO_ROOT> <cycle>` where `<cycle>` is the project subdirectory name (`proj` or `proj-parallel`). Optional third argument: `<phase>` ∈ `{setup, snapshot, capture}` to allow targeted invocations; default `capture`.

**Why a script (not a stepwise procedure with chained shell):** chained shell forms (`&&`, `$()`) are forbidden by BASH CONSTRAINTS for the agent's direct invocations. A script file encapsulates the multi-step logic in a single agent-visible command. Agent-visible commands stay simple (`bash scripts/repro/...sh "$REPRO_ROOT" proj`); BASH CONSTRAINTS govern those invocations and are satisfied. The script's internals are conventional shell — they execute inside `bash`, not under the contract-checked CLI surface.

**Responsibility:** automate Setup (steps 1-4) when invoked with `<phase>=setup`, snapshot `state.yaml` when invoked with `<phase>=snapshot`, and Capture (step 9 artifacts) when invoked with `<phase>=capture`. The Execution steps (5-7) require live supervisor runs and remain manual — the script does not start or stop the supervisor.

**Script outline (the downstream code-planning task writes the implementation):**

- Validate args: `$1` (REPRO_ROOT) is set and writable; `$2` (cycle) is non-empty; `$3` (phase) is one of `{setup, snapshot, capture}` or empty (defaulting to `capture`).
- `setup` phase: `mkdir -p "$REPRO_ROOT/$cycle/specs/vision"`, write README and vision-doc placeholders if absent (operator may overwrite), `git init "$REPRO_ROOT/$cycle"` if `.git` absent, stage and commit via heredoc-on-stdin per §5.3 step 2.
- `snapshot` phase: `cp "$REPRO_ROOT/$cycle/.liza/state.yaml" "$REPRO_ROOT/observations/$cycle/state-snapshots/state-<UTC-timestamp>.yaml"` (the script computes a UTC timestamp via `date -u +%Y%m%dT%H%M%SZ` invoked in a child process — that *is* command substitution, but it lives inside the script, not in the agent's shell command, so contract is preserved).
- `capture` phase: copy `.liza/agent-outputs/` and `.liza/agent-prompts/` (if present) into `"$REPRO_ROOT/observations/$cycle/agent-outputs/"` and `"$REPRO_ROOT/observations/$cycle/prompts/"`; copy `"$REPRO_ROOT/$cycle/supervisor.stdout.log"` if present; for each subdirectory under `"$REPRO_ROOT/$cycle/.worktrees/"`, run `git -C ... log --all --oneline --graph` and write to `worktree-git-logs/<id>.log`; run `git -C "$REPRO_ROOT/$cycle" ls-tree HEAD -- .pre-commit-config.yaml` and write stdout to `precommit-config-presence.txt`.
- Emit a per-cycle summary `observations-summary.txt` listing artifact paths and reminding the operator to fill in the corresponding rows in the goal-spec table (output[3] reads this list when populating cells).

**Hard constraints:**

- Script must not install anything host-level. It uses only `git`, `mkdir`, `cp`, `find`, `date`, `test`, plus standard POSIX utilities.
- Script is read-only against the operator's actual liza state; it operates entirely under `$REPRO_ROOT`.
- Script exit code 0 means "phase completed and artifacts written"; non-zero means a step failed (operator inspects which). Phases are idempotent — re-running `capture` overwrites prior captures.
- Script body may use any standard shell features (including `$(...)` and `&&`) since BASH CONSTRAINTS govern the *agent's* shell invocations, not the script file's contents. This separation is documented in §5.4 itself so reviewers do not flag the script body.

### 5.5 Observation recording

**Location:** the goal spec gains a new section **"Observed Failure Modes — Reproduction Run YYYY-MM-DD"** appended after the new "Greenfield Reproduction Procedure" section. The section structure:

```markdown
## Observed Failure Modes — Reproduction Run YYYY-MM-DD

Reproduction artifacts: `<path under repo, e.g. specs/goals/precommit-bootstrap-repro-artifacts/>` (committed as part of this run).

| # | Hypothesized mode (goal spec §Evidence, lines 14-17) | Observed? (yes/no/partial) | Evidence pointer | Notes |
|---|---|---|---|---|
| 1 | Coders inventing `.pre-commit-config.yaml` ad-hoc inside an unrelated coding task | … | `<artifact path>:<line>` | … |
| 2 | Parallel coders producing conflicting configs | … | … | … |
| 3 | Coders silently skipping pre-commit when it errors with "no config" | … | … | … |
| 4 | DoD step "Pre-commit passes on touched files" (CORE.md Rule 3) becoming vacuous | … | … | … |

### Newly observed modes (not in the hypothesized list)
<list of any additional failure modes discovered during reproduction; if none, write "None observed.">

### Confirm-or-Divert Decision
<one paragraph: either "Hypothesized modes confirmed; design ratified as written" or "Observed modes diverge from hypothesis: <summary>; design revisited at <link to revised section / new spec>.">
```

The "Newly observed modes" subsection is mandatory — even an empty "None observed" line is required so the absence is explicit.

### 5.6 Reproduction-artifacts commit policy

The captured artifacts (state snapshots, agent-outputs copies, logs) live under `specs/goals/precommit-bootstrap-repro-artifacts/<YYYY-MM-DD>/`. They are committed to the repo as part of the reproduction-run code-planning task's deliverable. Rationale: future goal revisits and reviewers inspect the same evidence the original decision rested on.

**Sanitization:** before commit, the operator scrubs absolute filesystem paths (`/home/<user>/...`) by replacing the user-home prefix with `~`. No secrets are expected in the captured material — the procedure runs against a synthetic vision doc and a fresh git identity — but the downstream coding task adds a pre-commit check (or a manual review checklist) confirming no `.env`, `*.key`, or `credentials*` files appear under the artifacts directory.

**Risks specific to §3:**

- **Risk: live LLM cost / non-determinism.** Live runs cost API budget and may require multiple iterations to produce a clean capture. Mitigation: §Capture allows up to 5 cycles; if the first cycle is inconclusive on any row, the operator iterates.
- **Risk: divert outcome blocks the goal.** If the run shows the design needs revision, every other goal-wide coding task waits. This is the **intended** behavior per goal-spec line 19 ("the ratified design is revisited before implementation proceeds"). The orchestrator handles divert by re-planning the affected scopes.
- **Risk: artifact directory grows large.** Captured logs and agent-outputs may be hundreds of KB. Acceptable — repo growth is bounded (one-off artifact per goal) and traceability value is high. If size becomes a concern in practice, future goals can move artifacts to LFS or external storage.

### 5.10 Shared risks across §2 and §3

- **Risk: live LLM runs are non-deterministic.** Two runs may produce different observed modes. Mitigation: the spec section accepts "yes/no/partial" per row and documents `<= 5` reproduction cycles in §Capture if the first cycle is inconclusive on any row. The procedure does not require a single deterministic run; it requires evidence-backed observations.
- **Risk: parallel runs collide on shared system state.** Two supervisor processes against two scratch repos under one `$REPRO_ROOT` should be filesystem-isolated, but the two processes share `~/.local/share/rtk/tee/`. Mitigation: the capture step records tee files by timestamp window per cycle; cycle-id is part of the artifact path.
- **Risk: reproduction does not exhibit any of the four hypothesized modes.** Per goal-spec line 19 ("If reproduction shows the actual failure mode is different from any of the above, the ratified design is revisited"), this is an explicit allowed outcome. The Confirm-or-Divert subsection (§5.5) handles it; output[3]'s done_when accepts both confirm and divert paragraphs.
- **Non-risk: missing pre-commit binary.** The whole point of the reproduction is to exercise greenfield where the binary is absent. The procedure does not pre-install anything; absence is part of the test.
