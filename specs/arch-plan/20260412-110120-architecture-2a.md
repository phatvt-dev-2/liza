# Architecture: Structured JSON Output via `--json` Flag

**Task:** architecture-2a
**Spec:** specs/goals/20260412-cli-native-access-control.md (Section 2)
**Scope:** `--json` flag on finite CLI commands, error code mapping, typed result serialization. Planning only.

## 1. Problem Summary

The CLI currently formats output for human consumption: success results printed as text, errors returned through Cobra (stderr + exit 1). Agents parse this text output, which is fragile and lossy. The MCP server provides structured JSON responses with typed error codes, but is being removed (Phase 3). The `--json` flag brings equivalent structured output to the CLI, making it the complete agent interface.

## 2. Component Boundaries

### 2.1 JSON Output Package (NEW: `internal/jsonout/`)

**Responsibility:** Provide a standard JSON envelope format, error classification, and output helpers for CLI commands. All JSON serialization for `--json` mode flows through this package.

**Not responsible for:** Flag registration (CLI layer), command-specific result construction (commands/ops layers), human-formatted output (existing code paths).

**Public API:**

| Symbol | Purpose |
|---|---|
| `Envelope` | Standard JSON response wrapper |
| `ErrorDetail` | Error code + message in error responses |
| `ClassifyError(err) (code, message)` | Maps Go errors to string error codes (extracted from MCP's `classifyError`) |
| `WriteResult(w, result, warnings, err)` | Writes success or error envelope; returns `ErrAlreadyWritten` on error |
| `ErrAlreadyWritten` | Sentinel error indicating JSON error was already written to stdout |

**Design rationale:** A dedicated package (not inline in `cmd/liza/`) because error classification logic is substantial (typed error checks + string fallbacks) and deserves unit tests independent of CLI wiring. The package is small (~150 lines) and has a clear boundary: envelope formatting + error classification.

### 2.2 CLI Helpers (MODIFIED: `cmd/liza/main.go`)

**Change:** Add `addJSONFlag(cmd)`, `isJSON(cmd)`, and modify `main()` to handle `ErrAlreadyWritten`.

**Pattern:** Mirrors existing `addAgentIDFlag(cmd)` / `addChangedByFlag(cmd)` pattern for flag registration.

### 2.3 Command Layer (MODIFIED: `cmd/liza/cmd_*.go`)

**Change:** Each included command's `RunE` gains a JSON branch. When `--json` is set:
1. A deferred error guard catches any error and writes a JSON error envelope to stdout.
2. The success path calls the ops layer directly and writes a JSON success envelope.
3. The non-JSON path remains unchanged (calls `commands.*Command()` as before).

**Unchanged:** The `commands.*Command()` functions in `internal/commands/`. These remain the human-formatted output path. No signature changes.

### 2.4 Existing Components (UNCHANGED)

| Component | Role in `--json` |
|---|---|
| `internal/ops/` | Returns typed result structs. No changes needed. |
| `internal/commands/` | Human-formatted output path. Called only when `--json` is not set. |
| `internal/render/` | `FormatJSON` used by existing `--format json` on `get`/`status`. Not used by `--json` envelope. |
| `internal/errors/` | `NotFoundError` type used by `ClassifyError`. No changes. |

## 3. Interface Contracts

### 3.1 JSON Envelope Format

All `--json` output follows this envelope on stdout:

**Success:**
```json
{
  "ok": true,
  "result": { ... },
  "warnings": ["warning 1", "warning 2"]
}
```

**Error:**
```json
{
  "ok": false,
  "error": {
    "code": "validation",
    "message": "task not found"
  }
}
```

**Fields:**
- `ok` (bool, always present): Discriminant for success/error.
- `result` (object|null): Typed result on success. `null` for void-success commands (write-checkpoint, set-task-output, set-discovery-disposition).
- `warnings` (string array, optional): Present only when non-empty. Contains non-fatal warnings that would otherwise go to stderr or `log.Printf`.
- `error` (object): Present only when `ok` is false. Contains `code` (string error code) and `message` (human-readable description).

**Invariant:** When `--json` is set, stdout contains exactly one JSON object (the envelope). No other output is written to stdout. stderr is silenced: ops-level `log.Printf` calls are suppressed via `log.SetOutput(io.Discard)` (Section 6.7), and command-level warnings (status resolver, validate, suspicious rates) are captured in the `warnings` field instead of being logged to stderr.

### 3.2 Go Types

```go
// internal/jsonout/envelope.go

// Envelope is the standard JSON response wrapper for --json mode.
type Envelope struct {
    OK       bool         `json:"ok"`
    Result   any          `json:"result,omitempty"`
    Warnings []string     `json:"warnings,omitempty"`
    Error    *ErrorDetail `json:"error,omitempty"`
}

// ErrorDetail contains structured error information.
type ErrorDetail struct {
    Code    string `json:"code"`
    Message string `json:"message"`
}

// ErrAlreadyWritten is a sentinel error indicating that a JSON error envelope
// has already been written to stdout. main() uses this to exit 1 without
// printing a duplicate error to stderr.
var ErrAlreadyWritten = errors.New("json output already written")

// WriteResult writes a JSON envelope to w. On error, writes an error envelope
// and returns ErrAlreadyWritten. On success, writes a success envelope and
// returns nil.
func WriteResult(w io.Writer, result any, warnings []string, err error) error

// ClassifyError maps a Go error to a string error code and safe message.
func ClassifyError(err error) (code string, message string)
```

### 3.3 Error Code Mapping

Extracted from MCP's `classifyError` (`internal/mcp/server_protocol.go:197-247`), adapted from numeric JSON-RPC codes to string codes per the goal spec:

| Go Error Type | String Code | Message Strategy |
|---|---|---|
| `*errors.NotFoundError` | `not_found` | `"resource not found"` |
| `*ops.PreconditionError` | `validation` | `err.Reason` (controlled field) |
| `*ops.PostWriteValidationError` | `validation` | `"validation failed: precondition not met"` |
| `*ops.IntegrationFailedError` | `validation` | `"integration failed: <reason>"` |
| `*ops.OperationalError` | `internal` | `err.Message` (controlled field) |
| Lock timeout (string match) | `lock_timeout` | `"lock acquisition timed out"` |
| Race condition (string match) | `race_condition` | `"state changed concurrently, retry"` |
| Validation patterns (string match) | `validation` | `"validation failed"` |
| Default | `internal` | `"internal error"` |

**Message safety:** Typed errors expose controlled fields (e.g., `PreconditionError.Reason`). Untyped errors use fixed messages — `err.Error()` is never exposed directly to prevent leaking implementation details. This matches MCP's existing behavior.

**String match patterns** (from `stringErrorRules`):
- Not-found: `"not found"`, `"does not exist"`
- Race condition: `"race condition"`, `"changed concurrently"`
- Validation: `"not IMPLEMENTING"`, `"not REVIEWING"`, `"must be"`, `"is required"`, `"invalid task ID"`, `"validation failed"`, `"must include"`, `"mandatory"`, and other status-check patterns

**MCP-specific error types excluded:** `NotInitializedError`, `RoleError`, `InputShapeError` are MCP-layer errors that don't occur in CLI context. They are not included in `ClassifyError`.

### 3.4 `--json` Flag Registration

```go
// cmd/liza/main.go

// addJSONFlag registers --json on a specific command.
func addJSONFlag(cmd *cobra.Command) {
    cmd.Flags().Bool("json", false, "output result as structured JSON")
}

// isJSON returns true if --json flag is set on the command.
func isJSON(cmd *cobra.Command) bool {
    v, _ := cmd.Flags().GetBool("json")
    return v
}
```

Registered per-command in `init()` blocks (not as a PersistentFlag on root). This prevents `--json` from appearing on excluded commands (`agent`, `tui`, `init`, etc.).

### 3.5 Deferred Error Guard Pattern

Each included command's `RunE` uses a deferred error guard to catch all errors (pre-ops, RBAC, ops) and format them as JSON:

```go
RunE: func(cmd *cobra.Command, args []string) (retErr error) {
    if isJSON(cmd) {
        // Suppress ops-level log output in JSON mode (see Section 6.7)
        log.SetOutput(io.Discard)
        defer log.SetOutput(os.Stderr)
        defer func() {
            if retErr != nil && !errors.Is(retErr, jsonout.ErrAlreadyWritten) {
                _, _ = jsonout.WriteResult(os.Stdout, nil, nil, retErr)
                retErr = jsonout.ErrAlreadyWritten
            }
        }()
    }

    // ... existing argument resolution, RBAC validation ...

    if isJSON(cmd) {
        result, err := ops.SomeOperation(...)
        return jsonout.WriteResult(os.Stdout, result, nil, err)
    }
    return commands.SomeCommand(...)
}
```

**How it works:**
- Log suppression: `log.SetOutput(io.Discard)` prevents ops-level `log.Printf` calls from writing to stderr. Restored via defer. Safe because CLI commands are single-process (see Section 6.7).
- On success: `jsonout.WriteResult` writes the success envelope and returns `nil`. The defer is a no-op.
- On ops error: `jsonout.WriteResult` writes the error envelope and returns `ErrAlreadyWritten`. The defer sees `ErrAlreadyWritten` and skips.
- On pre-ops error (requireProjectRoot, RBAC): the error propagates to `retErr`. The defer catches it, writes a JSON error envelope, and replaces `retErr` with `ErrAlreadyWritten`.

### 3.6 `main()` Modification

```go
func main() {
    if err := rootCmd.Execute(); err != nil {
        if !errors.Is(err, jsonout.ErrAlreadyWritten) {
            fmt.Fprintf(os.Stderr, "Error: %v\n", err)
        }
        os.Exit(1)
    }
}
```

When `ErrAlreadyWritten` is returned, the JSON error is already on stdout. `main()` exits 1 without printing to stderr, satisfying the stdout-only JSON contract.

## 4. Data Flow

### 4.1 Command Execution Flow with `--json`

**Non-JSON (unchanged):**
```
CLI args → identity/RBAC → commands.*Command() → ops layer → print to stdout → exit 0
                                                     |
                                                     └→ error → Cobra stderr → exit 1
```

**JSON success:**
```
CLI args → identity/RBAC → ops layer → jsonout.WriteResult(stdout, result, warnings, nil)
                                              ↓
                                        {"ok": true, "result": {...}} → exit 0
```

**JSON error (ops layer):**
```
CLI args → identity/RBAC → ops layer → error → jsonout.WriteResult(stdout, nil, nil, err)
                                                       ↓
                                                 {"ok": false, "error": {...}} → exit 1
```

**JSON error (pre-ops, e.g. RBAC):**
```
CLI args → identity/RBAC → error → deferred guard → jsonout.WriteResult(stdout, nil, nil, err)
                                                            ↓
                                                      {"ok": false, "error": {...}} → exit 1
```

### 4.2 Interaction with Existing `--format json`

Two commands (`get`, `status`) already have `--format json` that produces raw JSON output.

| Flag | Behavior | Use Case |
|---|---|---|
| `--format json` | Raw JSON, no envelope. Existing behavior. | Human scripting, `jq` pipelines |
| `--json` | Envelope-wrapped JSON with error codes. New behavior. | Agent consumption |

Both flags can coexist. When `--json` is set, it takes precedence: the result field contains the same data that `--format json` would produce, but wrapped in the standard envelope. When only `--format json` is set, existing behavior is preserved.

For `get`, `--json` wraps the `InspectCommand` result in the envelope. For `status`, `--json` wraps `statusData` in the envelope with warnings.

## 6. Special Case Handling

### 6.1 `update-sprint-metrics --json`: Typed SprintMetrics Result

**Current behavior (`internal/commands/update_sprint_metrics.go`):**
1. Calls `ops.UpdateSprintMetrics(projectRoot)` → returns `models.SprintMetrics`
2. Prints formatted text (e.g., `"Tasks done: 5"`)
3. Calls `ops.CheckSuspiciousRates(metrics)` → returns `[]string` warnings
4. Prints warnings to stdout

**`--json` behavior:**
1. Calls `ops.UpdateSprintMetrics(projectRoot)` → returns `models.SprintMetrics`
2. Calls `ops.CheckSuspiciousRates(metrics)` → returns `[]string` warnings
3. Writes envelope with `models.SprintMetrics` as result and warnings in warnings field

```json
{
  "ok": true,
  "result": {
    "tasks_done": 5,
    "tasks_in_progress": 2,
    "tasks_blocked": 1,
    "iterations_total": 12,
    "review_cycles_total": 8,
    "review_verdict_approvals": 6,
    "review_verdict_rejections": 2,
    "review_verdict_count": 8,
    "review_verdict_approval_rate_percent": 75,
    "task_submitted_for_review_count": 7,
    "task_outcome_approval_rate_percent": 71
  },
  "warnings": ["review verdict approval rate is 97% (29/30) - suspiciously high"]
}
```

**Implementation in RunE:**
```go
if isJSON(cmd) {
    metrics, err := ops.UpdateSprintMetrics(projectRoot)
    if err != nil {
        return err // deferred guard handles JSON
    }
    warnings := ops.CheckSuspiciousRates(metrics)
    jsonout.WriteResult(os.Stdout, metrics, warnings, nil)
    return nil
}
```

The `models.SprintMetrics` struct already has `yaml:"..."` tags. JSON tags need to be added for stable field names (using the same snake_case convention as YAML). This is the code planner's responsibility to audit.

**Key requirement:** The result is the full `models.SprintMetrics` struct — not a void success. This enables downstream consumers (agents, dashboards) to use the computed metrics programmatically.

### 6.2 `status --json`: Warnings in Result Payload

**Current behavior (`internal/commands/status.go:155-157`):**
```go
pr, prErr := ops.LoadResolverForModels(projectRoot)
if prErr != nil {
    log.Printf("WARNING: status: failed to load pipeline resolver: %v", prErr)
}
```

The warning goes to stderr via `log.Printf`. With `--json`, the warning must be captured and included in the envelope's `warnings` field instead.

**`buildStatusData` boundary analysis:**

`buildStatusData(state, detailed, projectRoot)` uses `projectRoot` in **three** places:
1. `ops.LoadResolverForModels(projectRoot)` — resolver loading (line 155)
2. `buildOrchestratorStatus(state, projectRoot)` — orchestrator wake trigger detection (line 161)
3. `ops.AvailableManualTransitions(&task, projectRoot)` — per-task transition availability (line 165)

`projectRoot` cannot be removed from the signature. The refactor targets only the resolver-loading seam: extract resolver loading to the caller so warnings can be intercepted, while `projectRoot` remains for the other two uses.

**Refactored signature:**
```go
func buildStatusData(state *models.State, detailed bool, projectRoot string,
    pr models.PipelineResolver, prWarnings []string) statusData
```

The function no longer calls `ops.LoadResolverForModels` internally. It receives the pre-loaded resolver and any warnings from the caller. Both JSON and non-JSON callers load the resolver themselves:

```go
// In StatusCommand (non-JSON path):
pr, prErr := ops.LoadResolverForModels(projectRoot)
if prErr != nil {
    log.Printf("WARNING: status: failed to load pipeline resolver: %v", prErr)
}
status := buildStatusData(state, detailed, projectRoot, pr, nil)
```

```go
// In RunE (JSON path):
pr, prErr := ops.LoadResolverForModels(projectRoot)
var warnings []string
if prErr != nil {
    warnings = append(warnings, fmt.Sprintf("failed to load pipeline resolver: %v", prErr))
}
data := buildStatusData(state, detailed, projectRoot, pr, warnings)
jsonout.WriteResult(os.Stdout, data, warnings, nil)
```

**Key:** `projectRoot` stays in the signature for `buildOrchestratorStatus` and `ops.AvailableManualTransitions`. Only the resolver-loading call moves to the caller.

**Result envelope:**
```json
{
  "ok": true,
  "result": {
    "goal": { "description": "...", "status": "IN_PROGRESS", "spec_ref": "..." },
    "sprint": { "id": "...", "number": 1, "status": "IN_PROGRESS", ... },
    "config": { "mode": "RUNNING" },
    "tasks": { "total": 10, "active": 5, "terminal": 5, "by_status": {...}, ... },
    "agents": [ ... ],
    "orchestrator_state": { "trigger": "NONE", ... },
    "work_queues": { "coder": {...}, "reviewer": {...} }
  },
  "warnings": ["failed to load pipeline resolver: file not found"]
}
```

The `statusData` struct already has `json:"..."` tags on all fields. No additional tagging needed.

### 6.3 `get --json` vs `get --format json`

**`--format json`** (existing): Calls `commands.InspectCommand(args, opts)` with `Format: "json"`. Returns raw JSON — no envelope, no error codes.

**`--json`** (new): Calls `commands.InspectCommand(args, opts)` with `Format: "json"`. The result string is parsed into `any` (since it's already JSON) and wrapped in the envelope. On error, the error is classified and wrapped.

**Implementation:**
```go
if isJSON(cmd) {
    opts.Format = "json" // force JSON format for the data
    resultStr, err := commands.InspectCommand(args, opts)
    if err != nil {
        return err // deferred guard handles JSON
    }
    var parsed any
    json.Unmarshal([]byte(resultStr), &parsed)
    jsonout.WriteResult(os.Stdout, parsed, nil, nil)
    return nil
}
```

### 6.5 `validate --json`

Validate returns `nil` on success (valid state) or `error` on validation failure. It also writes non-fatal warnings to `warnWriter` (defaults to `os.Stderr`). In JSON mode, the `warnWriter` must be redirected to a buffer to capture warnings for the envelope (see Section 6.7).

```go
if isJSON(cmd) {
    var warnBuf bytes.Buffer
    commands.SetWarnWriter(&warnBuf)
    defer commands.SetWarnWriter(os.Stderr)

    err := commands.ValidateCommand(statePath, skipSpecCheck)
    var warnings []string
    if warnBuf.Len() > 0 {
        for _, line := range strings.Split(strings.TrimSpace(warnBuf.String()), "\n") {
            if line != "" {
                warnings = append(warnings, line)
            }
        }
    }
    if err != nil {
        return err // deferred guard classifies as "validation" error
    }
    return jsonout.WriteResult(os.Stdout, map[string]bool{"valid": true}, warnings, nil)
}
```

This requires adding `SetWarnWriter(w io.Writer)` to `internal/commands/validate.go` — a trivial setter for the existing package-level `warnWriter` variable.

### 6.6 `clear-stale-review-claims --json`

if isJSON(cmd) {

### 6.7 Cross-Cutting stderr/log Handling in JSON Mode

**Problem:** Multiple code paths write to stderr via `log.Printf` or `io.Writer` during command execution. The `--json` contract requires stdout to contain exactly one JSON object. While `log.Printf` writes to stderr (not stdout) and thus doesn't corrupt JSON parsing, agents may treat any stderr output as an error signal. A deliberate strategy is needed.

**Audit of stderr-producing paths in included commands:**

| Source | Location | Nature |
|---|---|---|
| `log.Printf("WARNING: status: ...")` | `internal/commands/status.go:157` | Command-level warning (resolver load failure) |
| `warnWriter` (defaults to `os.Stderr`) | `internal/commands/validate.go:14` | Non-fatal validation warnings (e.g., expired agent leases) |
| `log.Printf("Watcher error, ...")` | `internal/ops/await_verdict.go:181` | Operational: fsnotify fallback to polling |
| `log.Printf("Watcher error, ...")` | `internal/ops/await_resubmission.go:164` | Operational: fsnotify fallback to polling |
| `log.Printf("WARNING: claim-task ...")` | `internal/ops/claim_task.go:218,295,380` | Operational: worktree cleanup, race recovery |
| `log.Printf("wt-merge ...")` | `internal/ops/wt_merge.go:159,312,389+` | Operational: CAS retry, worktree recovery |
| `log.Printf("WARNING: ...")` | Various ops files | Operational: recovery paths, non-fatal issues |

**Strategy: suppress ops-level logs, capture command-level warnings.**

In JSON mode, the deferred error guard suppresses `log` output for the duration of the command:

```go
if isJSON(cmd) {
    defer func() {
        // ... existing error guard ...
    }()
    // Suppress log output in JSON mode — ops-level logs are operational
    // noise that would pollute stderr. Command-level warnings are captured
    // explicitly in the warnings field.
    log.SetOutput(io.Discard)
    defer log.SetOutput(os.Stderr)
}
```

This is safe because CLI commands are single-process, single-goroutine at the command level. `log.SetOutput` is restored via defer before the process exits.

**Command-level warnings handled explicitly:**

| Command | Warning Source | Capture Mechanism |
|---|---|---|
| `status` | Resolver load failure | Caller intercepts `prErr`, adds to warnings (Section 6.2) |
| `update-sprint-metrics` | Suspicious approval rates | `ops.CheckSuspiciousRates()` returns `[]string`, added to warnings (Section 6.1) |
| `validate` | Non-fatal validation warnings | Redirect `warnWriter` to a `bytes.Buffer` in JSON mode, include captured output as warnings |

**`validate` warnWriter handling:**

The `validate` command uses a package-level `warnWriter` variable (defaults to `os.Stderr`). In JSON mode, redirect to a buffer to capture warnings:

```go
if isJSON(cmd) {
    var warnBuf bytes.Buffer
    commands.SetWarnWriter(&warnBuf) // set warnWriter to buffer
    defer commands.SetWarnWriter(os.Stderr) // restore

    err := commands.ValidateCommand(statePath, skipSpecCheck)
    var warnings []string
    if warnBuf.Len() > 0 {
        // Split buffer into lines, add as warnings
        for _, line := range strings.Split(strings.TrimSpace(warnBuf.String()), "\n") {
            if line != "" {
                warnings = append(warnings, line)
            }
        }
    }
    if err != nil {
        return err // deferred guard handles JSON error
    }
    return jsonout.WriteResult(os.Stdout, map[string]bool{"valid": true}, warnings, nil)
}
```

This requires adding a `SetWarnWriter(w io.Writer)` function to `internal/commands/validate.go` (trivial: sets the package-level `warnWriter` variable). The existing `warnWriter` variable is already package-level and test-overridable.

**Ops-level logs discarded:** The watcher fallback log in `await-verdict`/`await-resubmission`, CAS retry logs in `wt-merge`, worktree cleanup logs in `claim-task`, etc. are operational noise. They are useful for debugging but not for structured agent responses. Discarding them in JSON mode is correct — the final result (success or classified error) is what the agent needs.

## 7. Excluded Commands Rationale

| Command | Exclusion Reason |
|---|---|
| `agent` | Long-running supervisor process. Not a finite request/response — runs indefinitely spawning and monitoring agent sessions. |
| `tui` / `watch` | Interactive TUI dashboard with live updates. Output is a terminal UI, not a discrete result. |
| `init` | Interactive setup with user prompts (spec path, pipeline config, entry point). Requires stdin interaction. |
| `setup` | One-time global setup. Interactive prompts for CLI selection. Not used by agents. |
| `proceed` | Requires confirmation. Creates child tasks from transitions — human decision point. |
| `replan` | Human-initiated replanning at checkpoint. Requires judgment about plan amendments. |
| `delete task` | Interactive confirmation prompt (`os.Stdin`). Destructive operation requiring human approval. |
| `pause` | Human/supervisor operation (`--changed-by`). Agents don't pause the system. |
| `stop` | Human/supervisor operation (`--changed-by`). Agents don't stop the system. |
| `start` | Human/supervisor operation (`--changed-by`). Agents don't start the system. |
| `resume` | Human/supervisor operation (`--changed-by`). Agents don't resume the system. |
| `migrate` | One-time state migration. Not used by agents. |

**Rationale pattern:** Excluded commands are either (a) long-running/interactive (incompatible with single-JSON-object contract), (b) human-only operations that agents should not invoke, or (c) one-time setup/migration.

## 8. Exit Code Contract

| Condition | Exit Code | Stdout |
|---|---|---|
| Success | 0 | `{"ok": true, "result": {...}}` |
| Error (ops, RBAC, pre-ops) | 1 | `{"ok": false, "error": {"code": "...", "message": "..."}}` |

The exit code provides a fast signal for scripts/agents. The JSON envelope provides details. Both are always consistent: exit 0 iff `ok: true`.
