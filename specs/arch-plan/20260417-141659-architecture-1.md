**Task:** architecture-1
**Goal spec:** `specs/goals/20260417-precommit-bootstrap.md`
**Section:** Q2 — Idempotency / plan-time & execution-time dedup
**Scope of this plan:** (a) `kind` schema delta on `OutputEntry` and persisted `Task`, (b) authoritative dedup in `internal/ops/proceed.go` before child creation, (c) replan + cross-goal behavior, (d) test plan.

Out of scope (owned by sibling tasks): `internal/precommit/` detection helpers, `RoleContextData` wiring, architect prompt template, `base_prompt.tmpl:36` rewording, ADR-0036 amendment / supersede.

## 1. Component Boundaries & Responsibilities

| Component | File(s) | Responsibility |
|-----------|---------|----------------|
| Data model (OutputEntry) | `internal/models/task.go` | Carry a typed `kind` marker from doer output to child creation. |
| Data model (Task) | `internal/models/task.go` | Persist `kind` on the resulting child task so repo-wide scans can find it. |
| Proceed dedup (per-subtask) | `internal/ops/proceed.go` | Before creating per-subtask children, drop any output entry whose `kind` is already present on a repo-wide non-terminal task. Re-apply the same decision in crash recovery. |
| Replan propagation | `internal/ops/replan.go` | Copy `Kind` from the superseded task onto the replacement so the replacement stays counted as in-flight. |
| Test surface | `internal/ops/proceed_test.go`, `internal/models/task_test.go` (if present) | Exercise hits, misses, BLOCKED-in-flight, SUPERSEDED-unblocks-replacement, replan, cross-goal. |

**Non-responsibilities:**
- `Kind` is never derived, inferred, or defaulted by proceed.go. It arrives verbatim from the doer's `output[].kind`.
- proceed.go does not know the string `"bootstrap-precommit"`. The dedup primitive is generic over any non-empty `Kind`; the goal-spec-specific marker is a value, not a hard-coded literal.
- No renumbering of `output[]` indices when an entry is skipped. Indices remain as-emitted by the doer; `siblingIDs` is pre-computed over the full length of `task.Output` so downstream `depends_on` index resolution continues to work (see §3.3).

## 3. Authoritative Dedup in `proceed.go`

### 3.1 Insertion point

Inside `proceedInner` (`proceed.go:295`), after the `case "per-subtask":` output-validation block (around line 322) but **before** the `task.TransitionsExecuted[transitionName] = true` assignment at line 335.

```go

    for i, entry := range task.Output {

`skipEntry` is a `map[int]string` (output-entry index → reason summary for history); `remapSibling` is a `map[int]string` (output-entry index → existing in-flight task ID that takes its place in dep resolution).

The same resolution then flows into the existing per-subtask creation loop (L344-349):

```

`transitions_executed[transitionName]` is still set to `true` (the transition has fired — just with fewer children than entries). History record gains a `skipped_entries` summary so the decision is auditable.

### 3.2 Algorithm

```go
// collectNonTerminalByKind scans repo-wide (all goals, all sprints) for
// tasks with non-empty Kind in a non-terminal status. Returns map of
// kind → canonical task ID (lexicographic smallest, deterministic).
func collectNonTerminalByKind(s *models.State) map[string]string {
    byKind := map[string][]string{}
    for i := range s.Tasks {
        t := &s.Tasks[i]
        if t.Kind == "" {
            continue
        }
        if t.Status.IsTerminal() {  // MERGED, ABANDONED, SUPERSEDED
            continue
        }
        byKind[t.Kind] = append(byKind[t.Kind], t.ID)
    }
    out := map[string]string{}
    for k, ids := range byKind {
        slices.Sort(ids)
        out[k] = ids[0]
    }
    return out
}

// resolveKindDedup walks the output entries in order. Within-batch duplicates
// (same Kind appearing twice in one output[]) resolve to the first occurrence,
// so only one child is created per Kind per proceed call.
//
// `taskID` and `transitionName` are required so the helper can synthesize the
// deterministic sibling ID for the first occurrence of a within-batch Kind
// (mirrors `perSubtaskChildID` used by the surrounding code).
func resolveKindDedup(
    entries []models.OutputEntry,
    inFlight map[string]string,
    taskID, transitionName string,
) (
    skip map[int]string, remap map[int]string,
) {
    skip = map[int]string{}
    remap = map[int]string{}
    emittedThisBatch := map[string]string{}  // Kind → first sibling ID within this batch
    for i, e := range entries {
        if e.Kind == "" {
            continue
        }
        if existingID, hit := inFlight[e.Kind]; hit {
            skip[i] = fmt.Sprintf("kind %q already in flight on task %s", e.Kind, existingID)
            remap[i] = existingID
            continue
        }
        if firstID, dup := emittedThisBatch[e.Kind]; dup {
            skip[i] = fmt.Sprintf("kind %q emitted earlier in same output[] at sibling %s", e.Kind, firstID)
            remap[i] = firstID
            continue
        }
        // Will be created this batch; pre-registered so later duplicates hit.
        emittedThisBatch[e.Kind] = perSubtaskChildID(taskID, transitionName, i)
    }
    return
}
```

Invariants enforced:
- **Repo-wide**, not per-goal: the scan is over `s.Tasks` with no filtering by `Goal` or `Sprint`. Cross-goal parallelism is covered by construction.
- **BLOCKED counted as in-flight**: `IsTerminal` returns true only for `MERGED | ABANDONED | SUPERSEDED` (`task.go:146`). BLOCKED (and every other intermediate state) falls through the terminal check and is retained in the map.
- **SUPERSEDED excluded**: `IsTerminal(SUPERSEDED) == true`, so a superseded bootstrap is not counted, which is exactly what unblocks a replacement after rescope.
- **Determinism**: sorting candidate IDs lexicographically ensures the same canonical incumbent is picked every call, so dep remapping is stable across crash recovery runs.
- **Atomic within Modify**: `proceedInner` runs inside `blackboard.Modify`; any sibling proceed-call sees the in-memory State post-our-writes via the existing `s.Tasks = append(...)`. Two per-subtask transitions that both emit `bootstrap-precommit` in the same `ExecuteAvailableTransitions` batch are resolved sequentially by Phase-3's loop; the second call's `collectNonTerminalByKind` observes the first's fresh child.

### 3.3 Dependency resolution interaction

`buildChildTask` resolves `entry.DependsOn` indices against `siblingIDs[idx]` (`proceed.go:984-986`). When entry `idx` is skipped, we rewrite `siblingIDs[idx]` to the incumbent in-flight task ID (`remapSibling[idx]`). The result: any sibling declaring `depends_on: ["idx"]` correctly depends on the pre-existing bootstrap task, preserving bootstrap-first phase-gating even in the authoritative-dedup path.

Edge case — the incumbent is BLOCKED: `checkDependencies` (`task.go:522`) requires MERGED or SUPERSEDED for a dep to be satisfied. BLOCKED fails, so downstream children remain unclaimable until rescope lands (SUPERSEDED) and their dep is retargeted to the replacement by `replan.go`'s existing retargeting pass (`replan.go:149-163`). This is the correct behavior: coding cohort waits for bootstrap resolution.

### 3.5 Crash recovery (`recoverCrashedTransition`, `proceed.go:372`)

The per-subtask crash recovery branch (L374-406) currently walks `task.Output` and creates any child whose deterministic ID is absent. With dedup, we must **re-apply the same decision** so recovery does not create a child that the original proceed call correctly skipped.

**Precedence rule (authoritative, single source of truth):**

> **skipped? → existing? → create?**
>
> 1. If the entry is skipped by current-state dedup, `continue` immediately. Do NOT call `s.FindTask(siblingIDs[i])`. Do NOT call `patchInheritedDeps`.
> 2. Otherwise, if `s.FindTask(siblingIDs[i])` returns a task, patch inherited deps and `continue`.
> 3. Otherwise, the child is missing and must be created.

This ordering is not optional. It is a **safety invariant**: for a skipped entry, `siblingIDs[i]` was remapped (earlier in the same recovery call) to the incumbent task ID — which belongs to a different parent (possibly a different Goal). Calling `patchInheritedDeps` on that incumbent would append THIS task's inherited deps onto a foreign task's `DependsOn`, corrupting dependency ownership and potentially widening or deadlocking bootstrap gating across goals. The short-circuit on step 1 is what prevents that corruption.

Modified logic inside the per-subtask case:

```go
case "per-subtask":
    siblingIDs := make([]string, len(task.Output))
    for i := range task.Output {
        siblingIDs[i] = perSubtaskChildID(taskID, transitionName, i)
    }
    // Re-compute dedup decision against current repo-wide state.
    inFlightByKind := collectNonTerminalByKind(s)
    skipEntry, remapSibling := resolveKindDedup(task.Output, inFlightByKind, taskID, transitionName)
    for i, reID := range remapSibling {
        siblingIDs[i] = reID
    }

    var missingChildren []int
    for i := range task.Output {
        // (1) skipped? — short-circuit BEFORE any FindTask/patchInheritedDeps.
        //     siblingIDs[i] points at a foreign incumbent for skipped entries;
        //     patching its deps would corrupt a cross-goal task.
        if _, skipped := skipEntry[i]; skipped {
            continue
        }
        // (2) existing? — patch inherited deps on OUR child (not foreign).
        existing := s.FindTask(siblingIDs[i])
        if existing != nil {
            patchInheritedDeps(existing, inheritedDeps)
            continue
        }
        // (3) else create — flagged as missing for downstream creation.
        missingChildren = append(missingChildren, i)
    }
    // ...rest of existing recovery path unchanged (return or create missing)...
```

**Edge case: dedup decision diverges between original proceed and recovery.** The incumbent bootstrap may have transitioned (e.g. MERGED, ABANDONED, SUPERSEDED) between the two runs.

- **Original skipped, recovery finds no in-flight.** The incumbent has reached a terminal state. Recovery re-runs `collectNonTerminalByKind`, so `resolveKindDedup` returns no skip for that entry → falls through to step 2. `siblingIDs[i]` was NOT remapped (no remap entry for this index) because `resolveKindDedup` did not mark it. It still holds the original `perSubtaskChildID(taskID, transitionName, i)`. `s.FindTask` on that deterministic ID returns nil (child was never created during the original run) → step 3, create. Correct.
- **Original created, recovery finds in-flight on a foreign task only.** Impossible: a cross-goal call could be interleaved, but it would have hit our already-created child (same Kind) first and been dedup'd. If somehow two children with the same Kind exist (invariant violation upstream), the lexicographic tie-break in `collectNonTerminalByKind` makes the scan pick one deterministically; our entry either matches the picked one (siblingIDs[i] equals `s.FindTask(siblingIDs[i])` — step 2, patch our own deps) or is marked skipped by `resolveKindDedup` and short-circuits at step 1. Either way, no cross-task corruption.
- **Original created, recovery finds in-flight on OUR own child.** `collectNonTerminalByKind` includes our own child (it is in `s.Tasks`, non-terminal). `resolveKindDedup` marks the entry as skipped with `remapSibling[i]` equal to our own child's deterministic ID. Step 1 short-circuits → no `patchInheritedDeps` call. Acknowledged minor regression vs pre-dedup behavior: in this narrow case (own child exists, still non-terminal, inherited-dep set changed since prior run), the defensive patch the original recovery code applied is forgone. Accepted because (a) `computeInheritedDeps` is deterministic in state, so a changed inherited-dep set requires a change in upstream task graph that itself triggers its own transition execution, which will patch downstream deps through the regular path; (b) the safety invariant of never touching a possibly-foreign task dominates a conservative dep-patch on a task that already has the deps computed at its creation time.

In every case the skipped-first ordering keeps recovery from touching a task it has no ownership of.

### 3.6 What is explicitly NOT changed

- `one-to-one` and `many-to-one` transitions: no dedup, no Kind propagation. Bootstrap is exclusively a per-subtask `output[]` construct.
- `ExecuteAvailableTransitions` phase logic: no change. The dedup happens inside `proceedInner`, so all call-sites (`Proceed`, `ExecuteAvailableTransitions` phase 1a+3, crash-recovery phase 1b+3) automatically inherit the behavior.
- `computeInheritedDeps` (`proceed.go:1076`): no change. Inherited deps compute from upstream task's `output[]` length, which is unaffected by skip decisions because we never shrink `task.Output`.

## 4. Replan + Cross-Goal Behavior

### 4.1 Replan (goal re-planning after bootstrap approval but before merge)

The goal spec describes:

> If the goal is re-planned after bootstrap is approved but before integration merge, the second architect invocation re-runs the plan-time checks (Q2) […] Belt-and-braces: even if the architect emits anyway, `proceed.go`'s authoritative `kind`-based dedup (Q2) skips child creation.

For this plan the relevant mechanics are:

1. The first architect's bootstrap child is in state `DRAFT_CODE` / `CODE_APPROVED` / … — any non-terminal state. Its `Kind == "bootstrap-precommit"`.
2. Replan produces a new architect task inheriting the original's fields; `Task.Kind` is inherited in the same way (§2.4 is agnostic to whether the task being replanned is an architect, a bootstrap coding task, or anything else — the rule "copy Kind verbatim" is universal).
3. The new architect runs, potentially emits bootstrap in its `output[]` again.
4. At proceed-time, the repo-wide scan sees the first architect's bootstrap child still non-terminal → skip.
5. If the architect omitted bootstrap (plan-time check worked), there is no entry to skip. Unchanged behavior.

### 4.2 Rescope via SUPERSEDED

Spec requires that a BLOCKED bootstrap, when rescoped, be marked SUPERSEDED before/simultaneously with emitting a replacement. Our dedup honors this via `IsTerminal`:

- BLOCKED bootstrap present → repo-wide scan finds it → new emission is dedup'd away.
- Orchestrator rescopes: marks BLOCKED → SUPERSEDED. Replacement emission is the next architect cycle or direct task injection.
- `SUPERSEDED` is terminal → excluded from the scan → replacement is created.

The replacement path is either:

- **Architect-emitted** (second architect iteration): identical to §4.1 case 5.
- **`replan.go` rewrite** of the bootstrap coding task itself: the new task inherits `Kind` (§2.4), keeping the "in-flight" invariant intact across the replan boundary.

### 4.3 Cross-goal parallelism

Two concurrent Goals A and B, each with their own architect task, both emit `bootstrap-precommit` in `output[0]`. Timeline:

- A's architect merges first → A's proceed runs → creates `A-bootstrap` child with `Kind: bootstrap-precommit`.
- B's architect merges → B's proceed runs → `collectNonTerminalByKind` scans **repo-wide** → finds `A-bootstrap` (non-terminal) → dedup'd away.
- B's other children still declare `depends_on: ["0"]`; the remap rewrites `siblingIDs[0]` to `A-bootstrap.ID`. B's coding cohort now depends on A's bootstrap, correctly phase-gating on the single repo-wide bootstrap.

No per-Goal filtering in `collectNonTerminalByKind` is the critical design property that makes this work. It is called out explicitly in §3.2 so future refactors do not mistakenly scope the scan to a single Goal.

### 4.4 Failure modes explicitly handled

| Scenario | Observed in scan? | Outcome |
|----------|-------------------|---------|
| Bootstrap MERGED | No (terminal) | New entry is created — allowed. This is the brownfield re-plan / post-merge scenario; typically the architect-side plan-time check also suppresses emission because the file is now present on integration HEAD. |
| Bootstrap ABANDONED | No (terminal) | New entry created. Same rationale as MERGED: the abandoned bootstrap is not in flight. Orchestrator action, not a dedup concern. |
| Bootstrap SUPERSEDED | No (terminal) | New entry created — this is the unblock-after-rescope path. |
| Bootstrap BLOCKED | Yes (non-terminal) | New entry skipped, remap points at the BLOCKED task; coding cohort waits. |
| Bootstrap DRAFT_CODE / IMPLEMENTING_CODE / etc. | Yes | Skipped. |
| Bootstrap replanned (new task with same Kind, old SUPERSEDED) | Yes (new task) | Skipped correctly; the new task is the in-flight incumbent. |
| Two entries with same Kind in one output[] | Second+ skipped (within-batch dedup) | Exactly one child created; later entries remap to the first. |

## 6. Cross-cutting Concerns

### 6.1 Observability

Every dedup-skip is recorded in the trigger task's history (`skipped_entries` in the `TaskEventTransitionExecuted` extra). This is the only observable side-effect — no log output, no metric. Existing log analysis tooling (ADR-0029) surfaces history entries, so no new plumbing is required.

### 6.2 Concurrency

Blackboard singleton + `Modify` atomicity (ADR-0022) already linearizes `proceed.go` state writes. The dedup scan runs in the same `Modify` callback as the child creation, so no TOCTOU race between scan and append within a single call. Cross-call races are eliminated by the CAS-based `Modify` loop.

### 6.3 Migration

None. Adding `omitempty` fields with empty-string defaults is YAML-backward-compatible. Old state files load unchanged; new fields serialize only when set.

### 6.4 Error handling

- Empty `Kind`: default path — no scan participation, no dedup.
- `collectNonTerminalByKind` internal-error path: none. Pure in-memory computation over `s.Tasks`.
- `buildChildTask` remap: when `siblingIDs[i]` is an in-flight task ID rather than a "would-be" child ID, dep resolution still works because `buildChildTask` only ever reads `siblingIDs[idx]` by index — it does not care whether the ID corresponds to a child created _this_ call or a task created _earlier_. No error branch to add.

### 6.5 Prompt / orchestrator impact

None from this plan. The per-entry `Kind` field is populated by the architect prompt (sibling task) via `liza set-task-output` JSON payloads. The CLI already passes unknown JSON keys through to the struct on Go-side unmarshalling; adding the field makes the key known without changing CLI behavior.
