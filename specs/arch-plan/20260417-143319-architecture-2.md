# Architecture Plan — Plan-Time Detection & Context Enrichment for Pre-Commit Bootstrap

**Task:** architecture-2
**Goal spec:** `specs/goals/20260417-precommit-bootstrap.md`
**Section:** Q2 — Idempotency / plan-time & execution-time dedup
**Scope of this plan:** (a) new `internal/precommit/` package with `ConfigExistsOnIntegration` and `BootstrapInFlight` helpers, (b) `RoleContextData` extended with `PreCommitConfigExists` / `PreCommitBootstrapInFlight` booleans populated only for the architect role, (c) error propagation from helper failure to an explicit task-blocking orchestrator outcome (BLOCKED), (d) test plan for exists/absent/error and BLOCKED-in-flight counted as in-flight.

**Out of scope (owned by architecture-1, already MERGED):** `kind` field on `OutputEntry` / `Task`, `proceed.go` authoritative dedup, `replan.go` propagation, round-trip unit tests for `Kind`. This plan reuses the `Kind` field specified by that plan's §2.1/§2.2.

**Also out of scope (sibling tasks in the overall bootstrap implementation):** architect prompt template additions, `base_prompt.tmpl:36` rewording, ADR-0036 amendment or supersede, bootstrap task's `done_when` phrasing, greenfield reproduction run.

## 1. Component Boundaries & Responsibilities

| Component | File(s) | Responsibility |
|-----------|---------|----------------|
| Detection package | `internal/precommit/precommit.go` (new), `internal/precommit/precommit_test.go` (new) | Expose two pure detection helpers — one integration-branch file check, one repo-wide scan by `Kind` marker. Package has no knowledge of prompts, templates, or task lifecycle. |
| Context builder | `internal/agent/prompt.go` (`buildTaskRoleContextData`) | Call the two helpers only when the target role is `architect`; populate `RoleContextData.PreCommitConfigExists` / `PreCommitBootstrapInFlight`. Surface helper errors to the caller. |

| Prompt orchestration | `internal/agent/prompt.go` (`buildPromptWithContext`), `internal/agent/strategy_doer.go` (`BuildPrompt`), `internal/agent/supervisor.go` (the `BuildPrompt` call site at L817) | Propagate the helper error upward and translate it into a BLOCKED task outcome via the existing `blockTaskFromSupervisor` primitive. |
| Test surface | `internal/precommit/precommit_test.go`, `internal/agent/prompt_test.go` | Cover exists / absent / error on `ConfigExistsOnIntegration`; cover non-terminal hit (including BLOCKED) / miss / empty-kind skip on `BootstrapInFlight`; cover architect-only population and error-propagation path through the context builder. |

**Non-responsibilities:**

- The `precommit` package does **not** render template blocks. The two booleans feed prompt-template conditionals authored in a sibling task — this plan only ensures the data reaches `RoleContextData`.
- The package does **not** decide what "`bootstrap-precommit`" means. The `Kind` string constant is defined next to the helper (one-line, documented as mirroring the architect-prompt marker) and consumed locally; `proceed.go` remains agnostic per architecture-1 §2.
- The context builder does **not** itself mark the task BLOCKED. Blocking is the supervisor's responsibility, reusing `blockTaskFromSupervisor` (`supervisor.go:289`).
- The helper is **not** a `pipeline.Resolver` responsibility (goal spec §Q2, "Terminology"). Keep the package boundary distinct.

### 2.2 Package declaration and `Kind` constant

```go
// Package precommit provides repo-state detection helpers for the
// pre-commit bootstrap planning step: checking whether the integration
// branch already carries .pre-commit-config.yaml, and whether a
// bootstrap task is already in flight anywhere in the state.
package precommit

import "errors"

// Kind is the typed marker value (see architecture-1 §2.1) that the
// architect emits on output[].kind and that proceed.go treats as the
// authoritative dedup key. Exported so Go-side callers (this package's
// BootstrapInFlight scan, the architect prompt-build path surfacing
// the marker into RoleContextData, and any future doc/test that wants
// to reference the canonical string) cannot drift on the literal.
//
// Go templates cannot import constants — the architect prompt
// template reads this value via RoleContextData.PreCommitKind (§3.1)
// rather than by package import. The architect prompt template is
// the single template-layer consumer; other sibling docs that
// reference the marker (bootstrap task done_when phrasing, any ADR
// text) do so as free-form prose, not via import.
const Kind = "bootstrap-precommit"

// ErrContextBuild is the sentinel returned (wrapped) by every error
// path in this package. Callers discriminate precommit-originated
// failures from other BuildPrompt failures via errors.Is — that
// discrimination is what lets the supervisor narrow its BLOCKED
// recovery to this package's error domain (§4.2) without
// misclassifying unrelated template/pipeline defects as task-local
// failures.
var ErrContextBuild = errors.New("precommit context build failed")
```

**Sentinel contract (wrapping rules):**

- Every error returned by `ConfigExistsOnIntegration` wraps `ErrContextBuild` via `fmt.Errorf("…: %w", ErrContextBuild)` **and** preserves the underlying cause via a second `%w` chain where applicable (Go 1.20+ supports multi-error wrapping via `errors.Join`; alternatively, include the underlying cause as string context and rely on `errors.Is` on the sentinel). Pattern:

  ```go
  return false, fmt.Errorf("precommit: ls-tree %q: %w (%w)",
      integrationBranch, ErrContextBuild, err)
  ```

- `BootstrapInFlight` is error-free by construction and does not participate in the sentinel contract.
- `buildTaskRoleContextData` may add its own prefix via `%w` (per §3.2) — the sentinel survives because `errors.Is` walks the wrap chain.

The `Kind` constant lives in this package because the detection helpers are the primary consumer (`BootstrapInFlight` needs the exact value). `internal/ops/proceed.go` remains agnostic — it compares `Kind`s structurally, not against this constant (per architecture-1 §1 non-responsibility "proceed.go does not know the string `bootstrap-precommit`").

### 2.3 `ConfigExistsOnIntegration`

```go
// ConfigExistsOnIntegration reports whether .pre-commit-config.yaml
// exists at the tip of the integration branch in the given project
// root.
//
// Implementation reads committed state via git plumbing, not the
// working tree. A working-tree read would pick up uncommitted human
// drift (or in-progress worktree changes from a sibling coding task)
// and give a false positive. Precedent: checkSpecFileExists in
// internal/statevalidate/validate.go:92 uses git plumbing for the
// same reason, falling back to `git cat-file -e` when the filesystem
// check is insufficient.
//
// Returns (true, nil) when the file is tracked on the branch.
// Returns (false, nil) when the branch exists and the file is not
// tracked on it. Returns (_, err) only for plumbing failures that
// prevent a reliable answer — git not on PATH, projectRoot or
// integrationBranch empty/invalid, unexpected non-zero exit while
// capturing stderr, or IO error.
func ConfigExistsOnIntegration(projectRoot, integrationBranch string) (bool, error)
```

**Plumbing-command choice — behavior-verified, not doc-inferred.** The natural first choice, `git cat-file -e <branch>:.pre-commit-config.yaml`, is **rejected**: empirical verification in the target toolchain (scratch repo on this project's Linux/git stack) shows both "path absent on existing branch" and "invalid branch name" return exit code `128` with different stderr wording, not the `0/1/other` split that some older docs describe. A helper specified around that non-existent split would misclassify the normal greenfield "config absent" case as a hard error and block every architect task — the opposite of what the goal spec requires (goal spec §Q2 explicitly wants "absent" to be a silent signal, not a BLOCKED outcome).

**Chosen primitives (verified):**

| Primitive | Used for | Verified behavior |
|-----------|----------|-------------------|
| `git rev-parse --verify --quiet <branch>^{commit}` | Disambiguate "branch missing" from "path missing" before the tree lookup. Classifies real configuration errors. | Exit 0 on a valid ref; exit 1 (no stderr, no stdout) on a missing / invalid ref. Clean split. |
| `git ls-tree <branch> -- .pre-commit-config.yaml` | Classify "file absent on existing branch" vs "file present" via stdout content, not via exit code. | Exit 0 with empty stdout when the path is not tracked on the branch; exit 0 with one line `<mode> blob <sha>\t.pre-commit-config.yaml` when tracked. Exit 128 only on invalid ref (ruled out by the rev-parse pre-check). |

This is the same family of precedent as `checkSpecFileExists` (`gitenv.Command(...)` with `cmd.Dir = projectRoot`, and exit-0 means success), just with a command whose success/failure semantics match our "absent is not an error" contract.

**Algorithm:**

```go
func ConfigExistsOnIntegration(projectRoot, integrationBranch string) (bool, error) {
    if projectRoot == "" {
        return false, fmt.Errorf("precommit: projectRoot is empty: %w", ErrContextBuild)
    }
    if integrationBranch == "" {
        return false, fmt.Errorf("precommit: integrationBranch is empty: %w", ErrContextBuild)
    }

    // Step 1 — verify the ref exists. Isolates "branch invalid" as a
    //           hard configuration error.
    verify := gitenv.Command("rev-parse", "--verify", "--quiet",
        integrationBranch+"^{commit}")
    verify.Dir = projectRoot
    if err := verify.Run(); err != nil {
        var exitErr *exec.ExitError
        if errors.As(err, &exitErr) && exitErr.ExitCode() == 1 {
            return false, fmt.Errorf(
                "precommit: integration branch %q not found in %s: %w",
                integrationBranch, projectRoot, ErrContextBuild)
        }
        return false, fmt.Errorf("precommit: rev-parse --verify %q: %v: %w",
            integrationBranch, err, ErrContextBuild)
    }

    // Step 2 — read the tree entry. Exit 0 with empty stdout means
    //           "branch has no such path"; non-empty means tracked.
    lsTree := gitenv.Command("ls-tree", integrationBranch, "--",
        ".pre-commit-config.yaml")
    lsTree.Dir = projectRoot
    var stdout, stderr bytes.Buffer
    lsTree.Stdout = &stdout
    lsTree.Stderr = &stderr
    if err := lsTree.Run(); err != nil {
        return false, fmt.Errorf(
            "precommit: ls-tree %q -- .pre-commit-config.yaml: %v (stderr: %s): %w",
            integrationBranch, err, strings.TrimSpace(stderr.String()), ErrContextBuild)
    }
    return stdout.Len() > 0, nil
}
```

Every error path wraps `ErrContextBuild` so `errors.Is(err, precommit.ErrContextBuild)` is the single discriminator downstream. The underlying `*exec.ExitError` / `error` is captured in the format string via `%v` for diagnostic log capture — callers that need to classify the raw cause can still walk the error chain, but the supervisor's decision uses only the sentinel.

| Aspect | Spec |
|--------|------|
| Path checked | Literal `.pre-commit-config.yaml` at repo root of `integrationBranch`. Not configurable in this plan — the contract template (`commit_workflow.tmpl`) assumes that exact name, so a configurable path would introduce drift without a caller need. |
| Input guards | Empty `projectRoot` or empty `integrationBranch` → returns `(false, error)`. Configuration invariant, not a "file absent" answer. |
| Ref-verify | `git rev-parse --verify --quiet <branch>^{commit}`. Exit 0 → proceed. Exit 1 → wrap and return as "integration branch not found" error. Any other exit (extremely unusual, but possible on a corrupt repo) → wrap with `%w` and return. |
| Path-check | `git ls-tree <branch> -- .pre-commit-config.yaml`. Success + empty stdout → `(false, nil)`. Success + non-empty stdout → `(true, nil)`. Non-zero exit (should not happen after rev-parse succeeded; we keep the handler as a safety net) → `(false, err)` including stderr. |
| Plumbing wrapper | `gitenv.Command` for both calls. Ensures `LC_ALL=C` — stderr stays locale-stable for any future log capture. `cmd.Dir = projectRoot` on both. |
| Logging | None inside the helper. Diagnostic context belongs to the caller (the context builder wraps the error). |
| Symlink / LFS | Not handled. `.pre-commit-config.yaml` is a regular blob in every supported project; if a future project stores it differently, that is a separate design question. |

Precedent and divergence from `checkSpecFileExists` (`internal/statevalidate/validate.go:92`):

- Precedent reused: `gitenv.Command(...)` with `cmd.Dir = projectRoot`; git plumbing as the single source of truth for committed state.
- Divergence 1: `checkSpecFileExists` falls back to `git cat-file -e` **after** `os.Stat`. Here we do **not** consult the filesystem first, because we explicitly want to ignore working-tree drift (goal spec §Q2: "not direct filesystem read of the main working tree, which may include uncommitted human drift").
- Divergence 2: `checkSpecFileExists` treats any non-zero exit from git as "file not found" and returns a generic error. Here we must differentiate three classes — **file absent** (`false, nil`), **branch invalid** (`false, err`, hard configuration error), and **plumbing failure** (`false, err`). The `cat-file -e` exit-code story does not deliver that three-way split reliably (see verified behavior above); the `rev-parse` + `ls-tree` pair does. The divergence is necessary, not cosmetic.

### 2.4 `BootstrapInFlight`

```go
// BootstrapInFlight reports whether any task in the given state has
// Kind == precommit.Kind and a non-terminal status. "Non-terminal"
// here means Task.Status.IsTerminal() returns false — which excludes
// MERGED, ABANDONED, SUPERSEDED and includes BLOCKED (a blocked
// bootstrap will eventually merge once rescoped, so it genuinely is
// in flight; see goal spec §Q2 "Rescope invariant").
//
// Repo-wide: scans state.Tasks with no filter by Goal / Sprint. Two
// concurrent goals racing to emit bootstrap see each other via this
// scan; cross-goal parallelism is covered by construction. This
// mirrors the same design property in collectNonTerminalByKind
// (architecture-1 §3.2), and deliberately shares no code with it —
// the prompt-time read path runs outside proceed.go's Modify
// callback, so coupling the two would be a layering error.
func BootstrapInFlight(state *models.State) bool
```

| Aspect | Spec |
|--------|------|
| Input | Non-nil `*models.State`. On `nil` input, returns `false` — callers build their own state from disk before invoking this helper; a nil state is programmer error, not a runtime condition. No panic, no error (this helper does not return error — the scan is pure in-memory and cannot fail). |
| Scan cost | O(len(state.Tasks)). `state.Tasks` is already loaded in memory by the context builder. No new IO. |
| Terminal definition | `task.Status.IsTerminal()` (`models.Task.Status`, `task.go:146`). MERGED / ABANDONED / SUPERSEDED → terminal; every other status (including BLOCKED, DRAFT_*, IMPLEMENTING_*, IN_REVIEW, INTEGRATION_*, etc.) → in flight. |
| Empty-Kind skip | Tasks with `Kind == ""` are not candidates — same invariant as `proceed.go`'s scan (architecture-1 §3.2). |
| Early exit | Loop breaks on first match. Result is a boolean, not a list; the context builder only needs to know "is there at least one". |
| Concurrency | None. The caller passes a snapshot `*models.State` obtained through the normal `blackboard` read path. TOCTOU between the plan-time check and the later proceed-time dedup is acceptable and explicitly covered by the goal spec ("B is not sufficient on its own"); the authoritative dedup lives in `proceed.go` per architecture-1. |

### 2.5 Why no `ListBootstrapInFlight`?

A helper returning the list of matching task IDs would be needed only if the architect prompt surfaced incumbent IDs or if a later caller wanted to warn about orphaned multiples. Neither is in scope. YAGNI — add the richer primitive if and when a second caller appears.

## 4. Error Propagation — Helper Failure → BLOCKED Outcome

### 4.1 Flow

```
precommit.ConfigExistsOnIntegration  (git plumbing error)
        ↓ err
buildTaskRoleContextData             (returns err wrapped "precommit config check: ...")
        ↓ err
buildPromptWithContext               (returns err unwrapped — already descriptive)
        ↓ err
doerStrategy.BuildPrompt             (returns err unchanged — interface signature is already (string, error))
        ↓ err
supervisor loop (supervisor.go:817)  (current: wraps as "failed to build prompt: %w" and returns the entire supervisor error)
        ↓ detect + block
blockTaskFromSupervisor(...)         (marks task BLOCKED, releases lease, writes history)
```

### 4.2 Supervisor-side change (`supervisor.go:817-820`)

**Control-flow context (verified):** the `BuildPrompt` call at L817 sits inside the main `for` loop of `RunSupervisor`. The existing pre-exec BLOCKED paths at L790–792 (spinning) and L917–919 (crash-restart loop) both follow the pattern `blockTaskFromSupervisor(...); ...reset(...); continue`. Returning `nil` from inside this loop would exit the supervisor process, which is the wrong recovery mode — the agent session should remain alive, observe the BLOCKED state on the next poll, and move on (or shut down cleanly through the existing work-detection path). Using `continue` preserves the session-lifecycle invariant that the existing BLOCKED paths already rely on.

**Scope-of-blocking (refined after review iteration 2):** the BLOCKED recovery must cover **only the precommit helper error path**, not every possible failure in `strategy.BuildPrompt`. Other `BuildPrompt` errors — base-prompt rendering failures (prompt.go:33-45), resolver / section lookup errors (prompt.go:43-46), template rendering errors (prompt.go:50-52) — are **systemic** failures that affect the entire agent pool across every task and every role (both doer and reviewer strategies share this path per `strategy_doer.go:103-105` and `strategy_reviewer.go:157-159`). Reclassifying a systemic template defect as a task-local BLOCKED state would mask the systemic defect and let the supervisor keep draining tasks into BLOCKED until the pool is empty, when the correct response is to surface the defect and let the existing crash-restart machinery kick in. Discrimination is via the `precommit.ErrContextBuild` sentinel (§2.2), which every precommit-package error path wraps and no other error source touches.

Current code:

```go
prompt, err := strategy.BuildPrompt(stateBefore, config, taskID)
if err != nil {
    return fmt.Errorf("failed to build prompt: %w", err)
}
```

Modified:

```go
prompt, err := strategy.BuildPrompt(stateBefore, config, taskID)
if err != nil {
    if claimedTaskID != "" && errors.Is(err, precommit.ErrContextBuild) {
        reason := fmt.Sprintf("prompt context build failed: %v", err)
        GetLogger().Warn("Task blocked due to prompt-context build failure",
            "agent_id", config.AgentID,
            "task_id", claimedTaskID,
            "error", err)
        blockTaskFromSupervisor(bb, config.ProjectRoot, claimedTaskID, config.AgentID, reason)
        spinTracker.reset(effectiveTask)
        continue
    }
    return fmt.Errorf("failed to build prompt: %w", err)
}
```

| Aspect | Spec |
|--------|------|
| When do we block? | Both conditions must hold: (1) a task is actually claimed (`claimedTaskID != ""`), and (2) the error carries the precommit sentinel (`errors.Is(err, precommit.ErrContextBuild)`). Either missing condition falls through to the original wrapped-error return, preserving existing behavior for all non-precommit failures. |
| Control flow after blocking | `continue` — iterate the outer `for` loop. Matches the existing pattern at supervisor.go:790-792 (spinning detection) and supervisor.go:917-919 (crash restart loop). Does NOT exit `RunSupervisor`. |
| Spin-tracker reset | Call `spinTracker.reset(effectiveTask)` before `continue`, mirroring the L790-792 pattern. Prevents a subsequent spinning detection from firing on the same (now BLOCKED) task ID on the next loop iteration. The crash-tracker is not involved because the agent never ran, so there is nothing to reset there. |
| What reason do we record? | `"prompt context build failed: %v"` — includes the full wrapped chain (e.g. `precommit: integration branch "main2" not found in /path: precommit context build failed`). Only fires for precommit-domain errors (enforced by the `errors.Is` guard), so the phrasing is accurate. |
| Why not `return nil`? | `return nil` would terminate the supervisor session. The task is BLOCKED, but the agent process exits — no poll, no clean shutdown, no retry of the containing work item (work detection relies on the loop continuing so the agent can re-poll and observe BLOCKED as non-claimable). The existing pre-exec BLOCKED paths use `continue` for exactly this reason; deviating would be a control-flow regression caught in review iteration 1. |
| Why not return the error for precommit failures? | Returning the error bubbles to the outer loop and triggers crash-restart logic, which is wrong: the task is now BLOCKED, so restart is pointless and would race with the BLOCKED state. `continue` is the right primitive — but only for the narrow precommit-domain class, not systemic failures. |
| Why sentinel-guarded rather than uniform blocking? | Uniform blocking (the earlier iteration-2 design) would reclassify systemic template/resolver/pipeline defects as task-local BLOCKED and silently drain the task pool. The goal spec scopes this recovery to helper/context-builder failures only (goal spec §Q2 "Failure handling"); a typed sentinel is the minimum ceremony that preserves that scope contract while still letting the supervisor discriminate statically. |
| `blockTaskFromSupervisor` side effects (already existing) | Transitions task to BLOCKED via `TransitionWith(models.TaskStatusBlocked, ...)`; clears `AssignedTo`, `LeaseExpires`, `ReviewingBy`, `ReviewLeaseExpires`; appends a `TaskEventBlocked` history entry with `Reason` set. Default `BlockedQuestions` array (two generic questions about spec clarity / environment) is written — acceptable; this plan does not specialize the questions because the supervisor-side error site cannot know whether the precommit root cause was a git issue, an input-guard issue, or a plumbing failure without pattern-matching the error, which is brittle. |

### 4.3 Why not raise the error through to exit-code handling?

The existing exit-code dispatcher at `supervisor.go:841` handles exits from an agent that **was executed**. The agent never runs when `BuildPrompt` fails (control never reaches `executeAgent`). So the exit-code machinery cannot translate this failure; we must mark the task BLOCKED at the `BuildPrompt` site itself and then `continue` the outer loop. That is exactly the pattern the two existing pre-exec BLOCKED sites (supervisor.go:790-792 spinning; supervisor.go:917-919 crash-restart loop) use with `blockTaskFromSupervisor`. This is a natural third caller in that family.

### 4.4 Observable outcomes (contract)

| Outcome | Observable state |
|---------|------------------|
| Helper returns `(true, nil)` | Task prompt contains `PreCommitConfigExists=true`; architect omits bootstrap entry. Task status unchanged. |
| Helper returns `(false, nil)` and scan finds no in-flight | `PreCommitConfigExists=false`, `PreCommitBootstrapInFlight=false`. Architect MAY emit bootstrap. Task status unchanged. |
| Helper returns `(false, nil)` and scan finds in-flight | `PreCommitConfigExists=false`, `PreCommitBootstrapInFlight=true`. Architect omits (duplicate would be dedup'd by proceed.go anyway — belt-and-braces). Task status unchanged. |
| Helper returns `(_, err)` (wraps `precommit.ErrContextBuild`) | Architect agent is **not** executed. Supervisor's `errors.Is(err, precommit.ErrContextBuild)` matches → task transitions to BLOCKED with `BlockedReason` starting `"prompt context build failed: precommit:"`. Orchestrator sees BLOCKED on next poll; downstream gating is unchanged (dep resolution already treats BLOCKED as unsatisfied per `checkDependencies`, `task.go:522`). |
| Unrelated `BuildPrompt` failure (template, resolver, pipeline) | Sentinel does NOT match → supervisor returns the original wrapped `"failed to build prompt: %w"` error. Existing crash-restart / system-stop machinery handles it. This task's work is not reclassified as task-local BLOCKED. |

The fourth row is the explicit **not silent omission, not opaque crash** outcome the goal spec requires.

## 6. Cross-cutting Concerns

### 6.1 Dependency on architecture-1's downstream coding work

**State-vs-branch caveat (explicit).** At the time of writing this plan, the blackboard marks `architecture-1` as MERGED — but architecture-1's merge product is only its **architecture document** (`specs/arch-plan/20260417-141659-architecture-1.md`), not the Go code it describes. The actual `Kind` field on `OutputEntry` and `Task` (architecture-1 §2.1, §2.2) is delivered by architecture-1's output[] entry 0 — a code-planning task that then produces a coding task that writes `internal/models/task.go`. At this plan's review commit, that coding task has not yet claimed / merged, so `internal/models/task.go` does not yet carry the `Kind` field.

This is a source-vs-state split, not a contradiction:

| Source | Says about `Kind` |
|--------|-------------------|
| Blackboard `architecture-1` task status | MERGED — arch doc integrated. |
| Blackboard `architecture-1.output[]` child coding tasks | Not yet MERGED — code landing pending. |
| `internal/models/task.go` on the review branch | No `Kind` field yet. |

**Impact on this plan's code-planning child (§7.1):**

- `BootstrapInFlight` reads `task.Kind`; it cannot compile until the coding task from architecture-1 entry 0 has merged.
- The code-planning child emitted by this task MUST declare `depends_on` (at the coding-task layer, surfacing through phase-gate inheritance per ADR-0048 / `computeInheritedDeps`) on the coding task produced by architecture-1 entry 0. In practice this dependency is already carried: this architecture task's own `depends_on: ["architecture-1"]` propagates through the cohort's phase-gate inheritance, so the coding child of architecture-2 entry 0 will not be claimable until the coding children of architecture-1 entries 0 and 1 are MERGED.
- The "MUST NOT modify `internal/models/task.go`" constraint in §7.1 stands — the `Kind` field's authorship belongs to architecture-1's coding child, not this one. If by the time the coding child of this plan runs, architecture-1 entry 0's coding child has for any reason not landed, the coder hits a compile error on `task.Kind` and must mark BLOCKED (normal failure mode — phase-gate inheritance should prevent this but we explicitly do not add a second defensive path).

**No action required in this plan** beyond the explicit acknowledgment above. The existing dependency machinery carries the correct ordering; this section only ensures the review-iteration-1 source-vs-branch observation is captured in the doc so future readers do not trip on the same contradiction.

### 6.2 Concurrency / TOCTOU

The plan-time check is a best-effort early filter. The authoritative guarantee lives in `proceed.go`'s dedup (architecture-1 §3). This plan does not attempt to lock, snapshot, or otherwise coordinate between the prompt-time read and the eventual proceed-time write; any race there is resolved by the authoritative layer.

### 6.3 Observability

- Helper failures surface via the supervisor log line (`"Task blocked due to prompt-context build failure"`) and the task's history (`TaskEventBlocked` with a descriptive reason). No new log stream.
- Successful reads produce no output — standard Go pattern for pure detection helpers. If diagnostics are needed later, a single structured log line in the helper is trivial to add.

### 6.4 Migration / backward compat

- Adding two `bool` fields to `RoleContextData` is purely additive (zero-value for every existing caller).
- Template blocks that consume these fields are emitted by a sibling task; until that task lands, the fields are populated but not rendered. No user-visible change from this plan alone.
- Changing `buildTaskRoleContextData`'s signature from `*RoleContextData` to `(*RoleContextData, error)` is a compile-time-visible break for test files; the code-planning child is responsible for the mechanical rewrite. No on-disk state or protocol change.

### 6.5 Performance

Both helpers run once per architect prompt build — that is, once per claim of an architect task. `ConfigExistsOnIntegration` runs two `gitenv.Command` invocations: `git rev-parse --verify --quiet <branch>^{commit}` (microseconds — pure ref lookup) followed by `git ls-tree <branch> -- .pre-commit-config.yaml` (microseconds — single-entry tree read). `BootstrapInFlight` is O(n) over an already-loaded slice. Net cost is lost in the noise compared to the template-rendering pipeline already executing in `buildPromptWithContext`.

### 6.6 Error domain ownership

- `precommit` package owns: git plumbing errors, input-validation errors (empty `projectRoot` / empty branch), and the `ErrContextBuild` sentinel that every error path wraps.
- `agent.buildTaskRoleContextData` owns: adding the `"precommit config check: %w"` prefix while preserving the sentinel through the wrap chain.
- `agent.supervisor` owns: `errors.Is(err, precommit.ErrContextBuild)` discrimination at the `BuildPrompt` call site, translating the narrow class of matches to BLOCKED state via `blockTaskFromSupervisor` and leaving non-precommit failures on the existing crash-surfacing path.

No owner overlaps. The sentinel is the single discrimination primitive; all three layers agree on its meaning.
