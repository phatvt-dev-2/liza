# Worktree Isolation

Per-task git worktrees for isolated implementation and controlled integration.

## Description

Each implementation task operates in its own `task/<id>` branch under `.worktrees/<id>`, created from `integration`. Claiming logic enforces when worktrees are created, reused, or recreated; merge logic enforces integration safety and explicit failure states.

Isolation is preserved by role boundaries: Coders commit only in their assigned worktree, while integration branch updates happen through supervisor-controlled merge operations after approval.

## Features

- 1.4.1 — [Claim-Time Provisioning](../protocols/worktree-management.md) — Worktree creation for READY tasks and base commit capture
- 1.4.2 — [Reassignment Semantics](../protocols/worktree-management.md) — Same coder preserves worktree; different coder gets fresh worktree
- 1.4.3 — [Integration-Fix Ownership](../protocols/worktree-management.md) — INTEGRATION_FAILED tasks keep worktree and are claimable for targeted fixes
- 1.4.4 — [Branch Strategy](../protocols/worktree-management.md) — `main` → `integration` → `task/<id>` branches
- 1.4.5 — [Working-Tree-Less Merge](../protocols/worktree-management.md) — `merge-tree`/`commit-tree` + CAS `update-ref` integration updates
- 1.4.6 — [Failure Handling](../protocols/worktree-management.md) — HEAD mismatch, merge conflict, or integration-test failure → `INTEGRATION_FAILED`
- 1.4.7 — [Rollback and Cleanup Rules](../protocols/worktree-management.md) — Rollback on test failure; worktree cleanup after successful merge
- 1.4.8 — [State Invariants](../protocols/worktree-management.md) — `IMPLEMENTING` requires worktree; `MERGED` must clear worktree

## Boundaries

**In scope:**
- Worktree lifecycle and branch isolation
- Reassignment and integration-fix handling
- Merge safety and rollback behavior
- Worktree-related state invariants

**Out of scope:**
- General task workflow decisions (see 1.3 Task Management)
- Blackboard schema and coordination model (see 1.2 Multi-Agent Coordination)
