# Pattern-Triggered Checkpoint Escalation

## User Story

As an operator, I want anomaly-pattern detection to trigger explicit checkpoint escalation so systemic failures are surfaced before continued autonomous execution.

## Acceptance Criteria

- [ ] Pattern detection checks anomaly clusters in deterministic order and emits a single triggered pattern result.
- [ ] Triggered analysis writes a circuit-breaker report with evidence and raw anomaly context.
- [ ] Triggered analysis updates blackboard circuit-breaker state and sets system mode to `CIRCUIT_BREAKER_TRIPPED`.
- [ ] `liza watch` auto-creates sprint `CHECKPOINT` when a pattern trigger is detected.
- [ ] Non-triggered analysis records healthy status and clears trigger state while preserving manual pause semantics.
- [ ] Sprint metrics computation supports warning heuristics for suspiciously high approval rates.

## Implementation

- `internal/analysis/patterns.go` — anomaly pattern checks, severity classification, and report generation.
- `internal/ops/analyze.go` — blackboard analysis flow, trigger state mutation, and report persistence.
- `internal/commands/watch.go` — continuous pattern detection and automatic checkpoint escalation.
- `internal/ops/update_sprint_metrics.go` — aggregate metric recomputation and suspicious-rate warning checks.
