# Claim-to-Verdict Task Lifecycle

## User Story

As a coder/reviewer, I want claim, review submission, and verdict steps to enforce strict state transitions so task outcomes are reliable.

## Acceptance Criteria

- [ ] Claiming supports READY/REJECTED/INTEGRATION_FAILED entry points with state- and lease-safe updates.
- [ ] Claim flow handles worktree behavior correctly for first claim, reassignment after rejection, and integration-fix claims.
- [ ] Submit-for-review verifies task ownership, rebases onto integration, and transitions to READY_FOR_REVIEW with updated commit SHA.
- [ ] Verdict submission requires explicit REJECTED reason, applies APPROVED/REJECTED transitions, and releases reviewer assignment.
- [ ] Limit escalation can move repeatedly failing tasks to BLOCKED with structured blocked reasons/questions.

## Implementation

- `internal/ops/claim_task.go` — three-phase claim validation/worktree/state-commit protocol.
- `internal/ops/submit_review.go` — rebase-on-integration and READY_FOR_REVIEW transition.
- `internal/ops/submit_verdict.go` — verdict transition logic, rejection reason enforcement, and reviewer release.
- `internal/ops/iteration_limits.go` — review/iteration limit classification and escalation messaging.
