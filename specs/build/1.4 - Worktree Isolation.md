# Worktree Isolation

## Goal

Isolate task implementation work so concurrent changes can be developed safely and integrated with explicit failure handling.

## User Stories

- 1.4.1 (implemented) — As a coder, I want a dedicated task worktree so my changes do not interfere with other tasks.
- 1.4.2 (planned) — As a reviewer/supervisor, I want merge operations constrained to approved commits so integration remains deterministic.
- 1.4.3 (planned) — As a planner, I want reassignment rules (reuse vs fresh worktree) so rejected or failed tasks recover predictably.

## Success Criteria

- Claiming READY tasks provisions `task/<id>` worktrees from `integration` and records `base_commit`.
- Rejected-task reassignment preserves worktree for same coder and recreates it for a different coder.
- Integration failures (HEAD mismatch, merge conflict, integration-test failure) transition tasks to `INTEGRATION_FAILED`.
- Successful merges update task state to `MERGED`, clear task worktree state, and clean up worktree/branch artifacts.
- Validation enforces invariants: `IMPLEMENTING` tasks require worktrees; `MERGED` tasks must not retain worktrees.
