# Circuit Breaker

## Goal

Detect systemic failure patterns from anomaly signals and escalate to human checkpoints before the system continues unsafe or unproductive execution.

## User Stories

- 1.5.1 (implemented) — As an operator, I want pattern-triggered checkpoint escalation so systemic problems pause normal execution quickly.
- 1.5.2 (planned) — As a maintainer, I want configurable pattern thresholds so sensitivity can be tuned without code changes.
- 1.5.3 (planned) — As a planner, I want trend-based early warnings before hard triggers trip the system.

## Success Criteria

- Circuit-breaker analysis evaluates anomalies against deterministic pattern checks and severities.
- Triggered patterns generate a human-readable report and transition system mode to `CIRCUIT_BREAKER_TRIPPED`.
- Non-trigger analyses set circuit-breaker status to `OK`, append history, and clear `CIRCUIT_BREAKER_TRIPPED` mode to `RUNNING`; sprint `CHECKPOINT` (if set) still requires `liza resume`.
- Trigger history records timestamp, pattern, and severity for auditability.
- Aggregate sprint metrics support early warning interpretation (e.g., suspicious approval-rate signals).
