# Implementation Plan: Context Handoff as Blackboard Event

**Task:** code-planning-1
**Spec:** specs/build/4 - Sprint Retrospectives.md (section: "Context Handoff as Blackboard Event")
**Date:** 2026-03-17

## Current State Analysis

### Existing Handoff Mechanism

- `HandoffNote` struct (`internal/models/history.go:115`) has `Summary`, `NextAction` (required), plus optional `Approach`, `Blockers`, `FilesModified`, `NextSteps`
- `state.Handoff` is a `map[string]HandoffNote` — one entry per task, keyed by task ID, **overwritten** on each handoff (not append-only)
- `ops.Handoff()` writes one `HandoffNote` to `state.Handoff[taskID]` and sets `task.HandoffPending=true`
- `ops.ResumeHandoff()` clears `HandoffPending` but does NOT clear the HandoffNote
- Prompt builder (`internal/agent/prompt.go:153`) reads `state.Handoff[taskID]` to render resume context (coder-only)
- History entry for handoff is `handoff_initiated` with a flat concatenated `note` string
- `ops.SubmitForReview()` does NOT write any handoff/completion event
- `ops.MergeWorktree()` does NOT write any handoff/completion event
- Validation (`validateHandoff()`) checks `Summary`/`NextAction` are non-empty

### Key Problems the Spec Solves

1. **Not append-only**: Current map overwrites previous handoff — history is lost
2. **Not structured**: Rich fields (approach, blockers) exist on HandoffNote but history entry is flat string
3. **Missing on task completion**: Only context-exhaustion writes handoff data; task completion and submission write nothing
4. **Schema too narrow**: Doesn't capture "what worked" / "what failed and why" as distinct structured fields

## Design

### New `HandoffEvent` Schema

Replace `HandoffNote` with a new `HandoffEvent` struct on the Task:

```go
// HandoffEvent records a structured snapshot of agent epistemic state at a task lifecycle transition.
// Events are append-only and immutable — a task accumulates an ordered array.
type HandoffEvent struct {
    Timestamp  time.Time `yaml:"timestamp"`
    Agent      string    `yaml:"agent"`
    Trigger    string    `yaml:"trigger"`              // "context_exhaustion" | "submission" | "completion"

    // What was attempted and what worked (positive)
    Succeeded  []string  `yaml:"succeeded,omitempty"`

    // What was tried and failed, and why (negative)
    Failed     []string  `yaml:"failed,omitempty"`

    // Current hypothesis and recommended next step (optional)
    Hypothesis string    `yaml:"hypothesis,omitempty"`
    NextStep   string    `yaml:"next_step,omitempty"`

    // Files that matter, files that are dead ends (optional)
    KeyFiles   []string  `yaml:"key_files,omitempty"`
    DeadEnds   []string  `yaml:"dead_ends,omitempty"`
}
```

### Storage: Task-Level Array (Not Top-Level Map)

Move from `state.Handoff map[string]HandoffNote` (top-level) to `task.HandoffEvents []HandoffEvent` (task-level array). Rationale:

- Handoff events belong to a task's lifecycle, not to global state
- Append-only array naturally preserves ordered history
- Multiple handoff events per task (A exhausts context, B exhausts context, C completes) are first-class
- Simplifies validation: just check the array on each task rather than cross-referencing a global map

### Trigger Points

The spec says "mandatory in the submit/complete flow." The relevant code paths:

| Trigger | Code Path | When Written | Trigger Value |
|---------|-----------|-------------|---------------|
| Context exhaustion | `ops.Handoff()` | Agent calls `liza_handoff` | `context_exhaustion` |
| Submit for review | `ops.SubmitForReview()` | Agent calls `liza_submit_for_review` | `submission` |
| Task merge (completion) | `ops.MergeWorktree()` | Reviewer calls `liza_wt_merge` | `completion` |

**Note on "capture-only initially":** The spec says "Initially capture-only — no consumer reads these events at runtime." This means:
- `ops.Handoff()` writes a `HandoffEvent` (trigger=context_exhaustion) instead of writing `state.Handoff[taskID]`
- `ops.SubmitForReview()` writes a `HandoffEvent` (trigger=submission) as part of the state transition
- `ops.MergeWorktree()` writes a `HandoffEvent` (trigger=completion) as part of the merge state transition
- The prompt builder (`internal/agent/prompt.go`) continues to render resume context from the **latest** handoff event (backward compat)
- No other consumer reads handoff events (future: Sprint Analyzer)

### Migration: Deprecate `state.Handoff` and `HandoffNote`

- Remove `state.Handoff map[string]HandoffNote` from `State`
- Remove `HandoffNote` struct
- Add `HandoffEvents []HandoffEvent` to `Task` struct
- Add `HandoffEvent` struct to `models/`
- Update `HandoffPending` semantics: still needed for resume detection (set when trigger=context_exhaustion)

### MCP/CLI Interface Changes

- `liza_handoff` parameters: Replace `summary`/`next_action` with structured fields matching the schema. For backward compatibility and simplicity, keep `summary` and `next_action` as the required params but map them into the new schema internally. Add optional params: `succeeded`, `failed`, `hypothesis`, `key_files`, `dead_ends`.
- `liza_submit_for_review`: No new params needed — the submission event is auto-generated with trigger=submission. Coder doesn't need to provide handoff data at submission time (they already wrote a checkpoint).
- `liza_wt_merge`: No new params needed — completion event is auto-generated with trigger=completion by the reviewer.

### Validation Changes

- Remove `validateHandoff()` (validates `state.Handoff` map)
- Add validation for `task.HandoffEvents`: each entry must have timestamp, agent, trigger; trigger must be one of the allowed values
- Add validation: tasks in submitted/merged states should have at least one handoff event (enforcement of mandatory writes)
