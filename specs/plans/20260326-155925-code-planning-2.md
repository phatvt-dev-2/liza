# Phase 2: View Layer — Code Plan

**Task:** code-planning-2
**Spec:** specs/goals/20260326-tui.md
**Scope:** `internal/tui/view.go` rendering logic and adaptive layout. Depends on Phase 1 model/styles. No data fetching, no command execution, no integration.

---

## Overview

Implement the 7 rendering functions in `internal/tui/view.go` that compose the TUI dashboard. Each function reads model state (populated by Phase 1 types) and returns a styled string. The main `View()` method composes them into a vertical stack: Header → Alert banner → Agent panel → Task panel → Activity panel → Footer.

All rendering uses Lipgloss styles from `Styles` (Phase 1 Task 1) and column visibility from `ColumnTier`/`ColumnTierForWidth` (Phase 1 Task 3). The `Model` struct fields (Phase 1 Task 3) provide all data inputs.

## Dependencies & Imports

**Phase 1 types consumed (from merged code-planning-1):**
- `Styles` / `NewStyles(width)` — all Lipgloss style definitions
- `StatusColor(status)` / `StatusDot(status)` — status coloring and dot indicators
- `KeyMap` / `NewKeyMap()` — for help overlay key reference
- `Model` — all state fields (state, activities, alertBanner, inputMode, etc.)
- `ColumnTier` / `ColumnTierForWidth(width)` — progressive column disclosure

## Task 1: View scaffold + renderHeader

### Intent
Create `internal/tui/view.go` with the `View()` composition method and all 6 sub-renderer stubs, then fully implement `renderHeader` as the first real renderer.

### Design

#### View() composition (from spec §Adaptive Layout)

```go
// View renders the complete TUI dashboard.
// Vertical stack: Header → Alert banner → Agent panel → Task panel → Activity → Footer.
// The help overlay, when visible, replaces the activity panel area.
func (m Model) View() string
```

Layout calculation:
1. Fixed-height elements: header (1 line), footer (1 line), alert banner (0 or 1 line)
2. Remaining height split between: agent panel, task panel, activity panel
3. Agent panel height: min(len(agents)+3, remaining/3) — +3 for border+title+header row
4. Task panel height: min(len(tasks)+3, remaining/3)
5. Activity panel: remaining height after agents+tasks

When `m.ready == false`, return a centered "Loading..." message.

#### renderHeader (from spec §Header Bar)

```go
// renderHeader renders the header bar:
//   ⚡  LIZA  |  {goal.description}  |  sprint: {sprint.id}  |  {STATUS}
// Full-width, background-colored. STATUS colored per system mode.
func (m Model) renderHeader() string
```

System status derived from `m.state.Config.Mode`:
- `RUNNING` → `StatusColor("RUNNING")` (Cyan)
- `PAUSED` → `StatusColor("PAUSED")` (Yellow)
- `STOPPED` → `StatusColor("STOPPED")` (Red)
- `CIRCUIT_BREAKER_TRIPPED` → `StatusColor("STOPPED")` (Red)

Sprint checkpoint status from `m.state.Sprint.Status == "CHECKPOINT"` → override to Magenta.

Graceful nil handling: if `m.state` is nil, render "⚡  LIZA  |  Loading..."

#### Sub-renderer stubs

All other renderers defined as methods returning placeholder strings:
```go
func (m Model) renderAgentPanel(height int) string  { return "" }
func (m Model) renderTaskPanel(height int) string   { return "" }
func (m Model) renderActivityPanel(height int) string { return "" }
func (m Model) renderAlertBanner() string            { return "" }
func (m Model) renderFooter() string                 { return "" }
func (m Model) renderHelpOverlay(height int) string  { return "" }
```

### Scope
- **In scope:** `internal/tui/view.go` — View() method, renderHeader(), all sub-renderer stubs
- **Out of scope:** Actual sub-renderer implementations (Tasks 2-7), update logic, data fetching

### Done When
- `internal/tui/view.go` compiles as part of the `internal/tui` package
- `View()` returns a vertically composed string with header at top and footer at bottom
- `View()` returns "Loading..." when `m.ready == false`

## Task 2: renderAgentPanel — progressive column disclosure

**Depends on:** Task 1 (view.go scaffold)

### Intent
Implement the agent panel renderer with a full-width bordered table whose visible columns adapt to terminal width using progressive disclosure at 40/80/120/160 column thresholds.

### Design

```go
// renderAgentPanel renders the agent panel as a bordered table.
// Columns adapt to terminal width per spec §Agent Panel column priority table.
// Agents sorted by ID for stable display order.
func (m Model) renderAgentPanel(height int) string
```

#### Column tiers (from spec §Agent Panel)

| ColumnTier | Columns rendered |
|------------|-----------------|
| Minimal (< 80) | ID, STATUS (dot + text) |
| Standard (≥ 80) | + ROLE, CURRENT_TASK |
| Wide (≥ 120) | + TIME_ON_TASK, HEARTBEAT |
| Full (≥ 160) | + PID, CONTEXT |

#### Column rendering details

- **ID**: Agent ID string, left-aligned
- **STATUS**: `StatusDot(status) + " " + status` — dot colored per unified palette (Phase 1 `StatusColor`), filled ● for active (WORKING, REVIEWING, PLANNING, STARTING), hollow ○ for idle (IDLE, WAITING, HANDOFF)
- **ROLE**: `agent.Role` string
- **CURRENT_TASK**: `*agent.CurrentTask` or "—" if nil
- **TIME_ON_TASK**: Computed from task's `Created` timestamp via `render.FormatDuration(time.Since(task.Created))`, or "—" if no current task. Note: this is an approximation using task creation time; a more precise metric would use claim history, but that's sufficient for dashboard display.
- **HEARTBEAT**: `render.FormatDuration(time.Since(agent.Heartbeat)) + " ago"`, or "—" if zero
- **PID**: `agent.PID` as string, or "—" if 0
- **CONTEXT**: `fmt.Sprintf("%d%%", agent.ContextPercent)` or "—" if 0

#### Panel structure

```
● AGENTS
  ID           STATUS        ROLE              CURRENT_TASK
  agent-1      ● WORKING     coder             task-1
  agent-2      ○ IDLE        reviewer          —
```

- Panel title: "● AGENTS" using `PanelTitle` style
- Column headers: dimmed/bold row
- Data rows: one per agent, sorted by agent ID (`sort.Strings` on map keys)
- Bordered using `AgentPanel` style from Styles
- Column widths: proportionally distribute available width (after borders/padding) across visible columns
- Height parameter truncates rows if agents exceed available space (show first N agents that fit)

### Scope
- **In scope:** `internal/tui/view.go` — replace `renderAgentPanel` stub with full implementation
- **Out of scope:** Agent data fetching (Phase 3), agent status transitions (update.go)

### Done When
- `renderAgentPanel` produces a bordered table with "● AGENTS" title
- At width < 80: only ID and STATUS columns visible
- At width ≥ 80: ROLE and CURRENT_TASK columns added
- At width ≥ 120: TIME_ON_TASK and HEARTBEAT columns added
- At width ≥ 160: PID and CONTEXT columns added
- Agent status dots use correct filled/hollow indicator per active/idle state
- Status text is colored per unified palette via `StatusColor`
- Agents sorted by ID for stable ordering
- Empty state (no agents) renders panel with "No agents" message
- `go vet ./internal/tui/...` passes
- Unit tests verify: column count at each tier (2/4/6/8 columns at <80/≥80/≥120/≥160), status dot is "●" for WORKING and "○" for IDLE, agents appear sorted by ID

## Task 3: renderTaskPanel — progressive columns + sprint metrics

**Depends on:** Task 2 (shared file `view.go`)

### Intent
Implement the task panel renderer with a full-width bordered table, progressive column disclosure at 60/80/120/160, sprint metrics in the panel header, and terminal tasks dimmed at the bottom.

### Design

```go
// renderTaskPanel renders the task panel as a bordered table.
// Columns adapt to terminal width per spec §Task Panel column priority table.
// Terminal tasks (MERGED, ABANDONED, SUPERSEDED) shown dimmed at bottom.
// Panel header includes sprint metrics.
func (m Model) renderTaskPanel(height int) string
```

#### Column tiers (from spec §Task Panel)

| ColumnTier | Columns rendered |
|------------|-----------------|
| Minimal (< 80) | ID, STATUS (color-coded dot + text) |
| Standard (≥ 80) | + ATTEMPT, ASSIGNED_TO |
| Wide (≥ 120) | + AGE, DESCRIPTION (truncated to fit) |
| Full (≥ 160) | + REVIEWING_BY, DEPS, TIME_IN_STATUS |

#### Column rendering details

- **ID**: Task ID string, left-aligned
- **STATUS**: `StatusDot(status) + " " + status` — colored per unified palette pattern matching
- **ATTEMPT** (labeled ATT): `task.Attempt` + "." + `task.Iteration` (e.g., "1.2")
- **ASSIGNED_TO**: `*task.AssignedTo` or "—" if nil
- **AGE**: `render.FormatDuration(time.Since(task.Created))`
- **DESCRIPTION**: `task.Description`, truncated with "…" if exceeds column width
- **REVIEWING_BY**: `*task.ReviewingBy` or "—" if nil
- **DEPS**: `strings.Join(task.DependsOn, ",")` or "—" if empty
- **TIME_IN_STATUS**: Computed from task history (last status change timestamp) via `render.FormatDuration`. Falls back to task age if history is empty.

#### Panel header with sprint metrics (from spec §Task Panel)

```
✔ TASKS                                          3/5 done │ 1 blocked │ 72% approval
```

Sprint metrics sourced from `m.state.Sprint.Metrics`:
- `{Metrics.TasksDone}/{len(Sprint.Scope.Planned)} done`
- `{Metrics.TasksBlocked} blocked`
- `{Metrics.TaskOutcomeApprovalRatePercent}% approval`

Metrics right-aligned on the same line as panel title.

#### Task ordering

1. Active tasks first (non-terminal), sorted by priority then ID
2. Terminal tasks at bottom (MERGED, ABANDONED, SUPERSEDED), rendered with `Dimmed` style
3. Height parameter truncates rows if tasks exceed available space

- **Out of scope:** Task data fetching (Phase 3), task status transitions (update.go)

- Sprint metrics show `{done}/{total} done │ {blocked} blocked │ {rate}% approval` right-aligned
- At width < 80: only ID and STATUS columns visible
- At width ≥ 80: ATTEMPT and ASSIGNED_TO columns added
- At width ≥ 120: AGE and DESCRIPTION columns added
- At width ≥ 160: REVIEWING_BY, DEPS, and TIME_IN_STATUS columns added

## Task 4: renderActivityPanel — activity feed display

**Depends on:** Task 3 (shared file `view.go`)

### Intent
Implement the activity panel renderer that displays the merged chronological activity feed with proper formatting for each source type (log events, anomaly alerts, blackboard anomalies).

### Design

```go
// renderActivityPanel renders the activity feed as a bordered panel.
// Displays the tail of m.activities (most recent entries that fit in height).
// Three source formats per spec §Activity Panel.
func (m Model) renderActivityPanel(height int) string
```

#### Entry formatting (from spec §Activity Panel)

Three display formats based on `ActivityEntry.Source`:

1. **Log events** (`Source == "log"`):
   ```
   HH:MM:SS  {agent}        {action}      [{task}]  {detail}
   ```

2. **Anomaly alerts** (`Source == "alert"`):
   ```
   HH:MM:SS  ⚠️/🚨  {category}: {message}
   ```
   Level icon colored: ⚠️ in yellow, 🚨 in red.

3. **Blackboard anomalies** (`Source == "anomaly"`):
   ```
   HH:MM:SS  ⚠️  {reporter}: {type} [{task}]  {details}
   ```

#### Display behavior

- Auto-scroll: display the tail (last N entries that fit in `height` lines)
- The `height` parameter determines how many entries to show (panel height minus border/title lines)
- Panel title: "⚡ ACTIVITY" using `PanelTitle` style
- Bordered using `ActivityPanel` style from Styles
- Timestamp formatted as `HH:MM:SS` (UTC): `entry.Timestamp.UTC().Format("15:04:05")`
- Column alignment: fixed-width timestamp (8 chars), then source-specific columns padded for alignment

Note: The 200-entry buffer is managed by the update layer (Phase 3/4) when appending entries to `m.activities`. The view layer simply renders whatever is in the slice.

### Scope
- **In scope:** `internal/tui/view.go` — replace `renderActivityPanel` stub with full implementation
- **Out of scope:** Activity entry population (Phase 3 data layer), buffer management (Phase 3/4 update layer)

### Done When
- `renderActivityPanel` produces a bordered panel with "⚡ ACTIVITY" title
- Log entries formatted as `HH:MM:SS  {agent}  {action}  [{task}]  {detail}`
- Alert entries formatted as `HH:MM:SS  {level}  {category}: {message}` with level colored
- Anomaly entries formatted as `HH:MM:SS  ⚠️  {reporter}: {type} [{task}]  {details}`
- Only the last N entries that fit in `height` are displayed (tail/auto-scroll behavior)
- Empty state (no activities) renders panel with empty body
- `go vet ./internal/tui/...` passes
- Unit tests verify: log entry format includes timestamp+agent+action+task+detail, alert entry includes colored level icon, tail display shows last N entries when activities exceed height, empty activities produces valid bordered panel

specs/goals/20260326-tui.md §Activity Panel

## Task 6: renderFooter — context-sensitive footer bar

**Depends on:** Task 5 (shared file `view.go`)

### Intent
Implement the footer bar renderer with context-sensitive display that transforms based on input mode and shows transient command results.

### Design

```go
// renderFooter renders the context-sensitive footer bar.
// Normal mode: keybinding hints. Input mode: inline prompt. Form mode: form hints.
// Includes transient command result display (3s visibility).
func (m Model) renderFooter() string
```

#### Normal mode (from spec §Footer Bar)

```
[s] spawn  [p] pause  [r] resume  [a] add  [c] checkpoint  [?] help  [q] quit  [Q] stop
```

Renders keybinding hints from `m.keys.ShortHelp()`. Each binding formatted as `[{key}] {description}` using `FooterKey` and `FooterDesc` styles. Hints separated by double space.

#### Inline input mode (from spec §Input Mode Tier 1)

When `m.inputMode == InputModeInline`:
```
Role: epic-planner█                              [Tab] complete  [Enter] confirm  [Esc] cancel
```

Left side: prompt label + input text (actual textinput widget content rendered by Phase 4).
Right side: input-mode hints.

For Phase 2 planning: the footer renderer checks `m.inputMode` and renders the appropriate layout. The actual textinput widget integration is Phase 4's scope; this task provides the layout frame that Phase 4 fills in.

#### Form mode (from spec §Input Mode Tier 2)

When `m.inputMode == InputModeForm`:
```
                                                  [Enter] submit  [Esc] cancel
```

Simplified hints only (form renders in its own overlay area).

#### Command result display (from spec §Command integration)

When `m.cmdResult != nil` and `time.Now().Before(m.cmdExpiry)`:
- Success: green-tinted message appended to footer
- Error: red-tinted message appended to footer

### Scope
- **In scope:** `internal/tui/view.go` — replace `renderFooter` stub with full implementation
- **Out of scope:** Textinput widget (Phase 4), Huh form rendering (Phase 4), command execution (Phase 4), cmd result expiry logic (Phase 3/4)

### Done When
- `renderFooter` in normal mode renders all 8 keybinding hints matching spec §Footer Bar format
- `renderFooter` in inline input mode renders input-mode hints (`[Tab] complete  [Enter] confirm  [Esc] cancel`)
- `renderFooter` in form mode renders form-mode hints (`[Enter] submit  [Esc] cancel`)
- Command result (when present and not expired) appears in footer with success=green / error=red tinting
- Footer is full-width using `FooterBar` style
- `go vet ./internal/tui/...` passes
- Unit tests verify: normal mode contains all 8 key labels ([s], [p], [r], [a], [c], [?], [q], [Q]), inline mode contains [Tab]/[Enter]/[Esc] hints, form mode contains [Enter]/[Esc] hints, active cmdResult appears in footer output, expired cmdResult does not appear
