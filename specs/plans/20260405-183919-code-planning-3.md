# Code Plan: arch_ref Model, Validation, and Propagation

**Task:** code-planning-3
**Spec:** specs/goals/20260405-architecture-step.md
**Scope:** arch_ref on OutputEntry and Task in models/task.go, set-task-output normalization via NormalizeSpecRef, validate_task.go file-existence check (same pattern as plan_ref), CLI flag and MCP handler exposure (cli-mcp-surface-sync), proceed.go first-hop (output[] entry to child) and second-hop (parent task to child) propagation. Covers spec items 5 and 6. No implementation.

## Context

The architecture step produces a design document that downstream agents (code-planners, coders, reviewers) need to reference. `arch_ref` is the durable path to this document, flowing through the pipeline via `output[]` entries and `proceed.go` propagation. This plan adds the field to the data model, wires it through the set-task-output interface, validates it on tasks, and propagates it through transitions.

### Key Codebase Observations

1. **`PlanRef` is the exact precedent**: `PlanRef` exists on both `OutputEntry` (task.go:232) and `Task` (task.go:204). `set_task_output.go:50-51` normalizes both `SpecRef` and `PlanRef` via `NormalizeSpecRef`. `validate_task.go:208-214` checks worktree prefix and file existence for `PlanRef`. `proceed.go:794` copies `entry.PlanRef` in `buildChildTask`, and line 821 copies `parent.PlanRef` in `buildOneToOneChild`. `arch_ref` mirrors this pattern exactly.

2. **OutputEntry has no file-existence check**: `validate_task.go:369-370` only checks worktree prefix on `OutputEntry.PlanRef`, not file existence. This matches the spec: "On output[] entries: no file-existence check (entries are not validated against disk)."

3. **`buildChildTask` lacks parent task access**: The function takes `entry models.OutputEntry` but not the parent `*models.Task` (proceed.go:777). For the second hop (code-planning → coding), the code-planner's `output[]` entries don't carry `arch_ref` — it's on the parent task's `ArchRef` field. A new parameter is needed to propagate the parent's `ArchRef` as a fallback when the entry's `ArchRef` is empty.

4. **`buildManyToOneChild` (CP2) doesn't need arch_ref**: The only current many-to-one transition (`us-to-coding`) creates architecture tasks, which produce `arch_ref` via `output[]` rather than consuming it. US tasks upstream don't carry `arch_ref`.

5. **MCP is the sole surface for set-task-output**: There is no CLI command for `set-task-output` (confirmed: grep of `cmd/liza/main.go` returns no matches). The MCP handler in `handlers_complex.go:189-216` delegates to `extractOutputEntries` (handlers_helpers.go:253-271) for parsing, and `ops.SetTaskOutput` for normalization. Only these two layers need updating.

6. **`add-tasks` does not need arch_ref**: The orchestrator creates architecture tasks via `add-tasks`, but architecture tasks don't receive `arch_ref` — they produce it. `AddTaskInput` (add_tasks.go:20-31) doesn't need the field. The `add-tasks` MCP handler and its `extractTaskInputs` helper are out of scope.

## Design Decisions

### D1: Add parentArchRef parameter to buildChildTask

The spec requires second-hop propagation: "proceed.go copies arch_ref from the parent code-planning task to child coding tasks." Since `buildChildTask` only receives `entry models.OutputEntry` (not the parent task), we add a `parentArchRef string` parameter. Logic: use `entry.ArchRef` if non-empty, else fall back to `parentArchRef`. This keeps the function self-contained — callers pass `task.ArchRef` from the parent.

Alternative considered: inject parent's ArchRef into the entry before calling buildChildTask. Rejected: mutates the caller's data, unclear ownership.

### D2: Normalize arch_ref in SetTaskOutput, not in proceed.go

Normalization happens once at input time (`set_task_output.go`), not at propagation time. `proceed.go` uses `paths.NormalizeSpecRef` defensively when copying from entries (same as spec_ref/plan_ref), but the primary normalization is at the API boundary. This follows the established pattern.

### D3: No arch_ref on buildManyToOneChild

`buildManyToOneChild` (CP2) creates architecture tasks from US cohorts. Architecture tasks produce `arch_ref` via `output[]`, they don't receive it. Adding arch_ref to `buildManyToOneChild` would be speculative — no current pipeline path supplies it. If a future many-to-one transition needs arch_ref inheritance, it's a separate change.
