# Plan: Claim-Path Review-Budget Exhaustion Blocking & Attempt-2 Prompt Feedback

**Task:** code-planning-2
**Spec:** tofix.md lines 36–42, 52–58, 67–72
**Scope:** internal/ops/claim_task.go, internal/ops/transition_attempt.go, internal/agent/prompt.go, internal/agent/prompt_test.go

## Problem

Two related bugs in the attempt lifecycle:

### Bug A: Review-only exhaustion on attempt 2 never reaches BLOCKED

`classifyLimitEscalation()` correctly returns `LimitActionBlocked` when review cycles are exhausted on attempt 2. But the caller delegates the state transition to `enforceRejectedIterationLimit()` (claim_task.go:480), which re-checks only `task.Iteration >= blockedLimit`. When review cycles are exhausted but iterations are not, this guard fails and returns a "race condition" error — the task stays `CODE_REJECTED` and is re-claimable indefinitely.

### Bug B: Attempt-2 prompts discard actual reviewer rejection feedback

`TransitionToNewAttempt` Phase 3 (transition_attempt.go:157) clears `RejectionReason`. The `new_attempt` history entry only carries the synthetic escalation reason (e.g., "review budget exhausted"). The prompt builder at attempt 2, iteration 1 finds `PriorAttemptOutcome` (the synthetic reason) but never surfaces the reviewer's actual feedback — `PriorRejection` is gated by `task.Iteration > 1`, which is false at the start of a new attempt.

## Design

### Fix A: Generalize enforceRejectedIterationLimit to handle all escalation types

**File:** `internal/ops/claim_task.go`

Rename `enforceRejectedIterationLimit` → `enforceBlockedEscalation`. Accept the pre-computed `limitEscalation` as a parameter. Inside the `bb.Modify` closure, re-verify by calling `classifyLimitEscalation()` with current state values (TOCTOU safety). Use the escalation's reason and questions for the block transition instead of hardcoding iteration-only values.

**Changes:**

1. Change function signature from:
   ```go
   func enforceRejectedIterationLimit(
       bb *db.Blackboard, taskID, agentID string,
       expectedStatus models.TaskStatus,
       pipelineTransitions map[models.TaskStatus][]models.TaskStatus,
   ) (int, int, error)
   ```
   to:
   ```go
   func enforceBlockedEscalation(
       bb *db.Blackboard, taskID, agentID string,
       expectedStatus models.TaskStatus,
       pipelineTransitions map[models.TaskStatus][]models.TaskStatus,
       escalation limitEscalation,
   ) error
   ```

2. Replace the iteration-only guard (`task.Iteration < blockedLimit`) with a full re-verification:
   ```go
   iterationLimit := effectiveCoderIterationLimit(task, state.Config)
   reviewLimit := effectiveReviewCycleLimit(state.Config)
   _, stillEscalated := classifyLimitEscalation(
       task.ReviewCyclesCurrent, reviewLimit,
       task.Iteration, iterationLimit,
       task.EffectiveAttempt(),
   )
   if !stillEscalated {
       return fmt.Errorf("race condition: escalation condition no longer holds")
   }
   ```

3. Use `escalation.reason` and `escalation.questions` for `BlockedReason` and `BlockedQuestions` (replacing hardcoded `iterationLimitBlockedReason` / `defaultIterationLimitBlockedQuestions`).

4. Update call site (claim_task.go:178) to pass `escalation` and handle the new `error`-only return:
   ```go
   case LimitActionBlocked:
       if err := enforceBlockedEscalation(bb, taskID, agentID, taskStatus, pipelineTransitions, escalation); err != nil {
           return nil, fmt.Errorf("failed to enforce escalation limit: %w", err)
       }
       return nil, &PreconditionError{Reason: fmt.Sprintf(
           "task %s transitioned to BLOCKED — %s", taskID, escalation.reason,
       )}
   ```

**Backward compatibility:** For iteration-only escalation, `classifyLimitEscalation` returns `iterationLimitBlockedReason()` which contains "max iterations" — existing tests checking for that string in `BlockedReason` continue to pass. The error message changes from "reached max iterations (X/Y) and was transitioned to BLOCKED" to "transitioned to BLOCKED — max iterations reached without approval (X/Y)", but existing tests check for "transitioned to BLOCKED" which is preserved.

### Fix B: Preserve rejection feedback in new_attempt history entry

**File:** `internal/ops/transition_attempt.go`

In Phase 1, before resetting counters, capture the current `RejectionReason` and store it in the `new_attempt` history entry's `Note` field. Phase 3 still clears `RejectionReason` for lifecycle cleanliness — the feedback is preserved in history.

**Changes:**

1. In Phase 1 (inside bb.Modify, before mutations), capture rejection reason:
   ```go
   var rejectionNote *string
   if task.RejectionReason != nil && *task.RejectionReason != "" {
       rr := *task.RejectionReason
       rejectionNote = &rr
   }
   ```

2. Include in history entry:
   ```go
   task.History = append(task.History, models.TaskHistoryEntry{
       Time:   now,
       Event:  models.TaskEventNewAttempt,
       Reason: &reason,
       Note:   rejectionNote,
   })
   ```

### Fix C: Surface prior rejection feedback in attempt-2 prompts

**Files:** `internal/agent/prompt.go`, `internal/prompts/role_context.go`, `internal/prompts/templates/blocks/prior_attempt.tmpl`, `internal/agent/prompt_test.go`

Add `PriorAttemptRejection` field to `RoleContextData`. Extract it from the `new_attempt` history entry's `Note`. Render it in the `prior_attempt.tmpl` template.

**Changes:**

1. `internal/prompts/role_context.go`: Add field after `PriorAttemptOutcome`:
   ```go
   PriorAttemptRejection string // reviewer feedback from prior attempt (empty unless AttemptNum == 2 and Note present)
   ```

2. `internal/agent/prompt.go`: In the attempt-2 history scan (lines 155–161), also extract `Note`:
   ```go
   if data.AttemptNum == 2 {
       for i := len(task.History) - 1; i >= 0; i-- {
           if task.History[i].Event == models.TaskEventNewAttempt && task.History[i].Reason != nil {
               data.PriorAttemptOutcome = *task.History[i].Reason
               if task.History[i].Note != nil {
                   data.PriorAttemptRejection = *task.History[i].Note
               }
               break
           }
       }
   }
   ```

3. `internal/prompts/templates/blocks/prior_attempt.tmpl`: Render rejection feedback when present:
   ```
   {{- define "prior-attempt" -}}
   {{- if and (eq .AttemptNum 2) (ne .PriorAttemptOutcome "")}}

   === PRIOR ATTEMPT OUTCOME ===
   {{.PriorAttemptOutcome}}
   {{- if ne .PriorAttemptRejection ""}}
   LAST REVIEWER FEEDBACK (from prior attempt):
   {{.PriorAttemptRejection}}
   {{- end}}
   Do NOT repeat the same approach. The prior attempt was exhausted — try a fundamentally different strategy.
   {{- end}}
   {{- end -}}
   ```

4. `internal/agent/prompt_test.go`: The existing `TestBuildTaskRoleContextData_PriorRejectionGate_Attempt2Iteration1_Empty` (line 1048) remains correct — `PriorRejection` (same-attempt, iteration > 1 gate) is still empty at attempt 2 iteration 1. No change to this test.

**Note on PriorRejection vs PriorAttemptRejection:** These are distinct fields:
- `PriorRejection` — feedback from the current attempt's prior iteration (gated by `Iteration > 1`)
- `PriorAttemptRejection` — feedback from the prior attempt's final rejection (gated by `AttemptNum == 2`, sourced from history Note)
