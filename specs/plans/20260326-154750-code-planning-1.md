# Phase 1: TUI Foundation — Code Plan

**Task:** code-planning-1
**Spec:** specs/build/5 - New console.md
**Scope:** `internal/tui/` package skeleton — `model.go`, `styles.go`, `keymap.go` structure and type definitions. No view rendering, no data fetching, no command execution, no integration.

---

## Overview

Create the foundational `internal/tui/` package with three files containing the type definitions, constants, and structural scaffolding that all subsequent TUI phases build on. This phase establishes:

1. **styles.go** — Unified color palette and Lipgloss style definitions with status pattern matching
2. **keymap.go** — Key binding definitions using Bubbles `key.Binding`
3. **model.go** — Bubbletea `Model` struct, message types, input mode enum, and Init/Update/View stubs

## Dependencies & Imports

The project already has these dependencies (go.mod):
- `github.com/charmbracelet/lipgloss v1.1.0` (direct)
- `github.com/charmbracelet/bubbletea v1.3.6` (indirect → promoted to direct by `go mod tidy`)
- `github.com/charmbracelet/bubbles v0.21.1` (indirect → promoted to direct)

## Task 1: styles.go — Unified Color Palette and Lipgloss Styles

### Intent
Define the unified status color palette from the spec and Lipgloss style constructors for all TUI panels, enabling consistent visual theming across the entire TUI.

### Design

#### Color Constants (from spec §Unified Status Color Palette)

Define named `lipgloss.Color` constants for each semantic group:

| Semantic group       | Constant name          | Color value   |
|----------------------|------------------------|---------------|
| Active work          | `ColorActive`          | Cyan          |
| Planning/draft       | `ColorPlanning`        | Yellow        |
| Review               | `ColorReview`          | Blue          |
| Idle/waiting         | `ColorIdle`            | Gray          |
| Handoff              | `ColorHandoff`         | Magenta       |
| Approved/done        | `ColorApproved`        | Green         |
| Partially done       | `ColorPartialApproved` | Green (dim)   |
| Rejected/blocked     | `ColorRejected`        | Red           |
| Terminal/inactive    | `ColorTerminal`        | Gray          |
| Bare draft           | `ColorBareDraft`       | Dim white     |
| Fallback             | `ColorFallback`        | White         |

Use ANSI 256 color codes via `lipgloss.Color("N")` for terminal compatibility.

#### Status Pattern Matching (from spec §Unified Status Color Palette, matching order)

```go
// StatusColor returns the lipgloss.Color for a given status string.
// Matching order: exact match → suffix → prefix → fallback.
// This ensures pipeline-configurable statuses render correctly without TUI changes.
func StatusColor(status string) lipgloss.Color
```

Pattern rules (from spec):
- `IMPLEMENTING_*` → Cyan (active)
- `DRAFT_*` (qualified, i.e., has suffix) → Yellow (planning)
- `*_PLANNING` → Yellow (planning)
- `REVIEWING_*`, `*_TO_REVIEW`, `*_READY_FOR_REVIEW` → Blue (review)
- `*_APPROVED` → Green (approved)
- `MERGED` → Green (dim)
- `*_PARTIALLY_APPROVED` → Green (dim)
- `BLOCKED`, `*_REJECTED`, `INTEGRATION_FAILED` → Red (rejected)
- `ABANDONED`, `SUPERSEDED` → Gray (terminal)
- `DRAFT` (unqualified) → Dim white (bare draft)
- Any unrecognized → White (fallback)

Agent statuses use direct mapping (not pattern): WORKING→Cyan, PLANNING/STARTING→Yellow, REVIEWING→Blue, IDLE/WAITING→Gray, HANDOFF→Magenta.

System statuses: RUNNING→Cyan, PAUSED→Yellow, CHECKPOINT→Magenta, STOPPED→Red.

```go
// StatusDot returns "●" for active statuses or "○" for idle/waiting.
func StatusDot(status string) string
```

#### Lipgloss Style Definitions

```go
// Styles holds all Lipgloss style definitions for the TUI.
// Constructed via NewStyles(width int) to adapt to terminal width.
type Styles struct {
    // Header bar: full-width, background-colored
    HeaderBar     lipgloss.Style
    HeaderTitle   lipgloss.Style
    HeaderStatus  lipgloss.Style

    // Panel borders: rounded border, full-width
    AgentPanel    lipgloss.Style
    TaskPanel     lipgloss.Style
    ActivityPanel lipgloss.Style

    // Panel titles
    PanelTitle    lipgloss.Style

    // Footer bar: full-width
    FooterBar     lipgloss.Style
    FooterKey     lipgloss.Style
    FooterDesc    lipgloss.Style

    // Alert banner: highlighted bar
    AlertBanner   lipgloss.Style

    // Help overlay
    HelpOverlay   lipgloss.Style

    // Status text: colored per palette
    StatusActive    lipgloss.Style
    StatusPlanning  lipgloss.Style
    StatusReview    lipgloss.Style
    StatusIdle      lipgloss.Style
    StatusHandoff   lipgloss.Style
    StatusApproved  lipgloss.Style
    StatusRejected  lipgloss.Style
    StatusTerminal  lipgloss.Style
    StatusBareDraft lipgloss.Style
    StatusFallback  lipgloss.Style

    // Dim style for terminal tasks (MERGED, ABANDONED, SUPERSEDED shown dimmed)
    Dimmed        lipgloss.Style
}

// NewStyles creates a Styles instance adapted to the given terminal width.
func NewStyles(width int) Styles
```

### Scope
- **In scope:** `internal/tui/styles.go` only
- **Out of scope:** Actual panel rendering (owned by later view.go tasks), model integration

### Done When
- `internal/tui/styles.go` compiles as part of the `internal/tui` package
- All 11 semantic color constants from the spec palette are defined
- `StatusColor(status string) lipgloss.Color` function implements exact→suffix→prefix→fallback matching order per spec
- `StatusDot(status string) string` returns "●" or "○" based on active vs idle status
- `NewStyles(width int) Styles` returns a populated Styles struct with all panel, header, footer, alert, and status styles
- `go vet ./internal/tui/...` passes
- Unit tests verify StatusColor pattern matching: exact match (`BLOCKED`→red), suffix match (`CODE_REJECTED`→red), prefix match (`IMPLEMENTING_CODE`→cyan), qualified DRAFT (`DRAFT_CODING_PLAN`→yellow vs `DRAFT`→dim white), fallback (unknown status→white)

## Task 2: keymap.go — Key Binding Definitions

### Intent
Define all keyboard bindings from the spec using Bubbles `key.Binding`, organized in a `KeyMap` struct that implements the `help.KeyMap` interface for the help overlay.

### Design

#### KeyMap Struct (from spec §Keybindings)

```go
// KeyMap defines all key bindings for the TUI.
// Implements the help.KeyMap interface for the help overlay.
type KeyMap struct {
    Spawn      key.Binding // s — spawn agent
    Pause      key.Binding // p — pause system
    Resume     key.Binding // r — resume system
    AddTask    key.Binding // a — add task (Huh form)
    Checkpoint key.Binding // c — sprint checkpoint
    Help       key.Binding // ? — toggle help overlay
    Quit       key.Binding // q — quit TUI (system continues)
    Stop       key.Binding // Q — stop system then quit
}
```

#### Constructor

```go
// NewKeyMap returns a KeyMap with the default key bindings from the spec.
func NewKeyMap() KeyMap
```

Each binding uses `key.NewBinding(key.WithKeys(...), key.WithHelp(...))` matching the spec's footer display: `[s] spawn  [p] pause  [r] resume  [a] add  [c] checkpoint  [?] help  [q] quit  [Q] stop`.

#### help.KeyMap Interface

```go
// ShortHelp returns the key bindings shown in the footer bar.
func (k KeyMap) ShortHelp() []key.Binding

// FullHelp returns grouped key bindings for the help overlay.
func (k KeyMap) FullHelp() [][]key.Binding
```

### Scope
- **In scope:** `internal/tui/keymap.go` only
- **Out of scope:** Input mode handling (inline prompt, Huh form), key event processing in Update — those belong to later update.go tasks

### Done When
- `internal/tui/keymap.go` compiles as part of the `internal/tui` package
- `KeyMap` struct has all 8 key bindings from the spec (s, p, r, a, c, ?, q, Q)
- `NewKeyMap()` returns a fully initialized `KeyMap` with correct keys and help text matching spec §Footer Bar
- `ShortHelp()` returns all 8 bindings in spec footer order
- `FullHelp()` returns bindings grouped by category (actions, system, navigation)
- `go vet ./internal/tui/...` passes
- Unit tests verify: `NewKeyMap()` bindings match expected keys, `ShortHelp()` returns 8 bindings, `FullHelp()` returns non-empty groups

## Task 3: model.go — Bubbletea Model, Message Types, and Stubs

**Depends on:** Task 1 (styles.go), Task 2 (keymap.go) — references `Styles` and `KeyMap` types in Model struct fields.

### Intent
Define the main Bubbletea Model struct holding all TUI state, the custom message types for the event loop, the input mode enum, and stub implementations of Init/Update/View that compile and satisfy `tea.Model`.

### Design

#### Input Mode Enum (from spec §Input Mode)

```go
// InputMode represents the current input mode of the TUI.
type InputMode int

const (
    InputModeNormal   InputMode = iota // Normal keybinding mode
    InputModeInline                    // Inline text prompt (spawn role, pause reason)
    InputModeForm                     // Huh form overlay (add task)
)
```

#### Message Types (from spec §Data Sources & Refresh)

```go
// StateMsg carries a fresh state snapshot after a blackboard change.
type StateMsg struct {
    State *models.State
}

// TickMsg signals a periodic 10s poll tick for anomaly checks and heartbeat refresh.
type TickMsg time.Time

// AlertMsg carries an anomaly alert to display in the activity feed / banner.
type AlertMsg struct {
    Timestamp time.Time
    Level     string // "⚠️" or "🚨"
    Category  string
    Message   string
}

// CmdResultMsg carries the result of a command execution (success or error).
// Displayed as transient status in the footer for 3 seconds per spec §Command integration.
type CmdResultMsg struct {
    Success bool
    Message string
}

// LogEntriesMsg carries new log entries from log.yaml.
type LogEntriesMsg struct {
    Entries []log.Entry
}
```

#### Activity Entry (from spec §Activity Panel)

```go
// ActivityEntry is a unified entry in the activity feed, merging log events,
// anomaly alerts, and blackboard anomalies into a single chronological list.
type ActivityEntry struct {
    Timestamp time.Time
    Source    string // "log", "alert", "anomaly"
    Agent    string // empty for alerts/anomalies
    Action   string
    Task     string // empty if not task-specific
    Detail   string
    Level    string // empty for log entries, "⚠️" or "🚨" for alerts
}
```

#### Column Visibility (from spec §Agent Panel, §Task Panel)

```go
// ColumnTier defines which columns are visible at a given terminal width.
type ColumnTier int

const (
    ColumnTierMinimal  ColumnTier = iota // < 80 cols: ID, STATUS only
    ColumnTierStandard                   // ≥ 80 cols: + ROLE, CURRENT_TASK / ATTEMPT, ASSIGNED_TO
    ColumnTierWide                       // ≥ 120 cols: + TIME_ON_TASK, HEARTBEAT / AGE, DESCRIPTION
    ColumnTierFull                       // ≥ 160 cols: + PID, CONTEXT / REVIEWING_BY, DEPS, TIME_IN_STATUS
)

// ColumnTierForWidth returns the column tier for a given terminal width.
func ColumnTierForWidth(width int) ColumnTier
```

#### Model Struct (from spec §Architecture — "main Bubbletea model")

```go
// Model is the main Bubbletea model for the Liza TUI.
// It holds all state needed to render the dashboard and process input.
type Model struct {
    // State data
    state       *models.State     // current blackboard state snapshot
    activities  []ActivityEntry   // merged activity feed (last 200 entries per spec)
    logPosition int64             // byte offset for incremental log.yaml reads

    // Layout
    width       int               // terminal width
    height      int               // terminal height
    columnTier  ColumnTier        // current column visibility tier

    // Input
    inputMode   InputMode         // current input mode
    keys        KeyMap            // key bindings

    // Visual
    styles      Styles            // Lipgloss styles (adapted to width)

    // Alerts
    alertBanner *ActivityEntry    // current critical alert (auto-dismiss after 10s)
    alertExpiry time.Time         // when to auto-dismiss the alert banner

    // Command feedback
    cmdResult   *CmdResultMsg     // transient command result (3s display)
    cmdExpiry   time.Time         // when to clear cmdResult

    // Help
    showHelp    bool              // help overlay visible

    // Watch state (for anomaly throttling, same as WatchConfig.StateCache)
    stateCache  map[string]time.Time

    // Lifecycle
    ready       bool              // true after first state load
}
```

#### Constructor

```go
// New creates a new Model with default state.
// projectRoot is used to locate state.yaml, log.yaml, and alerts.log.
func New(projectRoot string) Model
```

#### tea.Model Interface Stubs

```go
// Init returns the initial command (start watcher subscription + first state read + tick timer).
// Stub: returns nil. Actual implementation in commands.go phase.
func (m Model) Init() tea.Cmd

// Update handles messages and returns the updated model.
// Stub: returns m, nil. Actual implementation in update.go phase.

// View renders the TUI.
// Stub: returns "Loading...". Actual implementation in view.go phase.
func (m Model) View() string
```

### Scope
- **In scope:** `internal/tui/model.go` only
- **Out of scope:** Actual Update logic (owned by update.go), actual View rendering (owned by view.go), Cmd functions (owned by commands.go), Huh form / textinput model fields (owned by later input handling tasks), integration with cmd_system.go

### Done When
- `internal/tui/model.go` compiles as part of the `internal/tui` package
- `Model` struct contains all fields listed above (state, activities, layout, input, visual, alerts, command feedback, help, watch state, lifecycle)
- `InputMode` enum has 3 values (Normal, Inline, Form)
- All 5 message types are defined (`StateMsg`, `TickMsg`, `AlertMsg`, `CmdResultMsg`, `LogEntriesMsg`)
- `ActivityEntry` struct captures the merged feed format from spec
- `ColumnTier` enum has 4 values with `ColumnTierForWidth` matching spec width thresholds (80, 120, 160)
- `New(projectRoot string)` returns a Model with initialized defaults (empty state, default KeyMap, default Styles)
- `Init()`, `Update()`, `View()` compile and satisfy `tea.Model` interface
- `go vet ./internal/tui/...` passes
- Unit tests verify: `ColumnTierForWidth` returns correct tiers at boundary widths (79→Minimal, 80→Standard, 119→Standard, 120→Wide, 159→Wide, 160→Full), `New()` returns a model satisfying `tea.Model` interface
