# Code Plan: Clean-Scan Terminal State & Integration Task Type

**Task:** code-planning-1
**Spec:** specs/goals/20260405-integration-pipeline.md
**Scope:** Implementation Cost items 4 (clean-scan completion path) and 1 (TDD check bypass via integration task type)

## Context

The integration sub-pipeline introduces a new pattern: the integration analyst may find zero issues (clean scan), requiring a completion path that bypasses per-subtask child creation. Today's schema and resolver assume all approved tasks either feed per-subtask transitions or reach MERGED. A clean scan with empty `output[]` currently fails at two points:

1. `SetTaskOutput` rejects empty output (`internal/ops/set_task_output.go:28-29`)
2. `Proceed` rejects empty output for per-subtask transitions (`internal/ops/proceed.go:146-148`)

The spec explicitly rejects relaxing these constraints (overloading "approved" to mean both "findings approved" and "no findings"). Instead, a dedicated terminal state (`INTEGRATION_ANALYSIS_CLEAN`) routes clean scans to completion without triggering per-subtask logic.

Additionally, the integration analyst is a doer role submitting without code changes. The TDD gate (`internal/ops/submit_review.go:122-132`) fires for `EffectiveType() == TaskTypeCoding` + doer + base_commit. Adding `TaskTypeIntegration` naturally bypasses this condition without a role-level exemption.

## Design Decisions

### D1: Clean state as RolePairStates optional field (not a hardcoded status)

The clean state follows the same declarative pattern as `partially-approved` and `reviewing-2`: an optional field in `RolePairStates` that role-pairs opt into. This keeps the pipeline schema generic — any future role-pair could declare a clean terminal state. The resolver treats it as terminal (empty transition list), just like approved.

### D2: SubmitVerdict routes to clean on empty output[]

The decision point for clean vs. approved routing is the `SubmitVerdict` function. When quorum is met (approval path, lines 245-249), the function checks:
- Does the role-pair declare a clean state?
- Is `task.Output` empty?

If both are true, the task transitions to the clean state instead of approved. This is the earliest point where the system can distinguish clean scans from findings-present scans without adding a flag or separate operation.

### D3: Clean tasks bypass merge entirely

`IsApprovedForMerge` (`internal/models/task.go:442-448`) only matches the role-pair's approved status. CLEAN tasks won't match, so the supervisor won't attempt to merge them. This is correct because the analyst has no code changes to merge. Worktree cleanup for CLEAN tasks is an operational concern handled by code-planning-4 (orchestrator lifecycle).

### D4: TaskTypeIntegration provides TDD bypass as side effect

The TDD gate condition is `task.EffectiveType() == models.TaskTypeCoding`. Adding `TaskTypeIntegration` to the type registry means integration tasks have `EffectiveType() == TaskTypeIntegration`, which != `TaskTypeCoding`, naturally bypassing TDD enforcement. No changes to `submit_review.go` are needed — only the type addition and workflow mapping.

### D5: Role constants added minimally in this phase

`taskWorkflows` requires role name strings. This plan adds `IntegrationAnalyst` and `IntegrationReviewer` as constants in `internal/roles/roles.go` (const declarations only). Full role registration (`validRoles`, `All()`) and pipeline.yaml definitions are in code-planning-2's scope. The constants are added here because `taskWorkflows` is the authoritative role-to-type mapping and must reference valid names.

## Cross-Cutting Concerns

### Worktree cleanup for CLEAN tasks

Tasks at CLEAN still have worktrees allocated by `ClaimTask`. `MergeWorktree` handles cleanup during merge, but CLEAN tasks skip merge. The orchestrator (code-planning-4) must handle worktree deletion for CLEAN tasks, likely via `DeleteWorktree` after detecting the terminal state.

### Sprint completion recognition

`SprintTerminalStates()` must include clean states so the sprint system recognizes CLEAN tasks as complete. This is handled in Task 3.

### ExecuteAvailableTransitions safety

`ExecuteAvailableTransitions` processes MERGED tasks and checks `resolver.AvailableTransitions(approvedStatus, ...)`. CLEAN tasks never reach MERGED, so they're never processed. No false-positive per-subtask attempts. No changes needed.

### Task 3: Resolver updates for clean terminal state

**Changes:**

New method:
```go
// CleanStatus returns the clean (terminal) state for the given role-pair.
// Returns an error if the role-pair does not declare a clean state.
func (r *Resolver) CleanStatus(rolePair string) (models.TaskStatus, error) {
    s, err := r.lookupStates(rolePair)
    if err != nil {
        return "", err
    }
    if s.Clean == "" {
        return "", fmt.Errorf("role-pair %q has no clean state declared", rolePair)
    }
    return models.TaskStatus(s.Clean), nil
}
```
