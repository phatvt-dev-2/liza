## Task 2: Update prompt.go — AttemptNum and PriorAttemptOutcome population

**Files:**
- `internal/agent/prompt.go` — fix AttemptNum computation, add PriorAttemptOutcome extraction
- `internal/agent/prompt_test.go` — add 2 tests

**Changes:**

1. **`internal/agent/prompt.go`** — Line 132: Replace `AttemptNum: len(task.Attempted) + 1` with `AttemptNum: task.EffectiveAttempt()`. The `Attempted` field is removed by CP2 Task 1 and `EffectiveAttempt()` is the new canonical source.

2. **`internal/agent/prompt.go`** — After the prior rejection block (after line 152), add PriorAttemptOutcome extraction for doer roles when AttemptNum == 2:
   ```go
   // Prior attempt outcome (attempt 2 only)
   if data.AttemptNum == 2 {
       for i := len(task.History) - 1; i >= 0; i-- {
           if task.History[i].Event == models.TaskEventNewAttempt && task.History[i].Reason != nil {
               data.PriorAttemptOutcome = *task.History[i].Reason
               break
           }
       }
   }
   ```
   This scans history backwards for the `new_attempt` event and extracts its `Reason` field. The `TaskHistoryEntry.Reason` field (`*string`, line 85 in history.go) is populated by `TransitionToNewAttempt` (CP2 Task 4, Phase 1: "Append history: `TaskEventNewAttempt` with reason string"). The extraction is unconditional on roleType because the template itself gates rendering on doer-only via pipeline context-sections.

3. **`internal/agent/prompt.go`** — Line 150: The prior rejection gate `task.Iteration > 1` is unchanged. At attempt 2 iteration 1, there's no within-attempt prior rejection (correct behavior per spec). At iteration 2+, prior rejection from within attempt 2 is shown.

4. **`internal/agent/prompt_test.go`** — Add 2 tests that call `buildTaskRoleContextData` directly (not `testBuildPrompt`) to verify data population independently of template rendering:
   - `TestBuildTaskRoleContextData_AttemptNum_UsesEffectiveAttempt`: Create state with task having `Attempt: 2`, `Iteration: 1`. Call `buildTaskRoleContextData`. Assert `data.AttemptNum == 2`. Also test with `Attempt: 0` → assert `data.AttemptNum == 1` (backward compat via `EffectiveAttempt()`).
   - `TestBuildTaskRoleContextData_PriorAttemptOutcome_Attempt2`: Create state with task having `Attempt: 2`, `Iteration: 1`, and a `new_attempt` history entry with `Reason: "review cycle limit reached after 5 rejections"`. Call `buildTaskRoleContextData`. Assert `data.PriorAttemptOutcome == "review cycle limit reached after 5 rejections"`. Also test with `Attempt: 1` → assert `data.PriorAttemptOutcome == ""` (not populated for attempt 1).

**done_when:** (a) `AttemptNum` in `buildTaskRoleContextData` uses `task.EffectiveAttempt()` instead of `len(task.Attempted) + 1`. (b) When `AttemptNum == 2`, `PriorAttemptOutcome` is populated from the last `new_attempt` history entry's `Reason` field. (c) `TestBuildTaskRoleContextData_AttemptNum_UsesEffectiveAttempt` passes in `internal/agent/prompt_test.go`: data from task with `Attempt: 2` has `AttemptNum == 2`, data from task with `Attempt: 0` has `AttemptNum == 1`. (d) `TestBuildTaskRoleContextData_PriorAttemptOutcome_Attempt2` passes in `internal/agent/prompt_test.go`: data from task at attempt 2 with `new_attempt` history entry has `PriorAttemptOutcome` matching the reason; data from task at attempt 1 has empty `PriorAttemptOutcome`.

**scope:** `internal/agent/prompt.go`, `internal/agent/prompt_test.go`.

**spec_ref:** Goal spec Change 8 — "Prompt building — internal/agent/prompt.go"

## Rejection Feedback Resolution

### Blocker 1 (iteration 1 → 2): Task 4 bundled too many independent changes

**Problem:** Old Task 4 spanned template display changes, a new template block, pipeline wiring, and integration tests across `internal/prompts/templates`, `internal/embedded`, and `internal/agent`, with 8 `done_when` clauses. That failed the scope-smell test (>2 packages and >4 conditions).

**Resolution:** Old Task 4 is decomposed into three focused tasks:
- **Task 4** (this plan): ATTEMPT display line in `assigned_task.tmpl` and `review_task.tmpl` — 2 template files + tests (5 conditions, 2 packages: templates + agent).
- **Task 5** (this plan): `prior_attempt.tmpl` creation + `pipeline.yaml` wiring — 1 new template + 1 yaml + tests (4 conditions).
- **Task 6** (this plan): LIMITS text update in `doer_state_transitions.tmpl` — 1 template + 1 test (2 conditions).

Each task has a single intent, and each task's tests are scoped to the behavior it owns.

### Blocker 2 (iteration 1 → 2): Task 4's validation didn't cover all claimed behaviors

**Problem:** Old Task 4 claimed `review_task.tmpl` attempt rendering and `pipeline.yaml` inclusion for code-planner, epic-planner, and us-writer, but only planned coder-path prompt tests. Reviewer prompts and non-coder doer prompts were unvalidated.

**Resolution:**
- **Task 4** now includes `TestBuildPrompt_ReviewerAttemptDisplay_Attempt2` — verifies the reviewer prompt at attempt 2 contains "ATTEMPT: 2" and "FINAL ATTEMPT" via the `review_task.tmpl` template.
- **Task 5** now includes `TestBuildPrompt_PriorAttemptOutcome_CodePlannerAttempt2` — verifies a non-coder doer role (code-planner) at attempt 2 contains PRIOR ATTEMPT OUTCOME, proving `prior-attempt` is wired through `pipeline.yaml` for that role.

Together, these tests prove: (a) reviewer templates render attempt display, (b) at least one non-coder doer role receives the prior-attempt block via pipeline wiring.
