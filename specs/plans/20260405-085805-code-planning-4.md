# Code Plan: Orchestrator Lifecycle and Integration Trigger

**Task:** code-planning-4
**Spec:** specs/goals/20260405-integration-pipeline.md
**Scope:** Orchestrator wake logic for all-coding-tasks-done detection, goal.base_commit snapshotting when first coding-subpipeline children are created, integration-analyst task spawning by orchestrator, end-to-end lifecycle from coding completion through integration scan to fix tasks to completion, BLOCKED escalation path when fixes require non-trivial work. Planning only, no implementation.

## Context

After all coding tasks for a goal are merged, the integration pipeline must activate. The orchestrator must detect this state, spawn an integration-analyst task, and the system must carry the lifecycle through analysis, review, fix task creation, fix implementation, and sprint completion.

Three new mechanisms are needed:
1. **Auto transition execution**: The `integration-to-fix` transition (defined in CP2 with `trigger: auto`) has no execution path today. `resolver.AvailableTransitions()` filters for `trigger: "manual"` only, and `ExecuteAvailableTransitions` uses that method.
2. **Coding-complete detection**: A new orchestrator wake trigger to distinguish "all coding tasks done, integration needed" from "sprint complete."
3. **Goal.base_commit snapshotting**: Capturing integration HEAD when the first coding-pair children are created, providing the diff base for the integration analyst.

### Dependency on CP1, CP2, CP3

**CP1** (MERGED plan) adds: `TaskTypeIntegration`, `Clean` field on `RolePairStates`, `CleanStatus()` resolver method, `SubmitVerdict` clean-scan routing, `SprintTerminalStates()` includes clean states.

**CP2** (MERGED plan) adds: integration-analyst and integration-reviewer roles in `pipeline.yaml`, integration-pair with states including `clean: INTEGRATION_ANALYSIS_CLEAN`, integration-subpipeline with `integration-to-fix` transition (`trigger: auto`, `cardinality: per-subtask`), role-pair reuse across sub-pipelines.

**CP3** (MERGED plan) adds: `Goal.BaseCommit *string` field on Goal struct, `CompletedTaskSummary` type and `RoleContextData` fields for integration context, `branch-integration-context` template, integration-reviewer review instructions, zero-diff submission test.

This plan builds on all three: CP1 provides the clean terminal state, CP2 provides the pipeline definitions, CP3 provides the schema and context templates. CP4 wires the lifecycle.

## Design Decisions

### D1: Auto transitions execute in reviewer PreWork, not via orchestrator checkpoint

Manual transitions (e.g., `code-plan-to-coding`) go through: orchestrator wakes on PLANNING_COMPLETE → checkpoint → human resumes → orchestrator PreWork calls `handleAvailableTransitions`. This is correct for manual transitions that need human validation.

Auto transitions (e.g., `integration-to-fix`) should execute without human intervention per spec: `trigger: auto # no human gate`. The reviewer supervisor already runs `handleApprovedMerges` in PreWork. After merging an integration-analyst task, the same PreWork calls `handleAvailableTransitions` (extended for auto transitions) to immediately create fix children. No checkpoint cycle needed.

Race window: between merge (state saved) and transition execution (state saved again), the orchestrator could poll and misclassify the state. This window is milliseconds vs. a 60s poll interval, making it acceptable. If the race triggers, the orchestrator sees unconsumed planning output and fires PLANNING_COMPLETE → human resumes → PreWork creates children (idempotent). No data loss.

### D2: AvailableAutoTransitions as separate resolver method

Adding a new `AvailableAutoTransitions` method parallels `AvailableTransitions`. Both use the same scan logic but filter on different trigger values. This keeps `AvailableTransitions` backward-compatible (callers expect only manual transitions) and makes the auto-vs-manual distinction explicit in calling code. Internal helper `availableTransitionsByTrigger(trigger string, ...)` avoids duplication.

### D3: Goal.base_commit snapshotted in ExecuteAvailableTransitions

The spec says goal.base_commit is "the integration branch HEAD at the time the orchestrator created the goal's first coding-subpipeline task." Coding-subpipeline tasks are coding-pair children created by `ExecuteAvailableTransitions` from code-planning output (the `code-plan-to-coding` transition). This is the natural snapshot point: the integration HEAD right before the first code change.

The snapshot happens once: if `state.Goal.BaseCommit == nil` when creating coding-pair children, fetch integration HEAD and set it. Subsequent transitions (more coding children, or fix children from integration) don't overwrite it.

Integration HEAD is fetched before the `blackboard.Modify` callback (git operation) and set inside it (state mutation), following the existing pattern for operations that combine git and state changes.

### D4: CodingComplete trigger detects integration need via goal.base_commit and role-pair absence

The wake trigger `WakeTriggerCodingComplete` fires when:
1. All planned tasks are terminal (`AllPlannedTasksTerminalWith`)
2. `goal.base_commit` is set (coding happened — if no coding tasks ran, no integration needed)
3. No task with `role_pair == "integration-pair"` exists in the sprint scope

This is inserted in the wake detection between PLANNING_COMPLETE and SPRINT_COMPLETE. If conditions (2) and (3) aren't both true, the existing SPRINT_COMPLETE handling proceeds.

### D5: BLOCKED escalation uses existing mechanisms

The spec states: "If a fix turns out to require non-trivial work, the fixer marks BLOCKED → orchestrator can promote it to a full coding sub-pipeline task with planning." This is already supported:
- Coder marks fix task BLOCKED via `liza_mark_blocked`
- Orchestrator wakes on `WakeTriggerBlocked`
- Orchestrator's BLOCKED_TASKS instructions tell it to assess and take action (supersede, replace with planning task)
No new code needed. The CodingComplete wake template documents this escalation path.

### D6: Clean task worktree cleanup via pipeline-aware DeleteWorktree

Tasks at `INTEGRATION_ANALYSIS_CLEAN` have worktrees from `ClaimTask` but skip merge (not approved-for-merge). `DeleteWorktree` currently only accepts BLOCKED, ABANDONED, SUPERSEDED, MERGED. Extending it to accept pipeline-defined clean terminal states requires loading the resolver and checking `CleanStatus()`. A new `handleCleanTaskCleanup` function in the reviewer PreWork scans for tasks at clean states with worktrees and calls `DeleteWorktree`.

### D7: Sprint completion recognizes CLEAN tasks

CP1 Task 3 already adds clean states to `SprintTerminalStates()`. CLEAN tasks are sprint-terminal. The orchestrator's SPRINT_COMPLETE wake template currently lists "MERGED, ABANDONED, or SUPERSEDED" as expected terminal states. This needs updating to include clean terminal states (or use generic "terminal state" language).

```go
func (r *Resolver) availableTransitionsByTrigger(trigger string, status models.TaskStatus, transitionsExecuted map[string]bool) []string {
    var available []string
    for _, sp := range r.config.Pipeline.SubPipelines {
        for _, t := range sp.Transitions {
            if t.Trigger != trigger {
                continue
            }
            if transitionsExecuted[t.Name] {
                continue
            }
            fromPair, fromPhase, err := parseRef(t.From)
            if err != nil {
                continue
            }
            if r.resolvePhase(fromPair, fromPhase) == status {
                available = append(available, t.Name)
            }
        }
    }
    // Also check pipeline-transitions
    for _, t := range r.config.Pipeline.PipelineTransitions {
        if t.Trigger != trigger {
            continue
        }
        if transitionsExecuted[t.Name] {
            continue
        }
        if r.resolve3PartPhase(t.From) == status {
            available = append(available, t.Name)
        }
    }
    slices.Sort(available)
    return available
}
```

```go
    if resolverErr != nil {
        return nil // no pipeline, skip
    }

    for _, task := range state.Tasks {
        if task.Worktree == nil || task.RolePair == "" {
            continue
        }
        cleanStatus, err := resolver.CleanStatus(task.RolePair)
        if err != nil || task.Status != cleanStatus {
            continue
        }
        result, err := ops.DeleteWorktree(projectRoot, task.ID)
        if err != nil {
            GetLogger().Warn("Failed to cleanup clean task worktree", "task_id", task.ID, "error", err)
            continue
        }
        if result.Existed {
            GetLogger().Info("Cleaned up worktree for clean-terminal task", "task_id", task.ID)
        }
    }
    return nil
}
```

## End-to-End Lifecycle

With all tasks implemented, the integration lifecycle proceeds as follows:

### Happy Path (findings exist)

1. **Coding tasks complete**: All coding-pair tasks reach CODE_APPROVED → supervisor merges → MERGED
2. **Orchestrator wakes**: `WakeTriggerCodingComplete` (all tasks terminal, base_commit set, no integration-pair task)
3. **Orchestrator spawns integration task**: `liza_add_tasks` with type=integration, role_pair=integration-pair
4. **Integration analyst claims task**: Gets worktree from integration HEAD, scans branch diff from goal.base_commit
5. **Analyst produces fix-task definitions**: `liza_set_task_output` with fix entries → `liza_submit_for_review` (zero-diff)
6. **Integration reviewer validates**: Approves → `SubmitVerdict` → INTEGRATION_ANALYSIS_APPROVED
7. **Supervisor merges**: No-op merge (analyst has no code changes) → MERGED
8. **Auto transition fires**: Reviewer PreWork calls `handleAvailableTransitions` → `ExecuteAvailableTransitions` processes `integration-to-fix` auto transition → per-subtask creates fix tasks (coding-pair)
9. **Fix tasks implemented**: Coders claim fix tasks → implement → submit → code-reviewer approves → MERGED
10. **Sprint complete**: All planned tasks terminal → orchestrator wakes on `WakeTriggerSprintComplete` → checkpoint

### Clean Scan Path (no findings)

1. Steps 1-4 same as above
2. **Analyst finds no issues**: Submits with empty output[] (or does not call set_task_output)
3. **Integration reviewer validates clean scan**: Approves → `SubmitVerdict` routes to INTEGRATION_ANALYSIS_CLEAN (CP1 Task 4)
4. **Worktree cleanup**: Reviewer PreWork calls `handleCleanTaskCleanup` → `DeleteWorktree` removes worktree
5. **Sprint complete**: CLEAN is sprint-terminal (CP1 Task 3) → orchestrator wakes on `WakeTriggerSprintComplete` → checkpoint

### BLOCKED Escalation Path

1. Fix task coder finds the fix requires non-trivial work
2. Coder calls `liza_mark_blocked` with reason
3. Orchestrator wakes on `WakeTriggerBlocked` → assesses the blocked task
4. Orchestrator supersedes the fix task → creates a full code-planning task (coding-subpipeline) for proper planning
5. Code-planning → coding → review → merge cycle proceeds normally
6. Sprint completion after all tasks terminal
