**Test cases (12 total):**

1. **`TestAwaitVerdict_Validation`** ŌĆö Table-driven precondition tests:
   - Empty task ID ŌåÆ error
   - Empty agent ID ŌåÆ error
   - Task not found ŌåÆ `NotFoundError`
   - Task status not submitted/reviewing ŌåÆ `PreconditionError`
   - Agent is not the last submitter ŌåÆ `PreconditionError`

2. **`TestAwaitVerdict_OwnershipAcquired`** ŌĆö On entry, verify:
   - Agent status set to `WAITING`
   - Agent `CurrentTask` set to task ID
   - State reads confirm these fields atomically

3. **`TestAwaitVerdict_BudgetExhausted`** ŌĆö When iteration/review-cycle limits are at capacity:
   - Returns `ErrBudgetExhausted`
   - Ownership released (agent `CurrentTask` is nil)
   - Setup: set task `Iteration` and `ReviewCycles` to max values

4. **`TestAwaitVerdict_Approved`** ŌĆö Task transitions to approved status:
   - Returns `Verdict: "APPROVED"`
   - `ReviewerAgent` populated from history
   - Ownership released
   - Setup: write state with submitted task, then in goroutine modify state to approved after short delay

5. **`TestAwaitVerdict_Rejected_SameAttempt`** ŌĆö Rejected within iteration budget:
   - Returns `Verdict: "REJECTED"`
   - `Reason` contains rejection reason
   - `Iteration` reflects new iteration number
   - `Guidance` non-empty (contains rejection guidance)
   - Task auto-reclaimed (assigned_to set to agent, status back to implementing)
   - Setup: modify state to rejected after short delay; iteration budget has room

6. **`TestAwaitVerdict_Rejected_NewAttempt`** ŌĆö Rejected but triggers new attempt (iteration limit reached):
   - Returns `Verdict: "NEW_ATTEMPT"`
   - Ownership released
   - Setup: set iteration to max-1 so next rejection triggers new attempt

7. **`TestAwaitVerdict_Rejected_Blocked`** ŌĆö Rejected and `ClaimTask` returns blocked:
   - Returns `Verdict: "TERMINAL"`
   - Ownership released
   - Setup: set both iteration and review cycles to max

8. **`TestAwaitVerdict_Blocked`** ŌĆö Task transitions to BLOCKED:
   - Returns `Verdict: "TERMINAL"` with reason "task entered terminal status: BLOCKED"
   - Ownership released

9. **`TestAwaitVerdict_PartiallyApproved`** ŌĆö Task transitions to partially_approved (quorum not met):
   - AwaitVerdict keeps waiting (does not return immediately)
   - After further transition to approved, returns `Verdict: "APPROVED"`
   - Setup: use pipeline config with quorum > 1; transition to partially_approved then approved with delays

10. **`TestAwaitVerdict_Timeout`** ŌĆö Deadline reached:
    - Returns `Verdict: "TIMEOUT"`
    - Ownership released
    - Setup: use very short timeout (e.g., 100ms), task stays in submitted status

11. **`TestAwaitVerdict_Aborted`** ŌĆö ABORT signal detected:
    - Returns `Verdict: "ABORTED"`
    - Ownership released
    - Setup: set `state.Config.Mode = SystemModeStopped` after short delay

12. **`TestAwaitVerdict_RaceGuard`** ŌĆö Another agent cannot claim task while awaiting agent holds ownership:
    - Start AwaitVerdict in goroutine (blocks)
    - Attempt `ClaimTask` with different agent ID ŌåÆ should fail (task is not in REJECTED status yet)
    - Transition task to rejected
    - AwaitVerdict returns with auto-reclaim (original agent still owns)
    - Verify second agent never acquired ownership

**Async state modification pattern:** Tests that need the state to change while `AwaitVerdict` blocks use a goroutine with `time.Sleep` to modify state via `bb.Modify()` after a short delay (~50-100ms). This is the standard approach for testing blocking functions.

**done_when:** `internal/ops/await_verdict_test.go` exists with 12 test functions covering: precondition validation (wrong status, wrong agent, empty IDs, missing task), ownership acquisition verification, budget exhaustion with ownership release, all 6 verdict paths (APPROVED, REJECTED same-attempt with auto-reclaim, NEW_ATTEMPT, TERMINAL, TIMEOUT, ABORTED), partially-approved wait-through, and race guard; `go test ./internal/ops/... -run TestAwaitVerdict` passes.

## Task 2: Integration test for await-verdict E2E rejection flow in `internal/integration/await_verdict_test.go`

**Intent:** Create an end-to-end test that simulates the full submit ŌåÆ await ŌåÆ rejected ŌåÆ fix ŌåÆ resubmit ŌåÆ approved ŌåÆ merge flow.

**Design:**

New file: `internal/integration/await_verdict_test.go` in `package integration`.

**Test: `TestAwaitVerdict_RejectionFlow`**

Follows patterns from `e2e_workflow_test.go` (setup, agent registration, claim/submit) and `sprint_and_merge_test.go` (rejection handling, merge verification).

```go
func TestAwaitVerdict_RejectionFlow(t *testing.T) {
    if testing.Short() { t.Skip("skipping integration test") }

    // --- Setup ---
    projectDir, cleanup := setupTestProject(t)
    defer cleanup()

    bb, statePath, _ := setupIntegrationTest(t, projectDir, []string{"task-1"})

    coderID := "coder-1"
    reviewerID := "code-reviewer-1"
    testhelpers.RegisterTestAgent(t, bb, coderID, "coder")
    testhelpers.RegisterTestAgent(t, bb, reviewerID, "code-reviewer")

    // --- Phase 1: Coder claims, implements, submits ---
    commands.ClaimTaskCommand(projectDir, "task-1", coderID)
    // Make changes in worktree, commit
    worktreePath := filepath.Join(projectDir, ".worktrees", "task-1")
    // ... git add/commit in worktree ...
    ops.WriteCheckpoint(projectDir, &ops.CheckpointInput{...})
    commitSHA := getHEAD(t, worktreePath)
    commands.SubmitForReviewCommand(projectDir, "task-1", commitSHA, coderID)

    // --- Phase 2: Coder calls AwaitVerdict (blocks in goroutine) ---
    var awaitResult *ops.AwaitVerdictResult
    var awaitErr error
    done := make(chan struct{})
    go func() {
        defer close(done)
        awaitResult, awaitErr = ops.AwaitVerdict(
            context.Background(), projectDir, "task-1", coderID, 30*time.Second)
    }()

    // --- Phase 3: Reviewer rejects ---
    time.Sleep(200 * time.Millisecond) // let AwaitVerdict start watching
    testhelpers.TransitionToReviewing(t, bb, "task-1", reviewerID)
    commands.SubmitVerdictCommand(projectDir, "task-1", "REJECTED", "Missing error handling", reviewerID, "")

    // --- Phase 4: Verify rejection result ---
    <-done
    if awaitErr != nil { t.Fatalf("AwaitVerdict error: %v", awaitErr) }
    if awaitResult.Verdict != ops.VerdictRejected { t.Fatalf("Verdict = %q, want REJECTED", awaitResult.Verdict) }
    if awaitResult.Reason == "" { t.Error("Expected non-empty rejection reason") }
    // Verify task auto-reclaimed
    state, _ := bb.Read()
    task := state.FindTask("task-1")
    if task.Status != models.TaskStatusImplementing { t.Errorf("Task status = %s, want IMPLEMENTING", task.Status) }
    if task.Iteration != 2 { t.Errorf("Iteration = %d, want 2", task.Iteration) }

    // --- Phase 5: Coder fixes, resubmits ---
    // ... make fix commit in worktree ...
    newSHA := getHEAD(t, worktreePath)
    commands.SubmitForReviewCommand(projectDir, "task-1", newSHA, coderID)

    // --- Phase 6: Reviewer approves ---
    testhelpers.TransitionToReviewing(t, bb, "task-1", reviewerID)
    commands.SubmitVerdictCommand(projectDir, "task-1", "APPROVED", "", reviewerID, "")

    // --- Phase 7: Merge ---
    commands.WtMergeCommand(projectDir, "task-1", reviewerID)
    state, _ = bb.Read()
    task = state.FindTask("task-1")
    if task.Status != models.TaskStatusMerged { t.Errorf("Task status = %s, want MERGED", task.Status) }
}
```

**Key design notes:**
- Uses real `ops.AwaitVerdict` call (not mocked) running in a goroutine since it blocks.
- Reviewer actions happen in the main goroutine after a short delay to let the watcher start.
- Verifies the full cycle: submit ŌåÆ await ŌåÆ reject ŌåÆ auto-reclaim ŌåÆ fix ŌåÆ resubmit ŌåÆ approve ŌåÆ merge.
- The second submit-approve cycle is a normal flow (no second AwaitVerdict call needed for the test ŌĆö the spec focuses on the rejection simulation).

**Scope:** `internal/integration/await_verdict_test.go` (new file).

**done_when:** `internal/integration/await_verdict_test.go` exists with `TestAwaitVerdict_RejectionFlow`; the test sets up project + task + coder + reviewer, coder submits and calls `AwaitVerdict` in goroutine, reviewer rejects, test verifies `Verdict == "REJECTED"` with reason and auto-reclaim (iteration incremented), coder resubmits, reviewer approves, merge succeeds; `go test ./internal/integration/ -run TestAwaitVerdict -count=1` passes.

**spec_ref:** Spec step 8 "internal/integration/await_verdict_test.go ŌĆö E2E rejection simulation" ŌĆö all bullet points

## Task 7: Contract and documentation updates for await-verdict

**Intent:** Update the multi-agent mode contract and user-facing documentation to describe the submit ŌåÆ await ŌåÆ handle verdict workflow.

**Design:**

### 7a. `contracts/MULTI_AGENT_MODE.md`

In the **Iteration Protocol** section (after "Review Exhaustion" subsection), add a new subsection:

```markdown
**Await Verdict (recommended post-submit flow):**
After `liza_submit_for_review`, doer agents should call `liza_await_verdict` to block until the review verdict arrives:
- **REJECTED**: Task is auto-reclaimed ŌĆö read rejection reason, fix, resubmit
- **APPROVED**: Exit normally ŌĆö task moves to merge
- **NEW_ATTEMPT**: Exit normally ŌĆö fresh doer spawns for new attempt
- **TIMEOUT/ABORTED**: Exit normally
- **Budget error**: Exit normally ŌĆö iteration limit reached

This eliminates cold-restart overhead between review cycles. The blocking call is budget-aware: it refuses if the iteration limit would be exceeded on rejection.
```

### 7b. `docs/USAGE_MULTI_AGENTS.md`

Add a subsection documenting the submit ŌåÆ await ŌåÆ handle verdict workflow. Place it in the section describing how agents interact during sprints. Content:

```markdown
### Submit ŌåÆ Await ŌåÆ Handle Verdict

Doer agents (coders, planners, writers) use a blocking workflow for review cycles:

1. `liza_submit_for_review` ŌĆö submit completed work
2. `liza_await_verdict` ŌĆö block until reviewer issues verdict
3. Handle verdict:
   - **REJECTED**: Fix issues, resubmit (session stays alive ŌĆö no cold restart)
   - **APPROVED/NEW_ATTEMPT/TIMEOUT/ABORTED**: Exit normally

This reduces per-rejection overhead from ~47s (cold restart) to near-zero.
```

### 7c. `docs/CONFIGURATION.md`

Add `liza_await_verdict` to the MCP tools listing section. Include tool name, description, required parameters (`task_id`, `agent_id`), optional parameters (`timeout_seconds`), and default timeout (1500s).

**Scope:** `contracts/MULTI_AGENT_MODE.md`, `docs/USAGE_MULTI_AGENTS.md`, `docs/CONFIGURATION.md`.

**done_when:** `MULTI_AGENT_MODE.md` Iteration Protocol section includes await-verdict subsection with all verdict types and budget-aware behavior; `USAGE_MULTI_AGENTS.md` documents the submit ŌåÆ await ŌåÆ handle verdict workflow; `CONFIGURATION.md` lists `liza_await_verdict` with parameters, defaults, and description.
