## Decomposition rationale (single atomic task)

The execution work is one intent — "produce a populated observation table + artifact directory satisfying §5.9". The eight procedural steps in my assigned task description are strictly sequential and interlocked:

- `setup → liza init → start supervisors → wait → capture → sanitize → commit artifacts → populate table`.

Splitting would fracture the atomic "run + evidence" deliverable: a separate "run" task without a "commit artifacts" task would leave the repo without evidence pointers; a separate "populate table" task would have nothing to reference. The architecture plan §8 table also lists this as one output entry (index 3). Following that decomposition preserves the falsifiable gate: reviewers can check the single task's done_when against the committed worktree diff + artifact directory.

TDD colocation: this deliverable is operational/evidence — no user-visible software behavior change. No unit tests are required. The "falsifiability check" specified in §5.9 (grep-based placeholder-absence check) is the review contract; it is exercised at review time, not embedded as a test in the task.

## Notes for the coder

- **Run against unfixed prompts.** Do not `git checkout` any branch that pre-merges sibling fixes (architecture-1, architecture-2, architecture-3 §0/§1, architecture-4). The synthetic greenfield projects' `.liza/` are initialized fresh under `$REPRO_ROOT` and are independent of the liza repo checkout, but the `liza` binary on PATH must be the currently-installed build (not a fix branch). If the operator has a locally-built `liza` from a fix branch, rebuild from this worktree's HEAD before starting supervisors.
- **BASH CONSTRAINTS apply to the coder's own shell, not the script body.** §5.4 of the architecture plan spells this out: script internals may use `$(...)`, `&&`, etc.; the coder's direct invocations of the script must remain simple (`bash scripts/repro/precommit-bootstrap-greenfield.sh "$REPRO_ROOT" <cycle> <phase>`).
- **Iteration budget.** Up to 5 cycles per project. If after 5 cycles the observation table still has rows where the coder cannot form a defensible `yes`/`no`/`partial` verdict, mark BLOCKED — do not record `…` or leave a cell blank, and do not fabricate observations.
- **Sanitization is non-negotiable.** The architecture plan §5.6 requires user-home-prefix replacement with `~` and a credential-file sweep. Evidence pointers in the goal-spec table must reference the sanitized copies. Committing unsanitized absolute paths or credential-shaped files is a Tier 0 (T0.5) violation.
- **Cross-architect ordering (§5.7) is NOT this coder's responsibility.** The orchestrator picks option (a) or (b); the execution coder's contract is simply to populate the table with evidence. The reviewer checks populated-ness, not downstream integration ordering.
- **Divert path is allowed.** Goal-spec line 19 explicitly permits divert ("If reproduction shows the actual failure mode is different from any of the above, the ratified design is revisited"). Divert requires a follow-up issue/task ID in the decision paragraph AND a `TECH_DEBT.md` entry. Both confirm and divert are valid terminations of this task.
