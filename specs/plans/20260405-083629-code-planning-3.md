## Context

The integration analyst and reviewer need different context than other doer/reviewer roles. Instead of a single task's diff, they work with the full branch diff from `goal.base_commit` to integration HEAD â€” capturing all changes made for a goal. This requires:

1. A new `base_commit` field on the Goal struct to store the diff base SHA
2. New RoleContextData fields for the branch context
3. A new block template (`branch-integration-context`) that renders the diff command and completed task list
4. Integration-reviewer review instructions (the reviewer validates task definitions, not code)
5. Verification that zero-diff submission works for the analyst (who produces findings, not code)

## Design Decisions

### D1: GoalBaseCommit as typed field, not Extra map

The Goal struct has `Extra map[string]any` for arbitrary fields, but `base_commit` is a first-class integration concept used in templates and prompt assembly. A typed `BaseCommit *string` field enables compile-time safety and clear documentation. Optional (pointer) because it's only set when the orchestrator creates the first coding task.

### D2: CompletedTaskSummary as dedicated type in prompts package

The branch-integration-context template needs a list of completed tasks with specific fields (ID, Description, DoneWhen, SpecRef). A dedicated `CompletedTaskSummary` type in `internal/prompts/role_context.go` follows the `SiblingTaskSummary` pattern already used for collective plan scoping.

### D3: Population logic scoped to integration roles via role name check

`buildTaskRoleContextData` in `internal/agent/prompt.go` already has role-specific branches: `config.Role == "coder"` for coder fields, `roleType == "reviewer"` for reviewer fields. Integration context (GoalBaseCommit, CompletedTasks) is populated when the role is `integration-analyst` or `integration-reviewer`, following the same pattern. Both roles share the same data because they both need the full branch context.

### D4: Template renders diff command, not diff content

Per spec: "The section provides the command (`git diff goal.base_commit..HEAD`) and the completed task list; the agent executes the diff in its worktree." The template renders the command string for the agent to execute, avoiding bloating the blackboard with potentially massive diff output.

### D5: Integration-reviewer review instructions as new role branch

The `review_instructions.tmpl` template uses role-conditional blocks (`eq .Role "code-reviewer"`, etc.). The integration reviewer needs distinct instructions focused on validating fix-task definitions (output[]) and enriching with systemic analysis, rather than reviewing code changes. Adding an `eq .Role "integration-reviewer"` branch follows the established pattern.

### D6: Zero-diff submission works with existing machinery

The analyst submits via `submit_for_review` without having committed code. Analysis of the flow:
1. **commit SHA validation**: Agent provides current worktree HEAD (the branch point from integration). Matches the actual HEAD. OK.
2. **Branch validation**: Worktree is on `liza/<task-id>` branch (set by ClaimTask). OK.
3. **TDD gate** (`submit_review.go:122-132`): `task.EffectiveType() == TaskTypeCoding` is false for `TaskTypeIntegration`. Gate skipped. (Requires CP1 Task 1.)
4. **Rebase**: `git rebase FETCH_HEAD` when the analyst's branch has no new commits either fast-forwards (if integration advanced) or is a no-op. Both succeed.
5. **BaseCommit/ReviewCommit**: Both set to integration HEAD (same or close). The reviewer doesn't use these for integration review â€” they use `branch-integration-context` for the meaningful diff.

No code changes needed. A test validates this path.

    for _, t := range state.Tasks {

            tasks = append(tasks, prompts.CompletedTaskSummary{
                ID:          t.ID,
                Description: truncateDescription(t.Description, 200),
                DoneWhen:    t.DoneWhen,
                SpecRef:     t.SpecRef,
            })
        }
    }
    return tasks
}

REVIEW SCOPE:

Enrich with systemic analysis: interaction effects across fixes, architectural drift, emergent risks.

   a. Interaction effects: could fixing A break the fix for B?
   b. Architectural drift: do the fixes collectively move the codebase away from its design intent?
   c. Emergent risks: are there cross-cutting concerns invisible at the individual finding level?
   d. Missing findings: are there integration issues the analyst missed?
5. If enrichment produces additional fix-tasks, include them in the rejection with specific task definitions.
{{- if ne .PriorRejection ""}}

PRIOR FEEDBACK REVIEW (MANDATORY for iteration 2+):
Before submitting verdict, compare this iteration against prior rejection:
- Which prior issues are now RESOLVED?
- Which prior issues are STILL PRESENT?
- Which prior issues are PARTIALLY ADDRESSED?
Include this assessment in your rejection reason if rejecting.
{{- end}}

POST-VERDICT (MANDATORY for REJECTED):
After submitting a REJECTED verdict, MUST call liza_await_resubmission immediately.
Do NOT exit. It blocks ~100s per call, returning POLL with remaining timeout_seconds.
- RESUBMITTED: re-review the new changes
- POLL: call await_resubmission again with the timeout_seconds from the response
- TERMINAL/TIMEOUT/ABORTED: exit normally
