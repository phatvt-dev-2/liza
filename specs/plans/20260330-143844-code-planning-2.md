## Task 1: Add `handleAwaitVerdict` MCP handler to `handlers_mutation.go`

**Intent:** Implement the MCP handler that extracts parameters, calls `ops.AwaitVerdict`, and formats the result for the agent.

**Design:**

```go
// handleAwaitVerdict implements the liza_await_verdict tool.
// Blocks until a review verdict arrives for a submitted task.
func (s *Server) handleAwaitVerdict(params map[string]any) (any, error) {
    taskID, err := requireString(params, "task_id")
    if err != nil {
        return nil, err
    }

    agentID, err := requireString(params, "agent_id")
    if err != nil {
        return nil, err
    }

    // Optional timeout with default 1500s (25 min, within Claude Code's 30 min MCP_TIMEOUT)
    timeoutSeconds := 1500
    if v, ok := params["timeout_seconds"].(float64); ok && v > 0 {
        timeoutSeconds = int(v)
    }
    timeout := time.Duration(timeoutSeconds) * time.Second

    result, err := ops.AwaitVerdict(context.Background(), s.projectRoot, taskID, agentID, timeout)
    if err != nil {
        if errors.Is(err, ops.ErrBudgetExhausted) {
            return textResult("Budget exhausted: iteration or review-cycle limit reached. Exit normally.")
        }
        return nil, fmt.Errorf("await verdict failed: %w", err)
    }

    msg := fmt.Sprintf("Verdict: %s\nStatus: %s\nReason: %s\nReviewer: %s",
        result.Verdict, result.TaskStatus, result.Reason, result.ReviewerAgent)
    if result.Guidance != "" {
        msg += fmt.Sprintf("\n\n%s", result.Guidance)
    }

    return textResult(msg)
}
```

**Key design decisions:**

1. **`context.Background()`**: The MCP handler has no parent context to propagate. The `AwaitVerdict` function manages its own deadline via the `timeout` parameter. If a context-aware MCP server is added later, this can be changed to pass the request context.

2. **Timeout extraction**: JSON numbers arrive as `float64` in Go's `map[string]any`. The pattern `params["timeout_seconds"].(float64)` matches the existing `params["priority"].(float64)` pattern in `handlers_helpers.go:200`.

3. **`ErrBudgetExhausted` handling**: Returns a success response (via `textResult`) rather than an error, because budget exhaustion is an expected business outcome, not an infrastructure failure. The agent should exit normally.

4. **Guidance field**: CP1's `AwaitVerdictResult.Guidance` carries inline rejection guidance equivalent to `prior_rejection.tmpl`. The handler appends it to the response so the agent gets rejection instructions without a second state read.

5. **Imports needed**: `context`, `errors`, `fmt`, `time`, plus existing `ops` and `paths` imports.

**Scope:** `internal/mcp/handlers_mutation.go` (add new handler method).

**done_when:** `handleAwaitVerdict` method exists in `handlers_mutation.go`; it extracts `task_id` (required), `agent_id` (required), `timeout_seconds` (optional, default 1500); calls `ops.AwaitVerdict`; returns formatted `textResult` on success and handles `ErrBudgetExhausted` distinctly; `go build ./internal/mcp/...` succeeds.

**spec_ref:** Spec step 2 "internal/mcp/handlers_mutation.go — MCP handler"

## Task 2: Register `liza_await_verdict` tool in `server_registration.go`

**Intent:** Add the tool definition to the mutation tools slice so the MCP server exposes `liza_await_verdict` with proper schema and role authorization.

**Design:**

Add to `registerMutationTools()` after the `liza_submit_for_review` entry (logical grouping — submit then await):

```go
// liza_await_verdict tool
{
    tool: protocol.Tool{
        Name:        "liza_await_verdict",
        Description: "Block until review verdict arrives for a submitted task. Budget-aware: refuses if iteration limit would be exceeded on rejection. Call after liza_submit_for_review.",
        InputSchema: protocol.InputSchema{
            Type: "object",
            Properties: map[string]protocol.Property{
                "task_id":         {Type: "string", Description: "Task ID to await verdict for"},
                "agent_id":        {Type: "string", Description: "Agent ID (for authorization)"},
                "timeout_seconds": {Type: "integer", Description: "Max wait time in seconds (default: 1500, must be < MCP_TIMEOUT)"},
            },
            Required: []string{"task_id", "agent_id"},
        },
    },
    handler:     s.handleAwaitVerdict,
    roleChecker: operationChecker(s.resolver, s.pipelineLoadErr, "liza_await_verdict"),
},
```

**Key design decisions:**

1. **Placement in `registerMutationTools()`**: Groups with `liza_submit_for_review` and `liza_handoff` — the doer submission flow tools. The spec explicitly says "mutation tools."

2. **`operationChecker`**: Uses `operationChecker` (same as `liza_submit_for_review`) rather than `typeChecker`. This delegates authorization to the pipeline's `allowed-operations` list, which CP1 already adds `await-verdict` to for all doer roles.

3. **`timeout_seconds` type "integer"**: JSON Schema "integer" (not "number") — timeout must be a whole number of seconds.

**Scope:** `internal/mcp/server_registration.go` (add toolDef entry to `registerMutationTools()`).

**done_when:** `liza_await_verdict` appears in `registerMutationTools()` with tool name, description, input schema (task_id required string, agent_id required string, timeout_seconds optional integer), handler `s.handleAwaitVerdict`, and roleChecker `operationChecker`; `go build ./internal/mcp/...` succeeds.

**spec_ref:** Spec step 3 "internal/mcp/server_registration.go — Register tool"

**depends_on:** Task 1 (registration references `s.handleAwaitVerdict` defined in Task 1)
