## Task 1: Add Attempt field, EffectiveAttempt helper, remove dead Attempted field, initialize on first claim

**Changes:**

1. **`internal/models/task.go`** — Remove `Attempted []string` field (line 192). Add `Attempt int` field with tag `yaml:"attempt,omitempty"` in the Task struct (after `FailedBy` at line 196, maintaining logical grouping with lifecycle fields).

2. **`internal/models/task.go`** — Add `EffectiveAttempt() int` method on `*Task`:
   ```go
   func (t *Task) EffectiveAttempt() int {
       if t.Attempt < 1 {
           return 1
       }
       return t.Attempt
   }
   ```
   Returns `max(Attempt, 1)` for backward compat with existing tasks where Attempt is 0 (unset).

3. **`internal/ops/claim_task_strategy.go`** — In `freshClaimStrategy.mutateTask()` (line 83), add `task.Attempt = 1` alongside the existing Worktree/BaseCommit assignments. This ensures clean lifecycle: 0 (created) → 1 (first claim) → 2 (attempt transition).

4. **`internal/models/task_test.go`** — Tests:
   - `TestEffectiveAttempt_ZeroReturnsOne`: Task with Attempt=0 returns 1
   - `TestEffectiveAttempt_ReturnsActualValue`: Task with Attempt=1 returns 1, Attempt=2 returns 2

**done_when:** (a) `Attempted` field no longer exists in `internal/models/task.go`. (b) `Attempt int` field with `yaml:"attempt,omitempty"` tag exists in the Task struct. (c) `EffectiveAttempt()` method exists and returns 1 when Attempt is 0, returns Attempt when >=1. (d) `freshClaimStrategy.mutateTask()` sets `task.Attempt = 1`. (e) `TestEffectiveAttempt_ZeroReturnsOne` and `TestEffectiveAttempt_ReturnsActualValue` in `internal/models/task_test.go` pass.

**scope:** `internal/models/task.go`, `internal/ops/claim_task_strategy.go`, `internal/models/task_test.go`.

## Task 2: Add sentinel claimability guard to IsClaimable and ClaimTask

**Files:**
- `internal/models/task.go` — sentinel check in IsClaimable()

- `internal/models/task_test.go` — IsClaimable sentinel test (append to file created in Task 1)

**Changes:**

1. **`internal/models/task.go`** — In `IsClaimable()` (line 311), add an early return before the `isClaimablePipeline` call:
   ```go
   if t.AssignedTo != nil && strings.HasPrefix(*t.AssignedTo, "$") {
       return false
   }
   ```
   This requires adding `"strings"` to the import block. Prevents sentinel-guarded tasks from appearing in candidate lists.

2. **`internal/ops/claim_task.go`** — Before strategy dispatch (~line 109, after pipeline status resolution and before the `switch task.Status` block at line 110): Add sentinel guard that rejects with PreconditionError when `AssignedTo` starts with `$`:
   ```go
   if task.AssignedTo != nil && strings.HasPrefix(*task.AssignedTo, "$") {
       return nil, &PreconditionError{Reason: fmt.Sprintf("task %s is in transition (assigned_to: %s)", taskID, *task.AssignedTo)}
   }
   ```
   Defense-in-depth: placed before strategy dispatch so that sentinel-guarded tasks are rejected before any strategy-specific validation or worktree operations can execute. The `strings` import is already present in claim_task.go. This matches the goal spec placement: "ClaimTask Phase 1 (before strategy dispatch)".

3. **`internal/models/task_test.go`** — Add test:
   - `TestIsClaimable_SentinelAssignedToReturnsFalse`: Task with `AssignedTo = "$transitioning"` and a claimable status (e.g., pipeline initial) returns false from IsClaimable.

4. **`internal/ops/claim_task_test.go`** — Add test:
   - `TestClaimTask_SentinelAssignedTo_Rejected`: Set up a REJECTED task with `AssignedTo = "$transitioning"`. ClaimTask returns PreconditionError containing "is in transition".

**done_when:** (a) `IsClaimable()` returns false when `AssignedTo` starts with `"$"`, verified by `TestIsClaimable_SentinelAssignedToReturnsFalse` in `internal/models/task_test.go`. (b) `ClaimTask` returns `PreconditionError` with "is in transition" when `AssignedTo` starts with `"$"`, verified by `TestClaimTask_SentinelAssignedTo_Rejected` in `internal/ops/claim_task_test.go`. (c) The sentinel guard in `ClaimTask` is placed before the strategy dispatch `switch task.Status` block, not after it.

**scope:** `internal/models/task.go`, `internal/ops/claim_task.go`, `internal/models/task_test.go`, `internal/ops/claim_task_test.go`.

## Task 4: Implement TransitionToNewAttempt 3-phase operation

**Files:**
- `internal/ops/transition_attempt.go` (new) — operation implementation
- `internal/ops/transition_attempt_test.go` (new) — comprehensive tests

**Changes:**

1. **`internal/ops/transition_attempt.go`** (new file) — Implement `TransitionToNewAttempt`:

   ```go
   func TransitionToNewAttempt(projectRoot, taskID, reason string) (*TransitionAttemptResult, error)
   ```

   **Result struct:**
   ```go
   type TransitionAttemptResult struct {
       TaskID          string
       NewAttempt      int
       WorktreeDeleted bool
       InitialStatus   models.TaskStatus
   }
   ```

   **Phase 1 — bb.Modify (mark attempt boundary, block claims via sentinel):**
   - Load pipeline bundle via `loadPipelineBundle(projectRoot)` (reuses existing helper from `pipeline_ops.go`)
   - Read task, verify `task.EffectiveAttempt() == 1` — return PreconditionError if not
   - Resolve initial status from pipeline resolver via `pr.InitialStatus(task.RolePair)`
   - Capture worktree path for Phase 2 (from `task.Worktree`)
   - Mutations inside `bb.Modify`:
     - `task.Attempt = 2`
     - `task.Iteration = 0` (next claim increments to 1)
     - `task.ReviewCyclesCurrent = 0`
     - `task.LeaseExpires = nil`
     - `task.AssignedTo = &sentinel` where `sentinel = "$transitioning"` — blocks concurrent claims via guards added in Task 2
     - Append history: `TaskEventNewAttempt` with reason string
     - Release previous agent via `state.ReleaseAgent()` if AssignedTo was set
     - **Preserved fields (Phase 1 invariants):** current `Status`, `Worktree`, `BaseCommit`, and `RejectionReason` are NOT modified in Phase 1. `RejectionReason` must stay set until Phase 3 to maintain the REJECTED field requirement from INVARIANTS.md §3.1 (REJECTED status requires `rejection_reason`). `Worktree`/`BaseCommit` are kept because Phase 2 needs the worktree path and clearing them before deletion would lose the reference.

   **Phase 2 — Git ops (outside lock):**
   - Instantiate `git.New(projectRoot)` via `paths.New(projectRoot).ProjectRoot()`
   - If worktree path was captured in Phase 1:
     - Call `gitWrapper.RemoveWorktree(taskID)` to delete worktree and branch
     - Log warning on failure, do NOT return error (best-effort)
   - Track whether deletion succeeded for result

   **Phase 3 — bb.Modify (release sentinel, make task claimable):**
   - Re-read task
   - Re-check: `task.AssignedTo != nil && *task.AssignedTo == "$transitioning"` — if sentinel was replaced by concurrent operation, return error (abort, no corruption)
   - `task.AssignedTo = nil`
   - `task.RejectionReason = nil` (cleared here, not Phase 1 — keeps REJECTED state valid during transition per INVARIANTS.md §3.1)
   - `task.Worktree = nil`
   - `task.BaseCommit = nil`
   - `task.TransitionWith(initialStatus, transitions)` — transition to initial pipeline status
   - Note: `ReviewCyclesTotal` is NOT reset (audit trail)

2. **`internal/ops/transition_attempt_test.go`** (new file) — Tests:
   - `TestTransitionToNewAttempt_Success`: Attempt 1 REJECTED task transitions to attempt 2, initial status. Verifies: Attempt=2, Iteration=0, ReviewCyclesCurrent=0, AssignedTo=nil, Worktree=nil, BaseCommit=nil, RejectionReason=nil, history contains `new_attempt` event with reason, previous agent released, status is initial pipeline status.
   - `TestTransitionToNewAttempt_Attempt2Rejected`: Task with Attempt=2 returns PreconditionError (only attempt 1 can transition).
   - `TestTransitionToNewAttempt_Attempt0DefaultsTo1`: Task with Attempt=0 (legacy/unset) passes EffectiveAttempt()==1 precondition and transitions successfully.
   - `TestTransitionToNewAttempt_WorktreeDeletionFailure`: Phase 2 git failure is tolerated — Phase 3 still runs, Worktree/BaseCommit cleared in state. Test verifies non-fatal (warning logged, no error returned).
   - `TestTransitionToNewAttempt_SentinelReplacedByPhase3`: Between Phase 1 and Phase 3, a concurrent operation changes AssignedTo from "$transitioning" to something else. Phase 3 detects mismatch and returns error.
   - `TestTransitionToNewAttempt_ReviewCyclesTotalPreserved`: ReviewCyclesTotal is not reset (verify it retains its pre-transition value).
   - `TestTransitionToNewAttempt_AgentRelease`: Previous agent is released from state.Agents (status=idle, current_task=nil).
   - `TestTransitionToNewAttempt_SentinelSetDuringTransition`: After Phase 1 and before Phase 3, task.AssignedTo is "$transitioning" (verifiable by reading state between phases — or by checking Phase 1 output).
   - `TestTransitionToNewAttempt_Phase1PreservesFieldInvariants`: After Phase 1 (before Phase 3), assert: (i) `RejectionReason` is still set (not nil), (ii) `Status` is unchanged from pre-transition value, (iii) `Worktree` is still set (not nil), (iv) `BaseCommit` is still set (not nil), (v) `LeaseExpires` is nil (cleared in Phase 1). This test reads state between Phase 1 and Phase 3 to verify Phase 1 preservation guarantees.
   - `TestTransitionToNewAttempt_RejectionReasonOnlyClearedInPhase3`: Verify that a REJECTED task with `RejectionReason` set retains it through Phase 1, retains it during Phase 2, and only has it cleared in Phase 3. The final state after Phase 3 has `RejectionReason = nil`.

**done_when:** (a) `TransitionToNewAttempt` function exists in `internal/ops/transition_attempt.go` with the 3-phase pattern. (b) Phase 1 sets Attempt=2, resets Iteration/ReviewCyclesCurrent to 0, clears LeaseExpires, sets sentinel `$transitioning`, appends `new_attempt` history, releases agent, and preserves current Status, Worktree, BaseCommit, and RejectionReason. (c) Phase 2 deletes worktree best-effort. (d) Phase 3 re-checks sentinel, clears AssignedTo/RejectionReason/Worktree/BaseCommit, transitions to initial pipeline status. (e) PreconditionError returned when EffectiveAttempt()!=1. (f) All 10 tests in `internal/ops/transition_attempt_test.go` pass: Success, Attempt2Rejected, Attempt0DefaultsTo1, WorktreeDeletionFailure, SentinelReplacedByPhase3, ReviewCyclesTotalPreserved, AgentRelease, SentinelSetDuringTransition, Phase1PreservesFieldInvariants, RejectionReasonOnlyClearedInPhase3.

**scope:** `internal/ops/transition_attempt.go` (new), `internal/ops/transition_attempt_test.go` (new).

**spec_ref:** Goal spec Change 3 — "New operation — internal/ops/transition_attempt.go (new file)"

**depends_on:** Task 1 (uses EffectiveAttempt, Attempt field), Task 2 (relies on sentinel guard in IsClaimable/ClaimTask), Task 3 (uses TaskEventNewAttempt constant)

## Rejection Feedback Resolution

### Blocker: Task 4 under-specified on Phase 1 preservation invariants

**Problem:** Task 4's prior `done_when` and tests did not require that Phase 1 keeps `RejectionReason` set, preserves current status and `Worktree`/`BaseCommit`, or clears `LeaseExpires`. A coder could satisfy the task while violating INVARIANTS.md §3.1 (REJECTED requires `rejection_reason`).

**Resolution:**
1. Task 4 `done_when` item (b) now explicitly lists: "preserves current Status, Worktree, BaseCommit, and RejectionReason" as Phase 1 requirements, and "clears LeaseExpires" in Phase 1.
2. Added `TestTransitionToNewAttempt_Phase1PreservesFieldInvariants` — asserts Phase 1 keeps `RejectionReason`, `Status`, `Worktree`, `BaseCommit` unchanged and clears `LeaseExpires`.
3. Added `TestTransitionToNewAttempt_RejectionReasonOnlyClearedInPhase3` — asserts `RejectionReason` survives Phase 1 and Phase 2, cleared only in Phase 3.
4. Test count increased from 8 to 10.

### Concern: Sentinel guard placement in ClaimTask

**Problem:** Prior plan described the guard as "after strategy dispatch (line 119, after the switch block)" — ambiguous and weaker than the spec's "before strategy dispatch" placement.

**Resolution:** Task 2 Change 2 now explicitly states the guard is placed "before strategy dispatch (~line 109, after pipeline status resolution and before the `switch task.Status` block at line 110)". Task 2 `done_when` item (c) adds: "The sentinel guard in `ClaimTask` is placed before the strategy dispatch `switch task.Status` block, not after it." This matches the goal spec's explicit placement.
