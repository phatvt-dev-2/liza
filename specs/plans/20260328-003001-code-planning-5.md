## Analysis

### Watcher Error Recovery

**Current behavior (`update.go:147-149`):**
```go
case errMsg:
    // No user-visible action for now; future: show in status bar
    return m, nil
```

The `watchStateCmd` function (`commands.go:26-41`) blocks on `watcher.Events()` and `watcher.Errors()`. On error, it returns `errMsg{err}`. But the `errMsg` handler:
- Does NOT re-arm `watchStateCmd` (contrast: `stateChangedMsg` at line 22-27 re-arms it)
- Does NOT mark the watcher degraded (contrast: `watcherClosedMsg` at line 151-153 nils `m.watcher`)
- Does NOT surface any visible indication to the operator

After a single fsnotify error, the watcher subscription goroutine exits and is never restarted. The TUI silently falls back to 10s tick-only polling. The model still believes `m.watcher != nil`, but nobody is listening to its channels.

**Comparison of all three watcher message handlers:**

| Message | Re-arms `watchStateCmd`? | Updates model? | Visible feedback? |
|---------|--------------------------|----------------|-------------------|
| `stateChangedMsg` | Yes | No | No (expected flow) |
| `watcherClosedMsg` | No | `m.watcher = nil` | No |
| `errMsg` | No | No | No |

**Design:** On `errMsg` from a watcher error, re-subscribe to the watcher by re-arming `watchStateCmd`, matching the `stateChangedMsg` behavior. This is the correct recovery: fsnotify errors are typically transient (e.g., buffer overflow) and the watcher channel remains open. If the watcher is truly broken, subsequent reads will return `watcherClosedMsg` when the channel closes, triggering the existing degradation path.

Additionally, log the error to the activity feed as a warning so operators have visibility into watcher instability.

### TIME_ON_TASK Label

**Current behavior (`view.go:170-176`):**
```go
timeOnTaskVal := func(_ string, a models.Agent) string {
    if a.CurrentTask == nil {
        return "ŌĆö"
    }
    // Approximate using heartbeat as proxy (task creation time not directly available on Agent)
    return render.FormatDuration(time.Since(a.Heartbeat))
}
```

The column header says `TIME_ON_TASK` but the value is `time.Since(a.Heartbeat)`. Heartbeats renew every few seconds for healthy agents, so the displayed value stays near zero ŌĆö misleading operators into thinking tasks just started when they may have been running for hours.

The spec suggestion (tofix.md line 30) offers two options:
1. Remove the column until task-start time is available
2. Relabel it to what it actually shows

**Design:** Relabel the column from `TIME_ON_TASK` to `LAST_HEARTBEAT` and change the display format to match the existing `heartbeatVal` pattern (`duration + " ago"`). This accurately describes what is shown: recency of the last heartbeat signal. The column remains useful as a health indicator (high values = stale agent). Removing the column entirely would lose that signal.

Changes:
- `view.go:214`: Change header string from `"TIME_ON_TASK"` to `"LAST_HEARTBEAT"`
- `view.go:170-176`: Replace `timeOnTaskVal` body with the same logic as `heartbeatVal` (lines 178-183), using `a.Heartbeat` with `" ago"` suffix. This eliminates the misleading `CurrentTask` guard ŌĆö heartbeat recency is meaningful regardless of task assignment.

**Test impact:** The `countHeaderColumns` helper in `view_test.go:952` hardcodes the column names list including `"TIME_ON_TASK"`. After renaming, this helper will no longer find the old column name in headers, causing 4 existing column-count tests (ColumnTierMinimal, ColumnTierStandard, ColumnTierWide, ColumnTierFull) to report wrong counts. The helper must be updated to replace `"TIME_ON_TASK"` with `"LAST_HEARTBEAT"`. Additionally, `TestRenderAgentPanel_ColumnTierWide` at line 337 explicitly asserts `assertContains(t, header, "TIME_ON_TASK", ...)` which must be changed to assert `"LAST_HEARTBEAT"`.

### Task Panel Active-First Sorting

**Current behavior (`view.go:303-308`, `view.go:429-431`):**
```go
sort.SliceStable(tasks, func(i, j int) bool {
    return tasks[i].Created.Before(tasks[j].Created)
})
// ... later:
for i, t := range tasks {
    if i >= maxRows { break }
```

Comment at line 266 says "Terminal tasks (MERGED, ABANDONED, SUPERSEDED) shown dimmed at bottom", but the sort is purely by `Created` time. In a long sprint, old terminal (merged/abandoned) tasks appear first (oldest Created time) and consume the visible rows, pushing active work off-screen.

**Design:** Partition tasks into active and terminal groups before sorting. Within each group, sort by Created ascending (preserving existing order invariant). Render active tasks first, then terminal tasks. This ensures active work is always visible at the top.

Implementation approach:
1. After the existing copy + sort by Created (lines 304-308), partition into two slices: `active` (where `!t.Status.IsTerminal()`) and `terminal` (where `t.Status.IsTerminal()`)
2. Concatenate: `ordered := append(active, terminal...)`
3. Use `ordered` in the rendering loop instead of `tasks`

Both partitions preserve Created-ascending order within their group (since the input was already sorted by Created).

## Task 3: Partition active tasks before terminal tasks in task panel

### Intent
Sort the task panel so active (non-terminal) tasks always appear before terminal tasks (MERGED, ABANDONED, SUPERSEDED), preventing old terminal tasks from pushing active work off-screen during truncation.

### Changes

**`internal/tui/view.go` (lines 303-308):**

After the existing sort by Created, add a partition step:

```go
// Sort tasks by created timestamp (oldest first)
tasks := make([]models.Task, len(m.state.Tasks))
copy(tasks, m.state.Tasks)
sort.SliceStable(tasks, func(i, j int) bool {
    return tasks[i].Created.Before(tasks[j].Created)
})

// Partition: active tasks first, terminal tasks after.
// Both partitions preserve Created-ascending order.
var active, terminal []models.Task
for _, t := range tasks {
    if t.Status.IsTerminal() {
        terminal = append(terminal, t)
    } else {
        active = append(active, t)
    }
}
tasks = append(active, terminal...)
```

The rendering loop and truncation logic (`for i, t := range tasks { if i >= maxRows { break } }`) remain unchanged ŌĆö active tasks now naturally occupy the top rows.
