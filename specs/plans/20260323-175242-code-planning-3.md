## Task 3: Wire attempt-aware escalation into ClaimTask iteration check

**Files:**
- `internal/ops/claim_task.go` — iteration limit enforcement refactor
- `internal/ops/claim_task_test.go` — new escalation tests

**Changes:**

1. **`internal/ops/claim_task.go`** — Replace the direct iteration comparison at line 151 with `classifyLimitEscalation`. The current code:
   ```go
   if strategy.enforceIterationLimit() && task.Iteration >= maxCoderIterations {
       blockedIteration, blockedLimit, err := enforceRejectedIterationLimit(...)
       ...
   }
   ```
   Becomes:
   ```go
   if strategy.enforceIterationLimit() {
       reviewLimit := effectiveReviewCycleLimit(state.Config)
       escalation, shouldEscalate := classifyLimitEscalation(
           task.ReviewCyclesCurrent,
           reviewLimit,
           task.Iteration,
           maxCoderIterations,
           task.EffectiveAttempt(),
       )
       if shouldEscalate {
           switch escalation.action {
           case LimitActionBlocked:
               blockedIteration, blockedLimit, err := enforceRejectedIterationLimit(
                   bb, taskID, agentID, taskStatus, pipelineTransitions,
               )
               if err != nil {
                   return nil, fmt.Errorf("failed to enforce iteration limit: %w", err)
               }
               return nil, &PreconditionError{Reason: fmt.Sprintf(
                   "task %s reached max iterations (%d/%d) and was transitioned to BLOCKED",
                   taskID, blockedIteration, blockedLimit,
               )}
           case LimitActionNewAttempt:
               if _, err := TransitionToNewAttempt(projectRoot, taskID, escalation.reason); err != nil {
                   return nil, fmt.Errorf("failed to transition to new attempt: %w", err)
               }
               return nil, &PreconditionError{Reason: fmt.Sprintf(
                   "task %s reached limits at attempt 1 — transitioned to attempt 2, claimable on next supervisor loop",
                   taskID,
               )}
           }
       }
   }
   ```
   Note: `TransitionToNewAttempt` is called before any filesystem work (Phase 2 worktree handling), so no cleanup is needed. The PreconditionError aborts the claim; the supervisor's next loop will dispatch a fresh claim against the now-claimable task at Attempt=2.

2. **`internal/ops/claim_task_test.go`** — New tests:
   - `TestClaimTask_IterationCapAttempt1_TriggersNewAttempt`: REJECTED task at attempt 1, iteration at limit. ClaimTask returns PreconditionError containing "transitioned to attempt 2". Verify task is in initial pipeline status with Attempt=2, Iteration=0, ReviewCyclesCurrent=0.
   - `TestClaimTask_IterationCapAttempt2_TriggersBlocked`: REJECTED task at attempt 2, iteration at limit. ClaimTask returns PreconditionError containing "transitioned to BLOCKED". Verify task in BLOCKED status.

**done_when:** (a) The iteration limit check in `ClaimTask` (around line 151) uses `classifyLimitEscalation` with `task.EffectiveAttempt()` instead of direct `task.Iteration >= maxCoderIterations` comparison. (b) `LimitActionNewAttempt` calls `TransitionToNewAttempt` and returns PreconditionError to abort the claim. (c) `LimitActionBlocked` delegates to existing `enforceRejectedIterationLimit()` (preserving its atomic re-check). (d) `TestClaimTask_IterationCapAttempt1_TriggersNewAttempt` passes: REJECTED task at attempt 1 hitting iteration limit → PreconditionError returned, task transitioned to initial status with Attempt=2. (e) `TestClaimTask_IterationCapAttempt2_TriggersBlocked` passes: REJECTED task at attempt 2 hitting iteration limit → PreconditionError returned, task in BLOCKED.

**scope:** `internal/ops/claim_task.go`, `internal/ops/claim_task_test.go`.

4. **`internal/ops/claim_task.go`** — Replace `handleRejectedClaimWorktree` (lines 372-432) with `ensureRejectedWorktreeExists`. The new function has no identity parameters — it checks worktree/branch existence and either validates (present) or recreates (missing):
   ```go
   func ensureRejectedWorktreeExists(
       gitWrapper *git.Git,
       ctx *claimContext,
   ) (claimWorktreePhaseResult, error) {
       result := claimWorktreePhaseResult{}
       branchName := paths.TaskBranchPrefix + ctx.taskID

       // Check if worktree and branch exist
       worktreeExists := false
       if _, err := os.Stat(ctx.worktreeDir); err == nil {
           worktreeExists = true
       }
       branchExists, err := gitWrapper.BranchExists(branchName)
       if err != nil {
           return result, fmt.Errorf("failed to check branch %s: %w", branchName, err)
       }

       if worktreeExists && branchExists {
           // Validate existing worktree (reuses validateRejectedSameCoderWorktree logic)
           if err := validateRejectedWorktree(gitWrapper, ctx.taskID, ctx.worktreeDir, ctx.worktreeRel); err != nil {
               return result, err
           }
           return result, nil
       }

       // Worktree or branch missing — recreate from integration branch.
       // Clean up partial state first.
       if worktreeExists {
           if err := gitWrapper.RemoveWorktree(ctx.taskID); err != nil {
               return result, fmt.Errorf("failed to remove partial worktree: %w", err)
           }
           result.deleted = true
       }
       if branchExists {
           _ = gitWrapper.DeleteBranch(branchName)
       }

       if _, err := gitWrapper.CreateWorktree(ctx.taskID, ctx.integrationBranch); err != nil {
           return result, fmt.Errorf("failed to recreate worktree from integration branch: %w", err)
       }
       result.created = true
       return result, nil
   }
   ```

5. **`internal/ops/claim_task.go`** — Rename `validateRejectedSameCoderWorktree` (line 508) to `validateRejectedWorktree` (drop "SameCoder" since identity is no longer relevant). The function body is unchanged (validates worktree dir exists, branch exists, HEAD readable).

6. **`internal/ops/claim_task.go`** — Remove the now-unused `handleRejectedClaimWorktree` function (lines 372-432) and `restoreRejectedWorktreeAfterCreateFailure` function (lines 532-553). The recovery-ref logic is no longer needed because within an attempt, missing worktrees are simply recreated from the integration branch (not restored to a prior branch state).

7. **`internal/ops/claim_task_test.go`** — Replace existing same-coder/different-coder worktree tests with identity-agnostic tests:
   - `TestClaimTask_RejectedWorktreePresent_Preserved`: REJECTED task with existing worktree and branch. Any coder claims → worktree preserved, validated, no deletion/recreation. `WorktreeRecreated=false`.
   - `TestClaimTask_RejectedWorktreeMissing_Recreated`: REJECTED task with missing worktree. Any coder claims → worktree recreated from integration branch. `WorktreeRecreated=true`.
   - `TestClaimTask_RejectedMutateTask_NoCounterReset`: REJECTED task claimed by different coder than `previousAssignee` → `ReviewCyclesCurrent` is NOT reset (inherited within attempt).
   - Remove or replace tests that assert same-coder vs different-coder branching in worktree handling.

**done_when:** (a) `rejectedClaimStrategy.mutateTask()` has no conditional on `previousAssignee` — body has no field mutations (counters, worktree, baseCommit are unchanged). (b) `rejectedClaimStrategy.handleWorktree()` calls `ensureRejectedWorktreeExists` (identity-free). (c) `ensureRejectedWorktreeExists` in `claim_task.go`: if worktree+branch exist → validate and preserve; if missing → recreate from integration branch. (d) `handleRejectedClaimWorktree` and `restoreRejectedWorktreeAfterCreateFailure` functions are removed from `claim_task.go`. (e) `validateRejectedSameCoderWorktree` renamed to `validateRejectedWorktree`. (f) `rejectedClaimStrategy.validate()` still captures `previousAssignee` for history events. (g) `TestClaimTask_RejectedWorktreePresent_Preserved` passes: existing worktree preserved regardless of coder identity. (h) `TestClaimTask_RejectedWorktreeMissing_Recreated` passes: missing worktree recreated from integration branch. (i) `TestClaimTask_RejectedMutateTask_NoCounterReset` passes: `ReviewCyclesCurrent` not reset on different-coder claim within same attempt.
