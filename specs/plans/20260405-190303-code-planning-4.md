# Code Plan: Prompts and Context Sections for Architecture Step

**Task:** code-planning-4
**Spec:** specs/goals/20260405-architecture-step.md
**Scope:** Planning only: architect context section in role_context.go (rendering N parent tasks' descriptions and outputs), architecture-reviewer context section, ArchRef field in role_context.go struct, code-planner/coder/reviewer template updates to render arch_ref when present, orchestrator prompt updates for detailed-spec path (create architecture tasks instead of code-planning). Covers spec items 4, 7, and 8. No implementation.

## Context

The architecture step adds architect and architecture-reviewer roles to the coding sub-pipeline. These roles need complete prompt support: context data population, template blocks for agent instructions, and orchestrator routing. Prior phases (CP1-CP3) establish the pipeline config, transition engine, and arch_ref data model. This plan wires those into the prompt layer so agents can function.

### Key Codebase Observations

1. **`RoleContextData` is the unified template data struct**: All template blocks receive the same `RoleContextData` from `role_context.go`. Fields not relevant to a role remain at zero value. Adding `ArchRef` and parent task data follows this pattern.

2. **`buildTaskRoleContextData` in `agent/prompt.go` populates the struct**: Lines 113-211 populate fields by role type (doer/reviewer) and specific role. The `PlanRef` population pattern (line 130, with `worktreeRelPath` and `splitPlanRefFile`) is the exact precedent for `ArchRef`.

3. **Template blocks have role-conditional branches**: `assigned_task.tmpl`, `doer_tools.tmpl`, `doer_state_transitions.tmpl`, `implementation_phase.tmpl` each use `{{if eq .Role "role-name"}}` to render role-specific content. Templates without a matching branch render nothing for that role — meaning architect/architecture-reviewer get empty sections for tools, state transitions, and implementation phase until branches are added.

4. **Reviewer templates also have role-conditional branches**: `review_task.tmpl`, `review_instructions.tmpl`, `reviewer_state_transitions.tmpl`, `reviewer_tools.tmpl` each branch by role. The architecture-reviewer currently gets no review instructions or state transitions rendered.

5. **CP1 defines architect context-sections without architecture-specific blocks**: CP1's plan (merged) defines the architect's context-sections as: `assigned-task, worktree-rules, prior-rejection, prior-attempt, doer-state-transitions, doer-tools, implementation-phase, mandatory-docs, skills-affinity`. CP1 explicitly defers "architecture-specific sections (rendering parent tasks' content)" to CP4. A new `parent-tasks-context` section needs to be created and added.

6. **`wake_initial_planning.tmpl` hardcodes `type: "coding"`**: Lines 37, 81, and 122 all use `"type": "coding"` for task creation. After CP1 retargets `detailed-spec` to `architecture-pair`, the resolved role-pair will be `architecture-pair` but the task type will still say `"coding"` instead of `"architecture"`. The type must be resolved from the pipeline config.

7. **`wakeTemplateData` in `wake.go` resolves role-pair and display name from pipeline config**: `buildWakeTemplateData` (line 92) already resolves `ResolvedRolePair` and `ResolvedDisplayName` from the pipeline. Adding `ResolvedTaskType` follows the same pattern using `models.TaskTypeForRole(doerRole)`.

8. **ParentTasks linkage from CP1**: `EffectiveParentTasks()` accessor returns the list of parent task IDs. For many-to-one children (architecture tasks), this returns N parent IDs (the US tasks in the cohort). The parent-tasks-context template iterates over these to render context.

9. **US tasks don't carry output[] entries**: US tasks produce story files in the repo but don't use the `output[]` mechanism (that's for tasks generating downstream task definitions). The parent-tasks-context should render parent task metadata (ID, description, done_when, spec/plan references) to help the architect locate relevant documents.

## Design Decisions

### D1: ArchRef uses same worktree resolution pattern as PlanRef

`PlanRef` is resolved via `worktreeRelPath(splitPlanRefFile(task.PlanRef), resolveWorktreePath(...))` at line 130 of agent/prompt.go. `ArchRef` follows the identical pattern. The worktree-relative path ensures agents can `Read` the file from within their worktree.

### D2: ParentTaskContexts as flat struct slice, not pre-rendered string

The parent task data could be pre-rendered to a string (like `DashboardOutput` for the orchestrator) or passed as structured data to the template. Structured data gives the template full control over formatting and allows future template iteration without Go code changes. This matches how `SiblingTasks` and `CompletedTasks` are handled.

### D3: Parent-tasks-context renders metadata, not file content

The architect's parent tasks (US tasks) produce story files in the repo. Embedding file content in the prompt would be wasteful and potentially large. Instead, the parent-tasks-context renders task metadata (ID, description, done_when, spec/plan refs) to orient the architect, who then reads relevant files via standard tools.

### D4: Architect implementation-phase reuses code-planner pattern with arch-doc focus

The architect's workflow (create doc, set output, submit) parallels the code-planner's. The implementation-phase template branch uses the same structure: file location, set_task_output instructions, pre-submit self-check. The arch doc goes to `specs/arch-plan/` (matching the spec's `specs/arch-plan/<goal-slug>.md` convention).

### D5: Dynamic task type in orchestrator template via pipeline config resolution

Rather than hardcoding task types per entry-point in the template, `ResolvedTaskType` is resolved from the pipeline config (role-pair → doer role → task type). This ensures the template stays correct as new roles and entry-points are added. `wakeEntryPointData` also gains a `TaskType` field for the classification branch.

### D6: Architecture-reviewer review checklist focuses on structural coherence

Per spec: "Reviews architectural coherence, interface contracts, structural soundness." The review-instructions branch provides architecture-specific guidance: decomposition completeness, interface clarity, cross-cutting concern coverage, and composability of downstream code plans.
