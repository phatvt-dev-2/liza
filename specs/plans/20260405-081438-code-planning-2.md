## Design Decisions

### D1: Relax single-sub-pipeline constraint for role-pair reuse

The spec explicitly defines `integration-subpipeline` with `steps: [integration-pair, coding-pair]`, reusing coding-pair from coding-subpipeline. Current validation (`config.go:252-254`) rejects role-pairs appearing in multiple sub-pipelines.

Analysis of the constraint's blast radius:
- **rpSubPipeline map**: Only used for the uniqueness check itself. Not referenced after the sub-pipeline validation loop.
- **validateTransition**: Receives `sp.Steps` (the sub-pipeline's own step list), not `rpSubPipeline`. Validates from/to pairs against the local steps — works correctly with coding-pair listed in both sub-pipelines.
- **Resolver**: Looks up role-pairs by globally unique name. `TransitionMap()`, `AllDeclaredStates()`, `SprintTerminalStates()`, `resolvePhase()`, `AvailableTransitions()` — none assume single sub-pipeline ownership.
- **SprintTerminalStates**: coding-pair has no outgoing transitions in either sub-pipeline. It remains non-source. No behavior change.
- **Existing test**: `TestLoad_RolePairInMultipleSubPipelines` asserts the error. Must be updated to expect success.

The constraint is defensive validation with no downstream dependents. Removing it is safe. The change is minimal: remove the error check, keep the step-existence check. The `rpSubPipeline` map can be removed entirely since it's unused after the loop.

### D2: Integration role definitions follow existing doer/reviewer patterns

The integration-analyst and integration-reviewer follow the same structural patterns as other doer/reviewer roles: same allowed-operations, same timeout structure, similar context-sections. The key differences are:
- **Skills**: analyst gets `code-review` + `clean-code`; reviewer gets `systemic-thinking` + `software-architecture-review` (per spec)
- **Context sections**: Both include a new `branch-integration-context` section (implemented by CP3) providing the branch diff command and completed task list
- **Analyst omissions**: No `collective-plan-scoping` (not part of a planning decomposition), no `task-decomposition` (output format is fixed), no `commit-workflow` (analyst produces findings, not code commits)

### D3: Auto transition declared but execution mechanism deferred to CP4

The spec defines `integration-to-fix` with `trigger: auto`. The pipeline schema already supports `auto` as a valid trigger value (`config.go:380`). `Proceed` explicitly rejects auto triggers with comment "Auto transitions are reserved for supervisor execution" (`proceed.go:857-859`). However, `ExecuteAvailableTransitions` currently only collects manual transitions via `AvailableTransitions` (`resolver.go:449`), so auto transitions have no execution path today.

This plan declares the auto transition in pipeline.yaml. The execution mechanism (modifying `ExecuteAvailableTransitions` or `AvailableTransitions` to handle auto triggers) belongs to CP4's end-to-end lifecycle scope, which covers the supervisor's integration pipeline orchestration.
