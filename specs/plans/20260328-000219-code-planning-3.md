# Code Planning 3: Stall Detection â€” Decouple from Heartbeat mtime

## Problem

`checkStalled` (`internal/commands/watch.go:565`) uses `os.Stat(statePath).ModTime()` to determine whether progress has occurred within the last 30 minutes. However, agent heartbeats (`internal/agent/heartbeat.go:75`) call `bb.Modify()` every heartbeat interval to renew agent and task leases. Since `Blackboard.Modify()` (`internal/db/blackboard.go:260`) rewrites `state.yaml` on every call, the file's mtime is refreshed continuously while any agent is alive â€” masking the exact failure mode stall detection is supposed to catch: agents are present and renewing leases, but no task is actually advancing.

**Spec reference:** `tofix.md` lines 1-5 and 16-18.

## Analysis

### Files Analyzed

**`internal/commands/watch.go`** â€” Contains `checkStalled` (line 565). Called from the alert-check list at line 163 with `lizaPaths.StatePath()` and `config.StateCache`. The caller already has the parsed `*models.State` available at line 163 (variable `state`). Current logic: `os.Stat(statePath) â†’ age := time.Since(info.ModTime()) â†’ if age > StallThreshold â†’ alert`.

**`internal/agent/heartbeat.go`** â€” `Heartbeat.beat()` (line 75) calls `bb.Modify()` to update `agent.Heartbeat`, `agent.LeaseExpires`, and optionally task lease timestamps. This modifies agent-level fields and task lease fields â€” NOT task history, status, or any progress-signaling field. No changes needed: the heartbeat's purpose (lease renewal) is correct; only the stall detector's signal source is wrong.

**`internal/agent/supervisor.go`** â€” Starts the heartbeat goroutine at line 460-473 via `NewHeartbeat` + `hb.Start(heartbeatCtx)`. The heartbeat runs for the full supervisor lifetime. No changes needed: the supervisor correctly starts a lifetime heartbeat, and the heartbeat correctly renews leases. The bug is in how stall detection interprets file-level mtime, not in the write pattern.

### Progress Signal: Task History

Every meaningful task state change appends a `TaskHistoryEntry` with a `Time` field (see `internal/models/history.go:81`). Events include: `claimed`, `submitted_for_review`, `approved`, `rejected`, `merged`, `blocked`, `pre_execution_checkpoint`, `new_attempt`, `handoff_initiated`, and many others (`internal/models/history.go:46-77`). Heartbeat writes do NOT create history entries.

The latest `TaskHistoryEntry.Time` across all tasks is therefore a precise semantic signal for "when did the last meaningful thing happen?" â€” exactly what stall detection needs.

### Edge Cases

| Scenario | Expected Behavior |
|----------|-------------------|
| No tasks exist | Not stalled (nothing to advance) |
| All tasks terminal (MERGED/ABANDONED/SUPERSEDED) | Not stalled (work is done) |
| Active tasks exist but no history entries | Use earliest `task.Created` as progress baseline |
| Heartbeat running, no task transitions for 30+ min | Stalled (correct detection) |
| Task just claimed (history entry added) | Not stalled (< 30 min since last history event) |
| Sprint checkpoint/paused state | Existing `checkSprintStalled` handles this separately â€” no overlap |

## Design

### Approach

Change `checkStalled` to derive progress from the latest `TaskHistoryEntry.Time` across all tasks, rather than from `os.Stat(statePath).ModTime()`.

### Why Not Alternative Approaches

**Option B: Add `LastProgress` timestamp to `State`** â€” Requires modifying the state model, all mutation paths, and distinguishing heartbeat writes from other writes in `Blackboard.Modify()`. Invasive and error-prone.

**Option C: Separate heartbeat file** â€” Breaks the single-file coordination model. Heartbeat needs to update task lease fields in state.yaml, so it can't be fully separated without duplicating state.

**Option A (chosen): Derive from task history** â€” Zero state model changes, zero heartbeat changes, zero supervisor changes. The signal is already there; we just need to read it instead of stat'ing the file.

## Task 1: Refactor `checkStalled` to derive progress from task history

### Intent

Replace mtime-based stall detection with task-history-based progress detection so that heartbeat writes no longer mask stalled tasks.

### Changes

**`internal/commands/watch.go`:**

1. **Change `checkStalled` signature** from `func checkStalled(statePath string, cache map[string]time.Time) []Alert` to `func checkStalled(state *models.State, cache map[string]time.Time) []Alert`.

2. **Replace mtime logic with history scan:**
   - Iterate `state.Tasks` to determine: (a) whether any active (non-terminal) task exists, and (b) the latest `TaskHistoryEntry.Time` across all tasks.
   - If no active tasks exist â†’ return no alerts (delete cache key).
   - If no history entries exist on any task â†’ fall back to the earliest `task.Created` timestamp as the progress baseline.
   - Compute `age := now.Sub(latestProgress)`. If `age <= StallThreshold` â†’ return no alerts.
   - Otherwise, throttle and emit alert (same 5-minute throttle logic as current).

3. **Update call site** (line 163): change `checkStalled(lizaPaths.StatePath(), config.StateCache)` to `checkStalled(state, config.StateCache)`.

4. **Remove `os` import usage** for stall detection (the `os.Stat` call). If `os` is still used elsewhere in the file, keep the import; otherwise remove it. (It IS used elsewhere â€” `os.Stat` is also used in other functions.)

5. **Update alert message** to reflect the new signal: `"no task progress for %d minutes"` (was `"no progress for %d minutes"`).

**`internal/commands/watch_test.go`:**

6. **Rewrite `TestCheckStalled`:** Replace file-mtime-based setup with `*models.State` construction. Test cases:
   - Active task with recent history entry â†’ no alert.
   - Active task with stale history entry (31+ min) â†’ 1 alert.
   - No tasks â†’ no alert.
   - All tasks terminal â†’ no alert.
   - Active task with no history entries, `Created` 31+ min ago â†’ 1 alert (fallback to Created).

7. **Rewrite `TestCheckStalledThrottling`:** Same throttle semantics but using state-based setup.

**`internal/agent/heartbeat.go`:** No changes. Heartbeat correctly updates lease fields; it does not and should not create task history entries.

**`internal/agent/supervisor.go`:** No changes. Supervisor correctly starts the heartbeat goroutine for the full agent lifetime.

### Test Strategy

| Test | Asserts |
|------|---------|
| `TestCheckStalled/active_task_recent_history` | State with one CODING task + history entry 5 min ago â†’ `len(alerts) == 0` |
| `TestCheckStalled/active_task_stale_history` | State with one CODING task + history entry 31 min ago â†’ `len(alerts) == 1`, message contains "no task progress" |
| `TestCheckStalled/no_tasks` | State with empty Tasks â†’ `len(alerts) == 0` |
| `TestCheckStalled/all_tasks_terminal` | State with one MERGED task + stale history â†’ `len(alerts) == 0` |
| `TestCheckStalled/no_history_fallback_to_created` | State with one CODING task, no history, Created 31 min ago â†’ `len(alerts) == 1` |
| `TestCheckStalled/no_history_recent_created` | State with one CODING task, no history, Created 5 min ago â†’ `len(alerts) == 0` |
| `TestCheckStalledThrottling` | Same throttle semantics: first call alerts, immediate second call suppressed, 5+ min later alerts again |

All test cases construct `*models.State` in memory â€” no file I/O needed. This makes tests faster, deterministic, and independent of filesystem mtime precision.

### Scope

- **Modified files:** `internal/commands/watch.go`, `internal/commands/watch_test.go`
- **Analyzed, no changes:** `internal/agent/heartbeat.go`, `internal/agent/supervisor.go`

### Risks

None material. The function signature change is internal (not exported). The only caller is in the same file. Tests are rewritten to match the new interface.
