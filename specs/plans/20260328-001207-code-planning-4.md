# Code Planning 4: CLI Commands — Init Wizard Flag Forwarding & Watch TTY Fallback

## Problem

Two issues in CLI command handling:

1. **Init wizard flag forwarding (blocker, tofix.md lines 11-14):** The interactive init wizard path (`cmd/liza/cmd_init.go:88-136`) does not forward cobra flag defaults for `--config` and `--post-worktree-cmd` to `InitCommandWithConfig`. While commit `8d62264` gates wizard entry on `!hasExplicitInitFlags(cmd)` — preventing explicit flags from being silently dropped — the wizard path still bypasses cobra defaults. Specifically: if `~/.liza/pipeline.yaml` exists, the CLI path (`cmd_init.go:162`) uses it as the default ConfigPath, while the wizard path passes `""` (triggering embedded config instead). Additionally, `InitWizardResult.Branch` (line 20 of `init_wizard.go`) is never populated by any wizard prompt — a dead field that defaults to `""`.

2. **Watch TTY fallback (concern, tofix.md lines 7-9, 20-22):** The `tui`/`watch` command (`cmd/liza/cmd_system.go:94-125`) unconditionally attempts to allocate a TTY via `tea.NewProgram(model, tea.WithAltScreen())` when `--headless` is not set. Running `liza watch </dev/null` fails with "could not open a new TTY." Non-interactive environments (CI, cron, background shells) should auto-fallback to headless mode.

## Analysis

### Files Analyzed

**`cmd/liza/cmd_init.go`** — Contains the init command dispatch. Three paths:
1. Wizard path (lines 88-136): entered when no args, no agent flags, no explicit init flags, and stdin is interactive. Runs `RunInitWizard`, then calls `InitCommandWithConfig` with wizard results. Missing: `ConfigPath` and `PostWorktreeCmd` not forwarded from cobra defaults.
2. Pairing path (lines 138-157): entered when agent flags present without description.
3. Full workspace path (lines 159-181): entered when description arg provided. Reads all flags from cobra.

The wizard path at lines 122-131 constructs `InitParams` with `ConfigPath` and `PostWorktreeCmd` implicitly zero-valued. The full workspace path (lines 162-165) explicitly reads these from cobra flags, getting defaults like `defaultPipelineConfigPath()` for `--config`.

**`internal/interactive/init_wizard.go`** — Contains `InitWizardResult` struct and `RunInitWizard`. The struct has a `Branch` field (line 20) that is never set by any wizard screen. Screen 3 (full mode) asks for Description, SpecRef, and EntryPoint — but not Branch, ConfigPath, or PostWorktreeCmd. This is acceptable since `8d62264` prevents the wizard from activating when these flags are explicit. However, the dead `Branch` field is misleading.

**`cmd/liza/cmd_system.go`** — Contains the `tui`/`watch` command (lines 78-126). The `headless` flag (line 100) controls whether to run headless mode or TUI mode. When `headless` is false, `tea.NewProgram(model, tea.WithAltScreen())` (line 122) attempts TTY allocation, which fails in non-interactive environments.

**`internal/interactive/tty.go`** — Contains `IsInteractive()` which checks `os.Stdin.Stat().Mode()&os.ModeCharDevice`. Already used by the init wizard gate at `cmd_init.go:90`.

**`internal/commands/init.go`** — Contains `InitParams` struct (line 22) and `InitCommandWithConfig` (line 411). When `Branch` is `""`, it defaults to `"integration"` (line 418). When `ConfigPath` is `""`, it auto-freezes embedded pipeline config (line 449). When `PostWorktreeCmd` is `""` and stdin is interactive, it auto-suggests a post-worktree command based on detected package manager (line 575).

### Behavioral Gap: ConfigPath Default

| Path | ConfigPath value | Behavior when `~/.liza/pipeline.yaml` exists |
|------|-----------------|----------------------------------------------|
| CLI: `liza init "desc"` | `defaultPipelineConfigPath()` → `~/.liza/pipeline.yaml` | Uses custom pipeline config |
| Wizard: `liza init` | `""` | Uses embedded config (different!) |

This is the concrete behavioral difference the fix addresses.

## Design

### Task 1: Forward cobra flag defaults through wizard init path

**Approach:** After the wizard completes, read cobra flag defaults for `config` and `post-worktree-cmd` and forward them into the `InitCommandWithConfig` call. Remove the dead `Branch` field from `InitWizardResult` and read the branch default from cobra instead.

**Why this approach:**
- Achieves behavioral parity between wizard and CLI paths for default config resolution
- Eliminates dead struct field that could mislead future developers
- Minimal change: 3-4 lines added to cmd_init.go, 1 line removed from init_wizard.go

**Why not add wizard prompts for Branch/Config/PostWorktreeCmd:**
- These fields have sensible defaults. Adding prompts would complicate the wizard for power-user settings that most interactive users don't need.
- Users who need non-default values can use `liza init "desc" --branch custom` directly.
- PostWorktreeCmd already has auto-suggestion in InitCommandWithConfig (line 575), which runs after the wizard.

### Task 2: TTY auto-fallback for watch/tui command

**Approach:** Before attempting TUI initialization, check `interactive.IsInteractive()`. If stdin is not a terminal and `--headless` was not explicitly set, auto-set `headless = true` and emit a notice to stderr.

**Why this approach:**
- Preserves existing behavior for interactive terminals (TUI mode)
- Preserves existing behavior when `--headless` is explicitly passed
- Adds graceful degradation for non-interactive environments
- Uses the same `interactive.IsInteractive()` check already used by the init wizard

**Why not keep watch headless-by-default:**
- Would be a breaking change for interactive users who expect TUI
- The spec suggests auto-fallback as the preferred approach

## Task 1: Forward cobra flag defaults through wizard init path

### Intent

Ensure the wizard init path forwards cobra flag defaults for `--config`, `--branch`, and `--post-worktree-cmd` to `InitCommandWithConfig`, achieving behavioral parity with the CLI path. Remove the dead `Branch` field from `InitWizardResult`.

### Changes

**`cmd/liza/cmd_init.go`:**

1. **Read cobra defaults after wizard completes (after line 106):** Add three lines to read cobra flag defaults:
   ```go
   configPath, _ := cmd.Flags().GetString("config")
   branch, _ := cmd.Flags().GetString("branch")
   postWorktreeCmd, _ := cmd.Flags().GetString("post-worktree-cmd")
   ```

2. **Forward defaults in full-mode InitCommandWithConfig call (lines 122-131):** Add `ConfigPath`, `PostWorktreeCmd`, and change `Branch` source:
   ```go
   if err := commands.InitCommandWithConfig(commands.InitParams{
       Description:     result.Description,
       SpecRef:         result.SpecRef,
       ConfigPath:      configPath,
       EntryPoint:      result.EntryPoint,
       Branch:          branch,
       PostWorktreeCmd: postWorktreeCmd,
       AutoResume:      autoResume,
       Agents:          result.Agents,
       Stdin:           os.Stdin,
       ContractAction:  result.ContractAction,
   }); err != nil {
   ```

**`internal/interactive/init_wizard.go`:**

3. **Remove dead `Branch` field from `InitWizardResult` (line 20):** Delete:
   ```go
   Branch         string   // integration branch name (full mode only; "" = default "integration")
   ```

## Task 2: TTY auto-fallback for watch/tui command

### Intent

When `liza tui` or `liza watch` runs in a non-interactive environment (no TTY on stdin), auto-fallback to headless mode instead of failing with "could not open a new TTY."

### Changes

**`cmd/liza/cmd_system.go`:**

1. **Add import for interactive package (line 4 area):**
   ```go
   "github.com/liza-mas/liza/internal/interactive"
   ```

2. **Add TTY detection after reading headless flag (after line 100):** Insert before the `if headless {` block:
   ```go
   if !headless && !interactive.IsInteractive() {
       fmt.Fprintln(os.Stderr, "No TTY detected, falling back to headless mode")
       headless = true
   }
   ```

3. **No changes to headless or TUI code paths** — the existing logic handles both cases correctly once `headless` is set.

### Test Strategy

| Test | File | Asserts |
|------|------|---------|
| `TestTuiCmd_FallsBackToHeadlessWithoutTTY` | `cmd/liza/cmd_system_test.go` (new) | Set up a temp dir with valid `.liza/state.yaml`, redirect stdin to a pipe (non-interactive), run `tui` command. Verify: (a) no "could not open a new TTY" error, (b) stderr contains "No TTY detected, falling back to headless mode". The command will still fail (no running sprint or context cancellation), but the error must not be TTY-related. |
| `TestTuiCmd_ExplicitHeadlessSkipsFallback` | `cmd/liza/cmd_system_test.go` | Run with `--headless` flag and non-interactive stdin. Verify no fallback message on stderr (headless was already requested). |
| Manual verification | — | `go run ./cmd/liza tui </dev/null` should print fallback notice to stderr and enter headless mode. |

### Scope

- **Modified files:** `cmd/liza/cmd_system.go`, `cmd/liza/cmd_system_test.go` (new)

### Risks

None material. The `interactive.IsInteractive()` function is well-tested (used by init wizard gate). The auto-fallback only activates when stdin is not a terminal, which is precisely the failure case.

## Dependency Order

Task 1 and Task 2 have **no shared files** and can run in parallel. No `depends_on` relationship needed.
