## Task 4: Add Await Resubmission to MULTI_AGENT_MODE.md contract

**Intent:** Document the reviewer-side await-resubmission workflow in the Iteration Protocol so reviewer agents learn about this flow from the contract.

**Spec ref:** Task scope includes `contracts/MULTI_AGENT_MODE.md`. This update mirrors the existing "Await Verdict" section for the reviewer side.

**Files:**
- `contracts/MULTI_AGENT_MODE.md`

**Changes:**

Add an "Await Resubmission" subsection after the "Await Verdict" subsection (after line 137) in the Iteration Protocol section:

```markdown
**Await Resubmission (recommended post-verdict flow for reviewers):**
After `liza_submit_verdict` with REJECTED (non-terminal), call `liza_await_resubmission` to block until the doer resubmits. This preserves the reviewer's accumulated context about the code under review and eliminates cold-restart overhead (~47s).

| Verdict | Agent action |
|---------|-------------|
| **RESUBMITTED** | Task reclaimed for review — review the new changes (same session) |
| **TERMINAL** | Exit normally — task blocked/superseded |
| **TIMEOUT** / **ABORTED** | Exit normally |

Do NOT call if the verdict result shows `EscalatedToBlocked` or `NewAttemptTriggered`.
```

**Done when:** `contracts/MULTI_AGENT_MODE.md` Iteration Protocol section contains an "Await Resubmission" subsection with verdict table and usage guidance, placed after the existing "Await Verdict" subsection. Pre-commit passes on the file.

## Task 5: Document reviewer await-resubmission workflow in USAGE_MULTI_AGENTS.md

**Intent:** Add reviewer-side await-resubmission workflow documentation for end users, and add the tool to the permission listing example.

**Spec ref:** Spec step 11, row: "docs/USAGE_MULTI_AGENTS.md — Document reject -> await -> re-review workflow"

**Files:**
- `docs/USAGE_MULTI_AGENTS.md`

**Changes:**

1. Add a new subsection after "Submit, Await Verdict, Handle Result" (after line 483):

```markdown
### Review, Reject, Await Resubmission

Reviewer agents use a blocking workflow after non-terminal rejections:

1. `liza_submit_verdict` — issue REJECTED verdict with feedback
2. `liza_await_resubmission` — block until doer resubmits
3. Handle result:
   - **RESUBMITTED**: Review the new changes (session stays alive — no cold restart)
   - **TERMINAL** / **TIMEOUT** / **ABORTED**: Exit normally

This mirrors the doer-side `liza_await_verdict` flow, reducing per-rejection overhead from ~47s (cold restart) to near-zero for reviewers. Do not call after terminal rejections (`EscalatedToBlocked` or `NewAttemptTriggered`).
```

2. Add `"mcp__liza__liza_await_resubmission"` to the sample `.claude/settings.json` permissions list (after `"mcp__liza__liza_await_verdict"`, line 412):

```json

      "mcp__liza__liza_await_resubmission"
```

**Done when:** `docs/USAGE_MULTI_AGENTS.md` contains a "Review, Reject, Await Resubmission" section after the existing "Submit, Await Verdict, Handle Result" section, and `mcp__liza__liza_await_resubmission` appears in the sample permissions list. Pre-commit passes on the file.
