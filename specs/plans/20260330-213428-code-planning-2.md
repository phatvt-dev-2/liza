## Overview

This plan covers MCP endpoint wiring and pipeline configuration for `liza_await_resubmission` — the reviewer-side mirror of `liza_await_verdict`. After issuing a REJECTED verdict, a reviewer calls this tool to block until the doer resubmits, preserving the review session.

1. **Task 1**: MCP handler (`handleAwaitResubmission`) in `handlers_mutation.go`
2. **Task 2**: Pipeline config for reviewer roles + pipeline test updates
3. **Task 3**: Tool registration in `server_registration.go` + `claude-settings.json` permission
4. **Task 4**: Handler-level tests

Tasks 1 and 2 run in parallel (different files, no dependency). Task 3 depends on Task 1 (handler reference). Task 4 depends on Tasks 2 and 3 (role check test needs both registration and pipeline config).

**No change needed:** `config_test.go` — uses an inline minimal fixture for config parsing validation with only `write-checkpoint` and `submit-for-review` on the coder role. The embedded pipeline's reviewer operations are validated by `resolver_test.go` and `migrate_test.go`.

## Task 1: Implement `handleAwaitResubmission` MCP handler

**Intent:** Add the MCP handler that extracts parameters, delegates to `ops.AwaitResubmission`, and formats the response.

**Design:**

```go
func (s *Server) handleAwaitResubmission(params map[string]any) (any, error)
```

**Parameter extraction:**
- `task_id` — required string, via `requireString(params, "task_id")`
- `agent_id` — required string, via `requireString(params, "agent_id")`
- `timeout_seconds` — optional float64 (MCP transmits integers as float64), default 1500s (25 min, within Claude Code's 30 min `MCP_TIMEOUT`)

**Delegation:**
```go
result, err := ops.AwaitResubmission(context.Background(), s.projectRoot, taskID, agentID, timeout)
```

**Error handling:**
- No `ErrBudgetExhausted` handling — spec explicitly states "Budget gate not needed" for `await_resubmission`
- All errors wrapped with `"await resubmission failed: %w"`

**Response formatting:**
```go
msg := fmt.Sprintf("Verdict: %s\nStatus: %s", result.Verdict, result.TaskStatus)
if result.Verdict == ops.ResubmissionResubmitted {
    msg += fmt.Sprintf("\nNew submission received. Review the changes at commit %s. Review cycle %d.",
        result.ReviewCommit, result.ReviewCycle)
}
if result.Reason != "" {
    msg += fmt.Sprintf("\nReason: %s", result.Reason)
}
return textResult(msg)
```

**Key design notes:**
1. `context.Background()` used because MCP handlers don't receive a context from the protocol layer. `ops.AwaitResubmission` handles its own deadline via the `timeout` parameter.
2. Unlike `handleAwaitVerdict`, there is no `ErrBudgetExhausted` path — the rejected status implicitly validates that the doer CAN iterate (spec: "Budget gate not needed").
3. The `RESUBMITTED` case includes inline guidance with the new commit SHA and review cycle count, enabling the reviewer to immediately begin re-review.
4. The `Reason` field is only included when non-empty (TERMINAL cases have a reason; RESUBMITTED/TIMEOUT/ABORTED may not).

**Scope:** `internal/mcp/handlers_mutation.go`

**done_when:** `handleAwaitResubmission` method on `*Server` extracts `task_id` (required), `agent_id` (required), `timeout_seconds` (optional, default 1500), calls `ops.AwaitResubmission(context.Background(), ...)`, formats RESUBMITTED verdict with commit SHA and review cycle, includes Reason only when non-empty, wraps errors with "await resubmission failed"; `go build ./internal/mcp/...` succeeds.

**spec_ref:** Spec step 4 "MCP handler"

## Task 3: Register `liza_await_resubmission` tool and add claude-settings permission

**Intent:** Wire the tool definition into `registerMutationTools` and enable auto-approval in `claude-settings.json`.

**Design — Registration (`server_registration.go`):**

Add to the `registerMutationTools` `toolDef` slice, after `liza_await_verdict`:

```go
{
    tool: protocol.Tool{
        Name:        "liza_await_resubmission",
        Description: "Block until doer resubmits after a rejection. Call after " +
            "liza_submit_verdict with REJECTED. Preserves review ownership.",
        InputSchema: protocol.InputSchema{
            Type: "object",
            Properties: map[string]protocol.Property{
                "task_id":         {Type: "string", Description: "Task ID to await resubmission for"},
                "agent_id":        {Type: "string", Description: "Agent ID (for authorization)"},
                "timeout_seconds": {Type: "integer", Description: "Max wait time in seconds (default: 1500, must be < MCP_TIMEOUT)"},
            },
            Required: []string{"task_id", "agent_id"},
        },
    },
    handler:     s.handleAwaitResubmission,
    roleChecker: operationChecker(s.resolver, s.pipelineLoadErr, "liza_await_resubmission"),
}
```

**Key design notes:**
1. `operationChecker` gates the tool to roles that have `await-resubmission` in their `allowed-operations` (all reviewer roles, per Task 2's pipeline config).
2. Placement after `liza_await_verdict` groups the two blocking tools together.
3. Description matches spec step 5: "Block until doer resubmits after a rejection."

**Design — Settings (`claude-settings.json`):**

Add `"mcp__liza__liza_await_resubmission"` to the `permissions.allow` array, after `"mcp__liza__liza_await_verdict"` (grouped with the other blocking tool):

```json
"mcp__liza__liza_await_resubmission",
```

**Scope:** `internal/mcp/server_registration.go`, `internal/embedded/claude-settings.json`

**done_when:** `liza_await_resubmission` appears in `registerMutationTools` with InputSchema having `task_id` (string, required), `agent_id` (string, required), `timeout_seconds` (integer, optional); handler is `s.handleAwaitResubmission`; roleChecker is `operationChecker` for `"liza_await_resubmission"`; `"mcp__liza__liza_await_resubmission"` is in `claude-settings.json` `permissions.allow` array; `go build ./cmd/liza-mcp/...` succeeds.

**spec_ref:** Spec step 5 "Register tool" and step 9 "Tool permission"

**depends_on:** Task 1 (handler method must exist for `s.handleAwaitResubmission` reference to compile)

## Task 4: Add handler-level tests for `handleAwaitResubmission`

**Intent:** Test the MCP handler's parameter validation, role enforcement, and successful resubmission flow.

**Design:**

Tests go in `internal/mcp/handlers_await_resubmission_test.go` (dedicated file, following the `handlers_await_verdict_test.go` pattern) and/or `internal/mcp/handlers_test.go`:

**Test cases:**

1. **`TestHandleAwaitResubmissionMissingTaskID`**: Call handler with `agent_id` only → error mentioning "task_id"
2. **`TestHandleAwaitResubmissionMissingAgentID`**: Call handler with `task_id` only → error mentioning "agent_id"
3. **`TestHandleAwaitResubmissionTaskNotInAwaitableStatus`**: Call with valid params on a task in READY status (not rejected/submitted) → error "await resubmission failed" (precondition failure from ops layer)
4. **`TestHandleAwaitResubmission_RoleCheck`**: Access via registered handler (`server.handlers["liza_await_resubmission"]`) with coder agent → error "not allowed for role coder" (only reviewer roles are authorized)
5. **`TestHandleAwaitResubmission_Success`**: Set up task in CODE_REJECTED status with rejection history by this reviewer agent; in goroutine, transition to CODE_READY_FOR_REVIEW (doer resubmits) after delay; call handler → result contains "Verdict: RESUBMITTED" and "Review the changes at commit"

**Test patterns:**
- Use `setupTestWorkspaceWithGit(t)` for workspace setup
- Use `NewServer(projectRoot, logPath)` for server creation
- For success test: use `testhelpers.CreateValidState()` + `testhelpers.BuildTaskByStatus()` with `CODE_REJECTED` status, add `TaskEventReviewVerdictRejected` history entry with reviewer agent ID
- Goroutine transitions task to submitted status via `bb.Modify`, triggers resubmission detection
- Use `testhelpers.WaitForAsyncSetup()` for reliable timing

**Key design notes:**
1. Tests 1-3 call the handler method directly — test parameter extraction and ops-layer preconditions.
2. Test 4 calls through the registered handler to verify the `operationChecker` middleware rejects non-reviewer roles.
3. Test 5 verifies the reviewer-specific response format: commit SHA and review cycle (not the Guidance field from `await_verdict`).
4. No budget exhaustion test needed — `await_resubmission` has no budget gate (spec: "Budget gate not needed").

**Scope:** `internal/mcp/handlers_test.go`, `internal/mcp/handlers_await_resubmission_test.go`

**done_when:** Tests pass for: (1) missing task_id returns error mentioning "task_id", (2) missing agent_id returns error mentioning "agent_id", (3) non-rejected task status returns "await resubmission failed" error, (4) coder role rejected by registered handler role checker with "not allowed for role coder", (5) successful RESUBMITTED verdict returns formatted text with "Verdict: RESUBMITTED" and "Review the changes at commit"; `go test ./internal/mcp/... -run TestHandleAwaitResubmission` passes.

**spec_ref:** Spec step 10 "Tests — handlers_test.go"

**depends_on:** Task 2 (pipeline config needed for `operationChecker` to authorize reviewer roles), Task 3 (tool must be registered for role check test)
