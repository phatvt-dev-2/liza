## Overview

Phase 5 is the integration and migration phase that connects the TUI built in Phases 1-4 to the existing `liza watch` command. The core changes are:

1. **watchCmd modification** — Add `--headless` flag; default behavior launches the Bubbletea TUI, `--headless` preserves the current headless monitoring loop
2. **console.sh deprecation** — Add deprecation notice to the embedded `console.sh` script

Anomaly monitoring (13 checks + circuit breaker + sprint stalled + state validity) and `alerts.log` parallel write are already wired by Phase 3's exported `RunChecksWithState` API and the TUI's `runChecksCmd`/`TickMsg` handler chain. Phase 5's integration in `cmd_system.go` simply launches the TUI, which activates that pipeline.

## Anomaly Monitoring Wiring

The 13 anomaly checks from `watch.go:runChecks()` are wired into the TUI through the following chain (established by Phase 3):

```
watchCmd (Phase 5)
  → tea.NewProgram(model).Run()
    → model.Init()
      → tickCmd() (starts 10s timer)
    → model.Update(TickMsg)
      → runChecksCmd(projectRoot, alertsLogPath, state, stateCache)
        → commands.RunChecksWithState(state, config)
          → all 13 checks + circuit breaker + sprint stalled + state validity
        → commands.WriteAlert(alertsLog, alert)  ← alerts.log parallel write
      → alertsMsg returned to Update
        → alerts converted to ActivityEntry items in activity feed
        → critical alerts (🚨) set alert banner
```

This chain runs every 10 seconds via `tea.Tick`. No additional wiring is needed in Phase 5 — launching the TUI via `tea.NewProgram` is sufficient to activate the entire pipeline.

### alerts.log Parallel Write

Phase 3's `runChecksCmd` (in `internal/tui/commands.go`) writes each alert to `alerts.log` via `commands.WriteAlert` before returning `alertsMsg`. This maintains the audit trail for both TUI and headless modes:
- **Headless mode** (`--headless`): `runChecks` → `writeAlert` (existing behavior, unchanged)
- **TUI mode** (default): `runChecksCmd` → `commands.WriteAlert` (same file, same format)

Both paths produce identical `alerts.log` entries.

## Task 1: Modify watchCmd for TUI launch with --headless flag (cmd_system.go)

### Intent
Modify the existing `watchCmd` in `cmd/liza/cmd_system.go` to launch the Bubbletea TUI program by default, with a `--headless` flag that preserves the current headless monitoring behavior.

### Design

#### Flag Addition

Add `--headless` bool flag to `watchCmd`:

```go
watchCmd.Flags().Bool("headless", false, "run in headless mode (no TUI, alerts to stderr + alerts.log)")
```

#### watchCmd RunE Modification

```go
RunE: func(cmd *cobra.Command, args []string) error {
    projectRoot, err := requireProjectRoot()
    if err != nil {
        return err
    }

    headless, _ := cmd.Flags().GetBool("headless")

    if headless {
        // Preserve current behavior: headless monitoring loop
        interval, _ := cmd.Flags().GetInt("interval")
        if interval <= 0 {
            return fmt.Errorf("interval must be positive")
        }

        config := commands.WatchConfig{
            ProjectRoot:   projectRoot,
            CheckInterval: time.Duration(interval) * time.Second,
            StateCache:    make(map[string]time.Time),
        }

        return commands.WatchCommand(context.Background(), config)
    }

    // Default: launch interactive TUI
    model, err := tui.New(projectRoot)
    if err != nil {
        return fmt.Errorf("failed to initialize TUI: %w", err)
    }

    p := tea.NewProgram(model, tea.WithAltScreen())
    _, err = p.Run()
    return err
},
```

#### Updated Command Description

Update `watchCmd.Short` and `watchCmd.Long` to reflect the new interactive behavior:

```go
Short: "Interactive TUI dashboard for monitoring Liza",
Long: `Launch an interactive TUI dashboard that monitors the Liza blackboard.

The TUI provides:
  - Live dashboard with color-coded status indicators for agents and tasks
  - Keyboard commands to operate the system (spawn, pause, resume, add task, checkpoint, stop)
  - Inline anomaly monitoring with alerts in the activity feed
  - Reactive updates via fsnotify with 10s poll fallback

Use --headless for non-interactive monitoring (alerts to stderr + alerts.log).
This is suitable for CI, cron, or running in a secondary terminal.

Press '?' in the TUI for a full keybinding reference.`,
```

#### Import Addition

Add to `cmd_system.go` imports:

```go
tea "github.com/charmbracelet/bubbletea"
"github.com/liza-mas/liza/internal/tui"
```

#### Interval Flag Scope

The `--interval` flag applies only to headless mode. In TUI mode, the 10s poll interval is fixed per spec §Data Sources & Refresh. No change needed to the flag itself — it is only read in the headless branch.

### Scope
- **In scope:** `cmd/liza/cmd_system.go` — modify `watchCmd` RunE, update Short/Long descriptions, add `--headless` flag, add imports
- **Out of scope:** `internal/tui/` changes (Phases 1-4 scope), `internal/commands/watch.go` changes (Phase 3 scope)

### Done When
- `watchCmd` RunE branches on `--headless` flag: headless runs existing `commands.WatchCommand`, default launches `tui.New(projectRoot)` + `tea.NewProgram(model, tea.WithAltScreen()).Run()`
- `--headless` flag defined as bool with default `false` and help text `"run in headless mode (no TUI, alerts to stderr + alerts.log)"`
- `watchCmd.Short` updated to `"Interactive TUI dashboard for monitoring Liza"`
- `watchCmd.Long` updated to describe TUI features and `--headless` usage
- `tui.New(projectRoot)` error is handled and wrapped with `fmt.Errorf("failed to initialize TUI: %w", err)`
- `tea.WithAltScreen()` option passed to `tea.NewProgram` for full-screen TUI
- `--interval` flag only consumed in headless branch
- `go vet ./cmd/liza/...` passes
- Unit test in `cmd/liza/cmd_system_test.go` verifies: `watchCmd` has `--headless` flag defined with default `false`; `--interval` flag still exists for headless mode backward compatibility
