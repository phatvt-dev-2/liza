# Code Plan: Pipeline Definition, Roles, and Task Model Changes for Architecture Step

**Task:** code-planning-1
**Spec:** specs/goals/20260405-architecture-step.md
**Scope:** New roles (architect, architecture-reviewer), architecture-pair role-pair with 6 states, ParentTasks multi-parent field on task model with callsite migration, entry point retargeting (detailed-spec to architecture-pair), us-to-coding transition update with many-to-one cardinality declaration. Covers spec items 1, 3, 9, and the task-model portion of item 2. No implementation of the many-to-one engine, arch_ref, context sections, or prompts.

## Context

The architecture step adds a consolidation point between upstream specs and downstream code planning. This plan covers the structural foundation: roles, pipeline configuration, and task model changes needed before the many-to-one transition engine (CP2), arch_ref propagation (CP3), and prompt/context assembly (CP4) can be built.

### Key Codebase Observations

1. **Pipeline YAML is self-validating**: Role names in YAML reference other YAML-defined roles, not Go constants. The YAML can be updated independently of Go role registration.
2. **Cardinality validation is centralized**: `validateTransitionHeader` in `config.go:376-387` is the single checkpoint for valid cardinality values. Both sub-pipeline and pipeline-transition validation use it.
3. **Architecture states are auto-recognized**: `IsPipelineValid` (task.go:150-155) accepts any pipeline-declared state via `resolver.AllDeclaredStates()`. No hardcoded state list update needed for the 6 new architecture states (DRAFT_ARCHITECTURE through ARCHITECTURE_REJECTED). Verified: `validateTaskStates` (validate_task.go:54-79) uses `IsPipelineValid` with the resolver, not bare `IsValid()`.
4. **TaskType must be registered**: `validateTaskStates` (validate_task.go:64) calls `task.EffectiveType().IsValid()` which checks `taskWorkflows`. A new `TaskTypeArchitecture` must be registered. `TaskTypeForRole` (task.go:81-88) derives task type from doer role — without the registration, architect tasks would fall back to `TaskTypeCoding`.
5. **ParentTask is a singular *string**: Used in 2 production write sites (proceed.go:792,819), 1 production read site (validate_task.go:243), and ~10 test sites. The migration is bounded.
6. **Test fixtures separate from embedded config**: `valid-phase2-full.yaml` and `valid-coding-subpipeline.yaml` are independent fixtures. Tests loading these must be updated alongside the fixture. `TestResolver_FixtureMatchesEmbeddedPipeline` enforces sync between `valid-phase2-full.yaml` and the embedded pipeline.

### Sibling Task Boundaries

- **CP2**: many-to-one engine in proceed.go (cohort detection, child creation from N parents, wake detection, crash recovery, pipeline resolver cardinality parsing for proceed logic). CP2 owns the behavioral interpretation of `many-to-one`; this plan only declares it as a valid YAML value.
- **CP3**: arch_ref field on OutputEntry and Task, set-task-output handling, validation, propagation in proceed.go.
- **CP4**: Architect and architecture-reviewer context sections, prompt templates, orchestrator prompt updates.

## Design Decisions

### D1: Keep ParentTask for backward compatibility, add ParentTasks alongside

The spec offers two approaches: "ParentTasks []string field (or converting ParentTask to a slice)." Converting ParentTask to a slice would break YAML deserialization for existing state.yaml files that have `parent_task: "string-value"` (YAML string vs sequence).

Approach: Add `ParentTasks []string` as a new field (`yaml:"parent_tasks,omitempty"`). Keep `ParentTask *string` (`yaml:"parent_task,omitempty"`) for backward compatibility. Add `EffectiveParentTasks() []string` accessor that returns `ParentTasks` if non-empty, else wraps `ParentTask` into a single-element slice. All new child creation code (proceed.go) sets `ParentTasks`. All reading code uses `EffectiveParentTasks()`. CP2 will use `ParentTasks` to set multiple parents for many-to-one transitions.

### D2: Architect context sections use existing doer template sections

The architect role definition uses standard doer context sections (assigned-task, worktree-rules, doer-state-transitions, doer-tools, etc.) that already exist. Architecture-specific sections (rendering parent tasks' content for the architect's input) are CP4's scope. This avoids referencing undefined sections in the YAML.

### D3: Architecture-pair uses default review quorum of 1

Per spec: "The consolidation value comes from having an explicit architecture step at all, not from multiple reviewers." No review-policy override needed. The architecture-reviewer carries `systemic-thinking` for cross-cutting coherence.

### D4: Validation-only cardinality update, no engine behavior

Adding `"many-to-one"` to `validateTransitionHeader` allows the YAML to declare it. The actual `many-to-one` behavior in `proceedInner` (case branch, cohort logic, etc.) is CP2's scope. `proceed.go:158-159` already has a `default` case that returns an error for unsupported cardinalities — `many-to-one` transitions won't accidentally execute until CP2 implements the handler.

```go
// EffectiveParentTasks returns the list of parent task IDs.
// Returns ParentTasks if non-empty. Otherwise, wraps ParentTask
// into a single-element slice for backward compatibility.
// Returns nil if the task has no parents.
func (t *Task) EffectiveParentTasks() []string {
    if len(t.ParentTasks) > 0 {
        return t.ParentTasks
    }
    if t.ParentTask != nil && *t.ParentTask != "" {
        return []string{*t.ParentTask}
    }
    return nil
}
```
